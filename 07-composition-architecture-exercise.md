## بخش الف: Composition

**۱.** Composition چیست و چه رابطه‌ای می‌سازد؟ در کد زیر چرا `class Car(Engine)` یک طراحی اشتباه است و شکل درست آن چیست؟

```python
class Engine:
    def start(self):
        return "started"

class Car(Engine):
    pass
```

**۲.** «انفجار کلاس» (Class Explosion) در ارث‌بری چیست؟ با مثال کارمندان (تمام‌وقت/پاره‌وقت × بانکی/نقدی) توضیح دهید و بگویید Composition چطور از آن جلوگیری می‌کند.

**۳.** جدول زیر را درباره‌ی تفاوت عملی ارث‌بری و Composition کامل کنید:

| ویژگی | ارث‌بری (Is-A) | Composition (Has-A) |
|-------|----------------|----------------------|
| زمان اتصال | ؟ | ؟ |
| نوع وابستگی | ؟ | ؟ |
| انفجار کلاس | ؟ | ؟ |

**۴.** آیا Composition **همیشه** از ارث‌بری بهتر است؟ Tradeoff آن چیست و چه زمانی ارث‌بری هنوز انتخاب درستی است؟

## بخش ب: اصول طراحی

**۵.** قانون دیمتر («با غریبه‌ها حرف نزن») چیست؟ زنجیره‌ی `self.customer.wallet.account.balance` چه مشکلی دارد و چطور آن را اصلاح می‌کنید؟

**۶.** اصل Tell, Don't Ask چیست؟ تفاوت دو رویکرد زیر را توضیح دهید و بگویید کدام بهتر است و چرا:

```python
# approach 1
if account.balance >= amount:
    account.balance -= amount
else:
    raise ValueError("insufficient balance")

# approach 2
account.withdraw(amount)
```

**۷.** تفاوت کلاس «Anemic» (بی‌رفتار) و «Rich» (غنی) چیست؟ Tradeoff هرکدام کدام است و چه زمانی ممکن است عمداً Anemic انتخاب کنیم؟

**۸.** معماری لایه‌ای چیست؟ سه لایه‌ی رایج آن را نام ببرید و جهت جریان وابستگی را مشخص کنید. یک تفاوت آن با معماری هگزاگونال چیست؟

## بخش ج: کدنویسی و کدخوانی

**۹. (بازنویسی)** کد زیر قانون دیمتر را نقض می‌کند. آن را طوری بازنویسی کنید که `Order` فقط با «دوست مستقیم» خود (`Customer`) حرف بزند:

```python
class Order:
    def total_with_discount(self):
        return self.customer.wallet.account.balance * 0.9
```

**۱۰. (کدنویسی)** یک سیستم اعلان با Composition و چندریختی بسازید: یک کلاس پایه‌ی `Channel` با متد `send`، دو زیرکلاس `EmailChannel` و `SMSChannel`، و یک کلاس `Notifier` که فهرستی از کانال‌ها را نگه دارد و متد `broadcast(message)` را روی همه اجرا کند. توضیح دهید افزودن یک کانال جدید چطور انجام می‌شود.

**۱۱. (بازنویسی رویه‌ای → شیءگرا)** تابع رویه‌ای زیر را با Composition/چندریختی بازنویسی کنید تا افزودن یک نوع مشتری جدید نیازی به ویرایش کد موجود نداشته باشد:

```python
def discount(customer_type, amount):
    if customer_type == "regular":
        return amount
    elif customer_type == "vip":
        return amount * 0.8
    elif customer_type == "employee":
        return amount * 0.5
```

**۱۲. (تست‌پذیری)** برای همان `Notifier` سوال ۱۰، یک `FakeChannel` بسازید که به‌جای ارسال واقعی، پیام‌ها را در یک لیست ذخیره کند. سپس نشان دهید چطور می‌توان بدون شبکه، رفتار `broadcast` را تست کرد. این با مفهوم Dependency Injection چه ارتباطی دارد؟

## بخش د: مقایسه و تحلیل

**۱۳.** چرا شیءگرایی «ریفکتور را امن‌تر می‌کند»؟ تفاوت تغییردادن یک قاعده در کد رویه‌ای و کد شیءگرا را توضیح دهید.

**۱۴.** چرا Composition به‌طور طبیعی «تست‌پذیری» می‌آورد؟ نقش «وابستگی به رابط به‌جای کلاس مشخص» را در این موضوع توضیح دهید.

## بخش ه: پرسش تحلیلی

**۱۵.** این جمله را نقد کنید: «چون قاعده می‌گوید Composition را بر ارث‌بری ترجیح بده، پس هرگز نباید از ارث‌بری استفاده کنیم و همه‌جا باید Composition باشد.»

**۱۶.** نویسنده تأکید می‌کند «هیچ راه‌حل مطلقاً درستی وجود ندارد» و این را «در این فصل بیش از هر جای دیگر جدی بگیرید». با مثال انتخاب بین معماری لایه‌ای و هگزاگونال، توضیح دهید چرا انتخاب درست به موقعیت بستگی دارد نه به یک قاعده‌ی ثابت.
