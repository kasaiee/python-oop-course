## بخش الف: کندی و تکنیک‌های بهینه‌سازی

**۱.** دو عاملِ اصلی:
- **جست‌وجوی attribute (Attribute Lookup):** هر بار که `obj.x` می‌نویسید، پایتون باید `x` را جست‌وجو کند — اول در `__dict__`ِ شیء، بعد در کلاس، بعد در والدها طبقِ MRO.
- **سربارِ فراخوانیِ متد (Method Call Overhead):** هر فراخوانیِ متد گران‌تر از یک تابعِ ساده است، چون `self` باید پاس داده شود و متد در MRO یافت شود.

این هزینه‌ها فقط در «حلقه‌های داغ» (میلیون‌ها تکرار) اهمیت پیدا می‌کنند، چون در یک اجرای معمولی آن‌قدر کوچک‌اند که به چشم نمی‌آیند؛ فقط وقتی میلیون‌ها بار تکرار شوند، جمعشان محسوس می‌شود.

**۲.** `__slots__` علاوه بر حافظه، **دسترسیِ سریع‌ترِ attribute** هم می‌دهد. چون attributeها در یک ساختارِ ثابت نگه‌داری می‌شوند نه در دیکشنری (`__dict__`)، جست‌وجویشان کمی سریع‌تر است. در ساختِ میلیون‌ها شیء و دسترسیِ مکرر، این تفاوت محسوس می‌شود. (Tradeoff: همان بی‌انعطافیِ فصل دهم پابرجاست.)

**۳.** بارگذاریِ تنبل یعنی محاسبه‌ی یک مقدارِ گران را تا لحظه‌ی واقعیِ نیاز به تعویق بیندازید؛ اگر هرگز لازم نشد، هرگز محاسبه نمی‌شود. `cached_property` هر دو را با هم می‌دهد: مقدار را تا **اولین دسترسی** به تعویق می‌اندازد (تنبل)، و بعد از محاسبه، نتیجه را **ذخیره** می‌کند (کش) تا دفعاتِ بعد دوباره محاسبه نشود. مناسب برای مقادیرِ گران و ثابت.

**۴.** `lru_cache` نتیجه‌ی یک تابع را برای هر ورودی **کش** می‌کند، پس فراخوانیِ دوم با همان ورودی، بدونِ محاسبه‌ی دوباره، مقدارِ ذخیره‌شده را برمی‌گرداند. **Tradeoff:** حافظه مصرف می‌کند (با `maxsize` کنترلش کنید). فقط برای **توابعِ خالص** (pure) امن است — توابعی که با ورودیِ یکسان همیشه خروجیِ یکسان می‌دهند و به حالتِ متغیرِ بیرونی وابسته نیستند؛ اگر خروجی به حالتِ متغیر بستگی داشته باشد، کش نتایجِ **غلط** می‌دهد.

## بخش ب: چندنخی و چندپردازشی

**۵.** GIL یک قفلِ سراسری است که اجازه می‌دهد در هر لحظه فقط **یک نخ** کدِ پایتون اجرا کند. محدودیت: حتی اگر چند نخ بسازید، برای اجرای کدِ پایتون نوبتی می‌شوند و واقعاً هم‌زمان اجرا نمی‌شوند.

**۶.**
- **CPU-bound** (محاسباتِ سنگین): `multiprocessing` مناسب است. چون GIL نخ‌ها را نوبتی می‌کند، چندنخی سرعتی نمی‌آورد؛ اما چندپردازشی پردازش‌های جدا (هرکدام GILِ خودش) می‌سازد که واقعاً موازی‌اند.
- **I/O-bound** (انتظار برای شبکه/دیسک): `threading` (یا `asyncio`) مناسب است. چون هنگامِ انتظار، GIL آزاد می‌شود و نخِ دیگر می‌تواند کار کند.

قاعده: CPU-bound → multiprocessing؛ I/O-bound → threading/asyncio.

**۷.** `UnsafeCounter` خطرناک است چون `self.count += 1` یک عملِ **اتمیک نیست**: در واقع سه مرحله است (خواندنِ مقدار، افزودنِ یک، نوشتنِ دوباره). اگر دو نخ هم‌زمان این کار را بکنند، ممکن است هر دو مقدارِ قدیمی را بخوانند و یکی از افزایش‌ها گم شود. اسمِ این مشکل **شرایطِ مسابقه (Race Condition)** است. رفع با `Lock`:

```python
import threading

class SafeCounter:
    def __init__(self):
        self.count = 0
        self._lock = threading.Lock()
    def increment(self):
        with self._lock:        # only one thread at a time
            self.count += 1
```

`Lock` تضمین می‌کند تغییرِ `count` هر بار توسطِ فقط یک نخ انجام شود.

**۸.** چون مشکلِ thread-safety ریشه در **حالتِ مشترکِ متغیر** دارد: وقتی چند نخ همان داده را می‌خوانند و می‌نویسند. اگر شیء **بی‌حالت** یا **تغییرناپذیر** باشد، اصلاً حالتِ مشترکِ متغیری وجود ندارد که خراب شود — پس نیازی به قفل نیست. قفل هرچند کار می‌کند، سربار اضافه می‌کند و اگر بد استفاده شود به **بن‌بست (Deadlock)** منجر می‌شود. حذفِ ریشه‌ی مشکل (حالتِ مشترک) ساده‌تر و امن‌تر از مدیریتِ آن با قفل است.

## بخش ج: اندازه‌گیری و ضدالگوها

**۹.** چون **حدس‌زدنِ گلوگاه تقریباً همیشه اشتباه است**؛ اغلب اوقات کندی جایی است که انتظارش را نداشتید. اگر بدونِ اندازه‌گیری بهینه کنید، ممکن است ساعت‌ها روی کدی وقت بگذارید که اصلاً در مسیرِ داغ نیست. دو ابزار:
- `cProfile`: نشان می‌دهد کدام **تابع** چقدر وقت می‌گیرد (برای پیداکردنِ گلوگاهِ کلی).
- `timeit`: برای زمان‌سنجیِ **قطعاتِ کوچکِ** کد و مقایسه‌ی دو رویکرد.

(ابزارهای دیگر: `line_profiler` خط‌به‌خط، `memory_profiler` برای حافظه.)

**۱۰.**
- **God Object:** کلاسِ همه‌کاره که هم طراحیِ بد است و هم بهینه‌سازی/کش‌کردنش سخت است چون همه‌چیز در آن گره خورده.
- **Lava Flow:** کدِ مرده یا قدیمی که کسی جرئتِ حذفش را ندارد، پس می‌ماند، حافظه و زمانِ بارگذاری می‌گیرد و خوانایی را کم می‌کند.
- **بهینه‌سازیِ زودرس:** پیچیده‌کردنِ کد برای سرعتی که نیازش نیست؛ فقط بدهیِ فنی می‌سازد.
- **N+1 Query:** به‌جای یک کوئریِ دسته‌ای، درونِ حلقه برای هر شیء یک کوئریِ جدا به دیتابیس زدن.

**۱۱.** N+1 Query یعنی به‌جای یک کوئری برای گرفتنِ همه‌ی داده، اول یک کوئری می‌زنید تا فهرست را بگیرید (۱ کوئری)، بعد **درونِ حلقه** برای هر عضو یک کوئریِ جدا می‌زنید (N کوئری) — جمعاً N+1 کوئری. در مقیاسِ بزرگ فاجعه است چون اگر ۱۰٬۰۰۰ رکورد داشته باشید، ۱۰٬۰۰۱ رفت‌وبرگشت به دیتابیس می‌زنید به‌جای یک یا دو تا؛ هر رفت‌وبرگشت تأخیرِ شبکه دارد و مجموعشان برنامه را فلج می‌کند. راه‌حل: گرفتنِ داده به‌شکلِ دسته‌ای (batch/join).

## بخش د: کدنویسی و کدخوانی

**۱۲.** فقط **یک بار**. `cached_property` مقدار را در اولین دسترسی محاسبه و ذخیره می‌کند؛ دسترسیِ دوم از کش می‌آید و بدنه اجرا نمی‌شود. خروجی:

```
محاسبه‌ی سنگین...
499500
499500
```

**۱۳.**

```python
import threading

class SafeCounter:
    def __init__(self):
        self.count = 0
        self._lock = threading.Lock()

    def increment(self):
        with self._lock:
            self.count += 1
```

`with self._lock` تضمین می‌کند در هر لحظه فقط یک نخ واردِ بخشِ بحرانی شود، پس افزایش‌ها گم نمی‌شوند.

**۱۴.**

```python
from functools import lru_cache

@lru_cache(maxsize=None)
def fibonacci(n):
    if n < 2:
        return n
    return fibonacci(n - 1) + fibonacci(n - 2)

print(fibonacci(30))     # 832040 — fast
```

بدونِ کش، `fibonacci` بازگشتیِ **نمایی** است: `fibonacci(30)` بارها `fibonacci(29)`، `fibonacci(28)` و... را دوباره و دوباره محاسبه می‌کند (تعدادِ فراخوانی‌ها به‌شکلِ نمایی رشد می‌کند). با `lru_cache`، هر مقدار فقط **یک‌بار** محاسبه و کش می‌شود، پس زمان از نمایی به خطی کاهش می‌یابد.

**راهِ دیگر (بدونِ بازگشت):** می‌توان همان را با یک حلقه‌ی ساده و بدونِ کش هم نوشت که حافظه‌ی ثابت مصرف می‌کند:

```python
def fibonacci(n):
    a, b = 0, 1
    for _ in range(n):
        a, b = b, a + b
    return a
```

Tradeoff: نسخه‌ی حلقه‌ای سریع و کم‌حافظه است؛ نسخه‌ی `lru_cache` وقتی مفیدتر است که فراخوانی‌های تکراری با ورودی‌های یکسان زیاد داشته باشید.

## بخش ه: پرسشِ تحلیلی

**۱۵.** جمله‌ی کنوت یعنی: **پیش از اینکه بدانید گلوگاه کجاست، بهینه‌سازی نکنید.** اکثرِ کدِ یک برنامه در «مسیرِ داغ» نیست و بهینه‌سازی‌اش هیچ سودِ محسوسی ندارد؛ اما پیچیده‌کردنِ آن به‌نامِ سرعت، کد را سخت‌فهم، پرباگ و سخت‌نگهداری می‌کند. «کدِ خوانا و درست که کمی کند است» بهتر است چون:
- اگر آن بخش در مسیرِ داغ نباشد، کندی‌اش اصلاً مهم نیست.
- خوانایی نگهداری، رفعِ باگ و توسعه را آسان می‌کند — که در طولِ عمرِ پروژه بسیار ارزشمندتر از چند میلی‌ثانیه است.
- اگر بعداً معلوم شد واقعاً کند است، می‌توانید همان **یک نقطه** را با اطمینان بهینه کنید.

پس اول درستی و خوانایی، بعد اندازه‌گیری، و بهینه‌سازی فقط جایی که ثابت شده لازم است.

**۱۶.** این جمله دقیقاً مصداقِ **بهینه‌سازیِ زودرس** است و چند اشکال دارد:
- **`threading` همه‌جا:** برای کارهای CPU-bound به‌خاطرِ GIL سودی نمی‌دهد و فقط پیچیدگی و خطرِ Race Condition اضافه می‌کند.
- **`__slots__` همه‌جا:** انعطاف را می‌گیرد و برای کلاس‌هایی با تعدادِ کمِ نمونه بی‌فایده است.
- **کش همه‌جا:** حافظه مصرف می‌کند و برای توابعِ ناخالص نتایجِ غلط می‌دهد.

مشکلِ بنیادی: بدونِ **اندازه‌گیری**، نمی‌دانید گلوگاه کجاست، پس این پیچیدگی‌ها را جایی اضافه می‌کنید که هیچ سودی ندارند و فقط کد را سخت‌تر می‌کنند. پاسخِ درست همان مسیرِ فصل است: اول کدِ درست و خوانا، بعد `cProfile`/`timeit` برای یافتنِ گلوگاهِ واقعی، و بهینه‌سازیِ هدفمندِ فقط همان نقطه. Tradeoffِ خوانایی در برابر سرعت را فقط جایی بپذیرید که اندازه‌گیری ثابت کرده ارزشش را دارد.
