# فصل صفر: آماده‌سازی — پیش‌نیازها و نقشه‌ی راه دوره

---

## مقدمه

سال‌ها تدریس یک چیز را به من ثابت کرده: بیشترِ کسانی که در میانه‌ی یادگیریِ شیءگرایی رها می‌کنند، مشکلشان شیءگرایی نیست — مشکلشان یک حفره‌ی کوچک در پیش‌نیازهاست که هیچ‌کس به آن‌ها نگفته بود. دانشجویی داشتم که تا فصلِ ارث‌بری آمده بود و گیج بود؛ نیم‌ساعت که با هم کد زدیم معلوم شد مشکلش ارث‌بری نیست، این است که هنوز نمی‌داند دیکشنری چه فرقی با لیست دارد. برگشت، دو روز روی پایه‌ها وقت گذاشت، و بعد بقیه‌ی دوره را با سرعتِ دو برابر رفت.

این فصل برای همین نوشته شده. اینجا **شیءگرایی یاد نمی‌گیریم**؛ مطمئن می‌شویم زمینِ زیرِ پایمان سفت است، و بعد نقشه‌ی کاملِ مسیر را پهن می‌کنیم تا بدانید قرار است از کجا به کجا برسید.

دو حالت دارید:

- **پایتون را تازه شروع کرده‌اید؟** این فصل را کامل و با حوصله بخوانید و حتماً کدهایش را خودتان اجرا کنید.
- **با پایتون راحتید؟** مستقیم بروید سراغِ [آزمون خودسنجی](00-getting-ready-exercise.md). اگر بالای ۸۰٪ درست جواب دادید، این فصل را رد کنید و از فصل اول شروع کنید. اگر نه، برگردید همین‌جا — بدونِ عذابِ وجدان. دو ساعت سرمایه‌گذاری اینجا، ده‌ها ساعت سردرگمی در فصل‌های بعد را حذف می‌کند.

## چرا دوره‌ی شیءگرایی به فصل صفر نیاز دارد؟

چون تقریباً همه‌ی دوره‌های شیءگرایی — فارسی و خارجی — با این جمله شروع می‌شوند: «فرض می‌کنیم با مقدمات پایتون آشنایید.» و همین فرضِ ساده، دیوارِ نامرئی‌ای می‌سازد که خیلی‌ها پشتش جا می‌مانند.

واقعیت این است که برای شروعِ این دوره به دانشِ عجیبی نیاز ندارید. فهرستش کوتاه است:

- متغیرها و انواعِ پایه (عدد، رشته، بولین)
- ساختارهای داده‌ی پایه (لیست، دیکشنری، تاپل)
- شرط‌ها و حلقه‌ها
- توابع — مهم‌ترینِ همه، چون «متد» که از فصل دوم می‌آید، همان تابع است با یک تفاوتِ کوچک
- اجرای یک فایل پایتون و خواندنِ پیغام خطا

همین. حالا یکی‌یکی، سریع و فشرده، مرورشان می‌کنیم — نه مثل یک کتابِ مرجع، بلکه فقط با تمرکز روی چیزهایی که در این دوره لازمتان می‌شود.

## ابزار کار: پایتون و محیط اجرا

در این دوره از **پایتون ۳.۱۰ یا جدیدتر** استفاده می‌کنیم؛ بعضی سینتکس‌ها (مثل `X | Y` در تایپ‌هینت‌ها که فصل نهم می‌بینید) در نسخه‌های قدیمی‌تر کار نمی‌کنند. نسخه‌تان را بررسی کنید:

```bash
python --version
# or on some systems:
python3 --version
```

کد را به دو شکل اجرا خواهید کرد:

```bash
python main.py        # running a file
python                # interactive mode (REPL) — for quick experiments
```

هر ویرایشگری که راحتید خوب است؛ VS Code با افزونه‌ی Python انتخابِ رایجی است. فقط یک قانون دارم که تا آخر دوره جدی بگیرید: **کدهای درس را کپی نکنید. تایپ کنید.** انگشت‌ها چیزهایی یاد می‌گیرند که چشم‌ها یاد نمی‌گیرند.

## متغیرها و انواع پایه

متغیر در پایتون فقط یک «برچسب» است که روی یک مقدار می‌چسبد — نه جعبه‌ای که مقدار داخلش ریخته شود. این تصویرِ ذهنی را نگه دارید؛ در فصل دوم که به `self` می‌رسیم و در فصل چهاردهم که به حافظه می‌رسیم، همین تصویر نجاتتان می‌دهد.

```python
price = 250_000          # int (underscores just improve readability)
rate = 4.5               # float
title = "Python Course"  # str
is_active = True         # bool
nothing = None           # None: the absence of a value

print(type(price))       # <class 'int'>
```

به خروجی `type` دقت کنید: `class 'int'`. از همین حالا پایتون دارد رازش را لو می‌دهد — **در پایتون همه‌چیز شیء است**، حتی یک عددِ ساده. فعلاً همین‌قدر بدانید؛ معنیِ عمیقش را در طول دوره می‌فهمید.

برای ساختنِ رشته از روی متغیرها، از **f-string** استفاده می‌کنیم که در تمام دوره خواهید دید:

```python
name = "Sara"
count = 3
print(f"{name} has {count} unread messages")   # Sara has 3 unread messages
print(f"total: {250_000 * 3:,}")               # total: 750,000  (the , adds separators)
```

## ساختارهای داده: چهار ظرف اصلی

فرض کنید داریم بخشی از یک فروشگاه اینترنتی را می‌سازیم (اتفاقاً پروژه‌های آخر دوره خیلی به همین حال‌وهوا نزدیک‌اند). داده‌های مختلف، ظرف‌های مختلفی می‌خواهند:

```python
# list: an ordered, changeable collection — the shopping cart
cart = ["keyboard", "mouse", "monitor"]
cart.append("webcam")          # add to the end
first = cart[0]                # access by index: "keyboard"
print(len(cart))               # 4

# dict: key-value pairs — one product's details
product = {"name": "keyboard", "price": 500_000, "stock": 12}
print(product["price"])        # 500000
product["stock"] -= 1          # update a value

# tuple: like a list, but immutable — a fixed (x, y) or (lat, long)
location = (35.7, 51.4)
# location[0] = 36  ← TypeError! tuples cannot be changed

# set: unordered, no duplicates — visited user IDs
visited = {101, 102, 103}
visited.add(102)               # already there; nothing happens
print(visited)                 # {101, 102, 103}
```

جمع‌بندیِ انتخاب ظرف — این جدول را گوشه‌ی ذهنتان نگه دارید:

| نیاز | ظرف مناسب |
|------|-----------|
| دنباله‌ای از چیزها با ترتیب، قابل تغییر | `list` |
| جست‌وجوی سریع با کلید (نام → مشخصات) | `dict` |
| چند مقدارِ به‌هم‌چسبیده که نباید عوض شوند | `tuple` |
| مجموعه‌ی بدونِ تکرار، تستِ عضویتِ سریع | `set` |

روی دیکشنری کمی بیشتر بایستیم، چون در شیءگرایی نقش ویژه‌ای دارد (فصل دوم خواهید دید که attributeهای هر شیء واقعاً در یک دیکشنری به نام `__dict__` نگه‌داری می‌شوند):

```python
user = {"username": "ali", "email": "ali@example.com"}

for key, value in user.items():      # iterating over a dict
    print(key, "=", value)

print(user.get("phone"))             # None — .get does not raise an error
print(user.get("phone", "unknown"))  # unknown — with a default value
```

## شرط‌ها و حلقه‌ها

منطق برنامه با شرط و حلقه ساخته می‌شود. یک سناریوی واقعی: می‌خواهیم سفارش‌های یک فروشگاه را پردازش کنیم.

```python
orders = [
    {"id": 1, "amount": 800_000, "paid": True},
    {"id": 2, "amount": 150_000, "paid": False},
    {"id": 3, "amount": 2_400_000, "paid": True},
]

total = 0
for order in orders:                 # loop over each order
    if not order["paid"]:            # condition: skip unpaid orders
        continue                     # go to the next iteration
    if order["amount"] > 2_000_000:
        print(f"order {order['id']}: needs manager approval")
    total += order["amount"]

print(f"paid total: {total:,}")      # paid total: 3,200,000
```

چند ابزارِ کوچک که زیاد به‌کارتان می‌آید:

```python
cart = ["keyboard", "mouse"]
total = 3_200_000

# range: counting loop
for i in range(3):
    print(i)          # 0, 1, 2

# while: loop until a condition breaks
attempts = 0
while attempts < 3:
    attempts += 1

# membership test
if "mouse" in cart:
    print("mouse is in the cart")

# compact if/else (ternary)
status = "vip" if total > 1_000_000 else "normal"
print(status)         # vip
```

## توابع: مهم‌ترین پیش‌نیاز

اگر قرار باشد فقط یک بخش از این فصل را عمیق بفهمید، همین است. کلِ فصل دوم به بعد روی توابع سوار می‌شود: **متد چیزی نیست جز تابعی که داخل کلاس زندگی می‌کند.**

تابع، یک تکه‌ی نام‌دارِ قابل‌استفاده‌ی مجدد از منطق است: ورودی می‌گیرد، کاری می‌کند، و (معمولاً) خروجی برمی‌گرداند.

```python
def calculate_discount(amount, percent):
    """Returns the discounted amount."""
    discount = amount * percent / 100
    return amount - discount

final = calculate_discount(500_000, 10)
print(final)     # 450000.0
```

سه نکته‌ی حیاتی درباره‌ی توابع که مستقیم به شیءگرایی وصل می‌شوند:

**۱. پارامترِ پیش‌فرض.** می‌توانید برای پارامترها مقدارِ پیش‌فرض بگذارید؛ در سازنده‌های کلاس (فصل دوم) این را زیاد می‌بینید:

```python
def create_user(username, role="member"):
    return {"username": username, "role": role}

print(create_user("ali"))                  # {'username': 'ali', 'role': 'member'}
print(create_user("sara", role="admin"))   # {'username': 'sara', 'role': 'admin'}
```

**۲. `return` در برابر `print`.** این را جدی بگیرید چون خطای شماره‌ی یکِ تازه‌کارهاست: `print` فقط چیزی را **نمایش** می‌دهد؛ `return` مقدار را **برمی‌گرداند** تا بقیه‌ی برنامه بتواند با آن کار کند. تابعی که فقط print می‌کند، خروجی‌اش `None` است:

```python
def bad_sum(a, b):
    print(a + b)          # only displays

def good_sum(a, b):
    return a + b          # hands the value back

x = bad_sum(2, 3)         # prints 5, but...
print(x)                  # None — nothing was returned!
y = good_sum(2, 3)
print(y * 10)             # 50 — the value is usable
```

**۳. تعداد آرگومانِ نامشخص.** گاهی تابع باید هر تعدادی ورودی بپذیرد. `*args` آن‌ها را به‌صورت تاپل و `**kwargs` آرگومان‌های نام‌دار را به‌صورت دیکشنری جمع می‌کند. فعلاً در همین حد کافی است که وقتی در فصل‌های چهارم و پنجم `*args, **kwargs` را دیدید، غافلگیر نشوید:

```python
def log_event(event_name, *details, **metadata):
    print(event_name, details, metadata)

log_event("login", "ali", "mobile", ip="1.2.3.4")
# login ('ali', 'mobile') {'ip': '1.2.3.4'}
```

و یک حقیقتِ جالب برای آینده: در پایتون خودِ توابع هم شیء هستند — می‌شود آن‌ها را در متغیر ریخت یا به تابعِ دیگری پاس داد. همین ویژگی است که در فصل پنجم «دکوراتورها» را ممکن می‌کند. فعلاً فقط ببینیدش:

```python
def shout(text):
    return text.upper()

f = shout            # no parentheses — the function itself, not its result
print(f("hello"))    # HELLO
```

## ماژول و import

برنامه‌های واقعی در چند فایل نوشته می‌شوند. هر فایلِ `.py` یک **ماژول** است و با `import` از امکاناتش استفاده می‌کنیم:

```python
import math
print(math.sqrt(16))        # 4.0

from datetime import date
print(date.today())         # today's date
```

در طولِ دوره از ماژول‌های کتابخانه‌ی استاندارد (مثل `datetime`، `json`، `enum`) زیاد استفاده می‌کنیم و فصل هجدهم به‌طور کامل به سازمان‌دهیِ ماژول‌ها و پکیج‌ها می‌پردازد.

## مهارت پنهان: خواندن Traceback

این بخش را هیچ دوره‌ای درس نمی‌دهد و اتفاقاً یکی از پرتکرارترین دلایلِ گیرکردنِ تازه‌کارهاست. وقتی برنامه خطا می‌دهد، پایتون یک **Traceback** چاپ می‌کند. بیشترِ مبتدی‌ها با دیدنش وحشت می‌کنند و کلش را نمی‌خوانند؛ در حالی که Traceback بهترین دوستِ شماست — دقیقاً می‌گوید چه شد و کجا.

```python
def get_price(product):
    return product["price"]

item = {"name": "keyboard"}     # price is missing!
get_price(item)
```

```
Traceback (most recent call last):
  File "shop.py", line 5, in <module>
    get_price(item)
  File "shop.py", line 2, in get_price
    return product["price"]
           ~~~~~~~^^^^^^^^^
KeyError: 'price'
```

راهِ خواندنش — **از پایین به بالا**:

1. **خطِ آخر، نوع و پیامِ خطاست:** `KeyError: 'price'` یعنی کلیدِ `price` در دیکشنری نبود.
2. **یکی مانده به آخر، محلِ دقیقِ وقوع:** فایل `shop.py`، خط ۲، داخلِ تابع `get_price`.
3. **خط‌های بالاتر، مسیرِ رسیدن به آنجا:** چه کسی این تابع را صدا زده بود (خط ۵).

از فصل چهارم به بعد که با استثناها کار می‌کنیم و در فصل یازدهم که استثنای سفارشی می‌سازیم، این مهارت طلا می‌شود. عادت کنید: خطا دیدید، اول خطِ آخر، بعد مسیر.

## نقشه‌ی راه دوره: از اینجا به کجا می‌روید؟

حالا که زمین سفت شد، بیایید کلِ مسیر را از بالا ببینیم. دوره سه پرده دارد:

```
پرده‌ی اول: پایه‌ها و تفکر            پرده‌ی دوم: طراحی حرفه‌ای         پرده‌ی سوم: عمق و ساخت
─────────────────────────           ─────────────────────────         ─────────────────────────
۱. فلسفه و تاریخچه                   ۷. Composition و معماری           ۱۴. مدیریت حافظه
۲. کلاس، شیء، self، سازنده           ۸. اصول SOLID                     ۱۵. Descriptorها
۳. تفکر شیءگرا                       ۹. ABC، Protocol، Generics        ۱۶. متاکلاس‌ها
۴. اصول چهارگانه                     ۱۰. مدل‌های داده مدرن              ۱۷. Performance و async
۵. Property و متدهای جادویی          ۱۱. استثناها و سریال‌سازی          ۱۸. ماژول‌ها و پکیج‌ها
۶. ارث‌بری پیشرفته و MRO             ۱۲. الگوهای طراحی                 ۱۹-۲۴. شش پروژه‌ی واقعی
                                     ۱۳. تست‌نویسی
```

منطقِ چینش ساده است: اول **نوشتن** یاد می‌گیرید (پرده‌ی اول)، بعد **خوب نوشتن** (پرده‌ی دوم)، بعد **عمیق فهمیدن و واقعی ساختن** (پرده‌ی سوم). هر فصل روی فصل‌های قبلش سوار می‌شود؛ برای همین پیشنهاد می‌کنم ترتیب را به‌هم نزنید، حتی اگر وسوسه شدید یک‌راست بروید سراغ متاکلاس‌ها.

و شش پروژه‌ی پایانی، جایی است که همه‌چیز به هم گره می‌خورد: سیستم کتابخانه، سیستم پرداخت، سیستم رزرو، و بعد سه پروژه‌ی خاص که کمتر جایی مشابهش را می‌بینید — **ساختنِ ORM شخصی، ساختنِ فریمورک تست، و یک موتورِ اعلانِ event-driven**. وقتی به آنجا برسید، دیگر «کاربرِ» ابزارها نیستید؛ می‌فهمید خودِ Django و pytest از چه جادویی ساخته شده‌اند — و آن جادو را خودتان بلدید.

## چطور از این دوره بیشترین بهره را ببرید؟

چند توصیه، از جنسِ تجربه نه شعار:

**تایپ کنید، اجرا کنید، خراب کنید.** هر بلاکِ کد را خودتان تایپ کنید و اجرا کنید. بعد یک چیزی‌اش را عمداً عوض کنید ببینید چه می‌شود. کسی که کد را «می‌خوانَد» با کسی که کد را «می‌شکنَد و می‌سازد» در پایانِ دوره دو آدمِ متفاوت‌اند.

**تمرین‌ها اختیاری نیستند.** هر فصل یک فایل تمرین دارد و یک فایل پاسخِ تشریحی. اول بدونِ نگاه‌کردن به پاسخ تلاش کنید — حتی اگر غلط، حتی اگر ناقص. مغز فقط چیزهایی را نگه می‌دارد که برایشان عرق ریخته.

**آهسته و پیوسته.** یک فصل در روز (با تمرین‌هایش) آهنگِ خوبی است. سه فصل در یک شب، فقط توهمِ پیشرفت می‌سازد.

**سوال «چرا» را رها نکنید.** هر جا نوشتیم «این Tradeoff دارد»، مکث کنید. حفظ‌کردنِ راه‌حل‌ها شما را برنامه‌نویس نمی‌کند؛ فهمیدنِ «چرا این و نه آن» می‌کند. کلِ این دوره حولِ همین سوال طراحی شده.

## خلاصه

- این دوره پایتون ۳.۱۰+ می‌خواهد و انتظارش از شما فقط پایه‌هاست: متغیر، لیست/دیکشنری، شرط/حلقه، و از همه مهم‌تر **توابع**.
- متغیرها برچسب‌اند نه جعبه؛ همه‌چیز در پایتون شیء است — این دو جمله را نگه دارید، بعداً بارها به‌کارتان می‌آید.
- `return` مقدار را برمی‌گرداند، `print` فقط نمایش می‌دهد.
- Traceback را از پایین به بالا بخوانید: اول نوعِ خطا، بعد محل، بعد مسیر.
- مسیرِ دوره: نوشتن (فصل‌های ۱-۶) → خوب نوشتن (۷-۱۳) → عمیق فهمیدن و ساختن (۱۴-۲۴).

حالا [آزمون خودسنجی](00-getting-ready-exercise.md) را بدهید. اگر نمره‌تان خوب بود، آماده‌اید؛ اگر نه، همین فصل را یک‌بارِ دیگر با حوصله و همراه با تایپ‌کردنِ کدها بروید — و بدانید که این برگشتن، عقب‌ماندن نیست؛ کوتاه‌ترین راه است.

---

**در فصل بعد:** می‌رویم سراغِ سوالی که کمتر دوره‌ای جدی‌اش می‌گیرد: شیءگرایی اصلاً **از کجا آمد و چه مشکلی را حل می‌کند؟** پاسخِ این سوال، بقیه‌ی دوره را از «حفظ‌کردنِ سینتکس» به «فهمیدنِ طراحی» تبدیل می‌کند.
