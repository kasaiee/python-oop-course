# تمرین‌های فصل پانزدهم: Descriptorها

> این تمرین‌ها کل مطالب فصل پانزدهم را پوشش می‌دهند. ابتدا خودتان پاسخ دهید، سپس با فایل پاسخ (`15-descriptors-answer.md`) مقایسه کنید. این فصل «فهم از درون» است — اگر تمرین‌های کدنویسی‌اش را خودتان بنویسید، دیگر هیچ‌وقت property و self برایتان جادو نخواهند بود.

---

## بخش الف: پرسش‌های مفهومی

**۱.** پروتکل descriptor را تعریف کنید: چه متدهایی، و مهم‌تر — descriptor باید کجا تعریف شود تا کار کند و چرا؟

**۲.** سه پارامتر `__get__(self, instance, owner)` هرکدام چیستند؟ چه زمانی `instance` برابر `None` است و پیاده‌سازی استاندارد در آن حالت چه برمی‌گرداند؟

**۳.** `__set_name__` چه مشکلی را حل می‌کند و چه زمانی صدا زده می‌شود؟ بدون آن، تعریف descriptorها چه شکلی می‌شد؟

**۴.** فرق data descriptor و non-data descriptor چیست؟ برای هرکدام دو نمونه از خود پایتون نام ببرید.

**۵.** چرا داده‌ی هر instance را نباید داخل خود descriptor ذخیره کرد؟ این اشتباه شبیه کدام دام فصل دوم است؟

**۶.** جمله‌ی «راز self یک پروتکل است، نه جادو» را باز کنید: وقتی `obj.method()` می‌نویسیم، دقیقاً چه زنجیره‌ای اجرا می‌شود؟

## بخش ب: خواندن کد و پیش‌بینی

**۷.** خروجی این کد چیست و چرا؟ (به ترتیب جست‌وجوی attribute دقت کنید.)

```python
class Ten:
    def __get__(self, instance, owner):
        return 10

class Box:
    value = Ten()

b = Box()
print(b.value)
b.__dict__["value"] = 99
print(b.value)
```

**۸.** حالا یک خط به `Ten` اضافه شده. خروجی چه تغییری می‌کند و چرا؟

```python
class Ten:
    def __get__(self, instance, owner):
        return 10
    def __set__(self, instance, value):
        raise AttributeError("read-only")

class Box:
    value = Ten()

b = Box()
b.__dict__["value"] = 99
print(b.value)
```

**۹.** این کد باگ کلاسیک descriptorها را دارد. اجرایش چه چیز غلطی نشان می‌دهد؟ ریشه‌ی باگ چیست؟

```python
class Field:
    def __get__(self, instance, owner):
        return self._value
    def __set__(self, instance, value):
        self._value = value          # <-- look here

class Order:
    total = Field()

a = Order()
b = Order()
a.total = 100
b.total = 999
print(a.total, b.total)
```

**۱۰.** بدون اجرا: `print(o.pay)` و `print(o.pay())` هرکدام چه چیزی چاپ می‌کنند (حدودی) و چرا `Order.pay` با `o.pay` فرق دارد؟

```python
class Order:
    def pay(self):
        return "paid"

o = Order()
```

## بخش ج: کدنویسی

**۱۱. (اعتبارسنج)** یک descriptor به نام `NonEmpty` بنویسید که فقط رشته‌ی غیرخالی بپذیرد (فضای خالی خالص هم مردود: `"   "` نه). با `__set_name__` نام فیلد را در پیام خطا بیاورید و با کلاس `Article` (فیلدهای `title` و `author`) تستش کنید.

**۱۲. (پارامتردار)** descriptorی به نام `InRange(min_value, max_value)` بنویسید که عدد را فقط در بازه‌ی داده‌شده بپذیرد. با کلاس `Exam` (فیلد `score` بین ۰ تا ۲۰ و `age` بین ۱۸ تا ۹۹) تستش کنید.

**۱۳. (cached_property خانگی)** بدون نگاه‌کردن به متن فصل، `my_cached_property` را بازنویسی کنید: دکوراتوری که بار اول محاسبه می‌کند و دفعات بعد از `__dict__` برمی‌گردد. با یک متد «سنگین» (که هنگام محاسبه پیامی چاپ می‌کند) نشان دهید بار دوم پیامی چاپ نمی‌شود. توضیح دهید چرا نباید `__set__` به آن اضافه کنید.

**۱۴. (ردیاب)** descriptorی به نام `Audited` بنویسید که هر تغییر مقدار را در لیست `instance.history` ثبت کند: هر عنصر یک تاپل `(field_name, old_value, new_value)` (برای اولین مقداردهی، `old_value` برابر `None`). با کلاس `Settings` و دو فیلد تستش کنید.

## بخش د: تحلیل و مقایسه

**۱۵.** جدول را کامل کنید:

| ابزار | data یا non-data؟ | چه چیزی را می‌بندد / چه رفتاری دارد؟ |
|-------|-------------------|----------------------------------------|
| تابع/متد | ؟ | ؟ |
| `property` | ؟ | ؟ |
| `classmethod` | ؟ | ؟ |
| `cached_property` | ؟ | ؟ |

**۱۶.** برای هر سناریو بگویید `@property` کافی است یا descriptor سفارشی می‌ارزد، و چرا:

- الف) کلاس `Circle` باید `area` را از روی شعاع حساب کند.
- ب) دوازده مدل داده‌ی مختلف، هرکدام چند فیلد «رشته‌ی غیرخالی با سقف طول» دارند.
- ج) فیلد `email` فقط در یک کلاس `User` اعتبارسنجی می‌خواهد.
- د) می‌خواهید کتابخانه‌ای بدهید که کاربرانش مدل را با `name = StringField(max_length=50)` تعریف کنند.

## بخش ه: پرسش تحلیلی (تفکر انتقادی)

**۱۷.** همکاری استدلال می‌کند: «descriptor فقط پیچیده‌بازی است؛ همان اعتبارسنجی را می‌شود با یک متد `validate()` که در `__init__` صدا زده شود انجام داد.» این استدلال را ارزیابی کنید: کجاها حق با اوست و کجاها راه‌حلش سوراخ دارد؟ (راهنمایی: تغییر مقدار *بعد از* ساخت، تکرار منطق بین کلاس‌ها، و «همه‌ی مسیرها از یک در».)
