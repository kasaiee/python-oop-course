## بخش الف: کپسول‌سازی

**۱.** کپسول‌سازی یعنی بسته‌بندیِ داده و رفتارِ مربوط در یک واحد، و **کنترلِ دسترسی** به آن داده. مشکلی که حل می‌کند: بدونِ کنترل، هرکسی می‌تواند شیء را به حالتِ نامعتبر ببرد. در کدِ مثال، `acc.balance = -5000` مستقیماً موجودی را منفی می‌کند و هیچ‌کس جلویش را نمی‌گیرد — یک فاجعه. با کپسول‌سازی، داده را داخلی می‌کنیم (`_balance`) و فقط از راهِ متدهای کنترل‌شده اجازه‌ی تغییر می‌دهیم:

```python
class BankAccount:
    def __init__(self, balance):
        self._balance = balance

    def withdraw(self, amount):
        if amount > self._balance:
            raise ValueError("insufficient balance")
        self._balance -= amount
```

حالا هر تغییرِ موجودی از فیلترِ قوانینِ کلاس عبور می‌کند.

**۲.**

| نام‌گذاری | نامِ سطح | معنا |
|----------|---------|------|
| `name` | Public | آزادانه قابلِ استفاده |
| `_name` | Protected | «داخلی است؛ لطفاً از بیرون دست نزن» (فقط قرارداد) |
| `__name` | Private | Name Mangling فعال می‌شود (تقریباً پنهان) |

**۳.** Name Mangling یعنی پایتون نامِ `__x` را درونِ کلاس دستکاری می‌کند: `__pin` در کلاسِ `Account` در پشتِ صحنه به `_Account__pin` تبدیل می‌شود. پس `a.__pin` خطا می‌دهد، اما `a._Account__pin` هنوز کار می‌کند — یعنی این یک مکانیزمِ امنیتیِ **واقعی نیست**؛ فقط دسترسی را سخت‌تر می‌کند. هدفِ اصلیِ Name Mangling، **جلوگیری از تصادمِ نام در ارث‌بری** است (که یک زیرکلاس سهواً attribute والد را بازنویسی نکند)، نه پنهان‌کاریِ امنیتی.

**۴.** یعنی «اینجا همه‌ی ما بزرگسالیم». پایتون به‌جای اینکه با قفل و اجبار جلوی دسترسی به داده‌ی داخلی را بگیرد، به مسئولیت‌پذیریِ برنامه‌نویس تکیه می‌کند. **Tradeoff:** جاوا با `private` اجباری امنیتِ بیشتری می‌دهد اما انعطافِ کمتری (گاهی برای تست یا کارهای خاص، به آن دسترسیِ ممنوع نیاز دارید و نمی‌توانید). پایتون آزادی و انعطاف می‌دهد اما مسئولیت را به شما می‌سپارد. هیچ‌کدام مطلقاً بهتر نیستند؛ دو فلسفه‌ی متفاوت‌اند.

## بخش ب: انتزاع و ارث‌بری

**۵.**
- **کپسول‌سازی** درباره‌ی **کنترلِ دسترسی** است و به این سوال پاسخ می‌دهد: «چه کسی اجازه دارد این داده را ببیند یا تغییر دهد؟»
- **انتزاع** درباره‌ی **ساده‌سازیِ رابط** است و به این سوال پاسخ می‌دهد: «کاربر چه چیزی را باید بداند و چه چیزی را نه؟»

این دو معمولاً با هم کار می‌کنند (داده را پنهان می‌کنیم و یک رابطِ ساده بیرون می‌دهیم) اما یکی نیستند: یکی «محافظت» است، دیگری «ساده‌سازی».

**۶.** ارث‌بری رابطه‌ی **«یک نوعِ...» (Is-A)** می‌سازد: «مدیر یک نوع کارمند است»، پس `Manager(Employee)`. مشکلی که حل می‌کند: **جلوگیری از تکرارِ کد** — رفتارِ مشترک را یک‌بار در والد می‌نویسیم. **کِی نباید:** وقتی رابطه در واقع **Has-A («دارد»)** است، نه Is-A. مثلاً «ماشین یک موتور دارد»؛ پس `Car` نباید از `Engine` ارث ببرد، بلکه باید یک `Engine` را در خود نگه دارد (Composition، فصل هفتم).

**۷.** `super()` به پیاده‌سازیِ کلاسِ والد دسترسی می‌دهد.
- **Override کامل:** والد را کاملاً کنار می‌گذاریم و از نو می‌نویسیم.
- **Override توسعه‌ای:** رفتارِ والد را با `super()` می‌گیریم و چیزی به آن می‌افزاییم.

```python
class Employee:
    def describe(self):
        return f"Name: {self.name}"

class Manager(Employee):
    def describe(self):                  # extending
        base = super().describe()
        return f"{base} | role: manager"
```

انتخاب بستگی دارد به اینکه رفتارِ والد هنوز ارزش دارد (توسعه‌ای) یا باید کاملاً عوض شود (کامل).

**۸.**
- `__str__`: نمایشِ **خوانا برای کاربر**؛ هنگامِ `print(obj)` یا `str(obj)`.
- `__repr__`: نمایشِ **دقیق برای برنامه‌نویس** (ایدئال: بتوان با آن شیء را بازساخت)؛ هنگامِ `repr(obj)` یا تایپِ شیء در کنسول.

وقتی شیء داخلِ یک کانتینر است (مثلِ `print([m])`)، پایتون از **`__repr__`** استفاده می‌کند. قاعده: همیشه دست‌کم `__repr__` را تعریف کنید (کمکِ بزرگی به دیباگ است).

**۹.** با ارث‌بری از `Exception`، خطاهای معنادارِ مخصوصِ پروژه می‌سازیم که خواناتر و قابلِ مدیریت‌ترند. مزیتِ **سلسله‌مراتب:** اگر `InsufficientFundsError` و `CardExpiredError` هر دو زیرکلاسِ `PaymentError` باشند، با یک `except PaymentError` می‌توانید **همه‌ی خطاهای پرداخت را یکجا** بگیرید، یا در صورتِ نیاز هرکدام را جداگانه. این انعطاف با یک رشته‌ی خطای ساده ممکن نیست.

## بخش ج: چندریختی

**۱۰.** چندریختی («چند شکلی») یعنی توانایی کار با اشیای مختلف از طریقِ یک رابطِ مشترک، بی‌آنکه نوعِ دقیقشان را بدانید. تفاوت با Override:
- **Override یک مکانیزم است:** زیرکلاس متدِ والد را بازنویسی می‌کند.
- **چندریختی یک نتیجه است:** می‌توانید با انواعِ مختلف یکسان رفتار کنید.

Override یکی از راه‌های رسیدن به چندریختی است، اما خودِ چندریختی هدفِ نهایی است.

**۱۱.** Duck Typing یعنی «اگر مثلِ اردک راه می‌رود و مثلِ اردک صدا می‌کند، اردک است» — مهم **نوعِ** شیء نیست، مهم این است که **رفتارِ لازم** را دارد. `total_area` فقط به این نیاز دارد که هر عضو یک متدِ `area()` داشته باشد؛ اینکه آن عضو `Circle` است یا `Rectangle` یا هر شکلِ آینده، برایش بی‌اهمیت است. برای همین نیازی به دانستنِ نوع ندارد.

**۱۲.**
- `type(obj) == Cls` نوع را **دقیقاً** می‌سنجد؛ ارث‌بری را نمی‌بیند.
- `isinstance(obj, Cls)` علاوه بر خودِ کلاس، **زیرکلاس‌بودن** را هم می‌پذیرد.

مثال: اگر `Dog(Animal)` باشد، `isinstance(dog, Animal)` برابرِ `True` است اما `type(dog) == Animal` برابرِ `False`. چون یک `Dog` واقعاً یک `Animal` هم هست، `isinstance` رفتارِ درست‌ترِ شیءگرا دارد. (هرچند بهترین کد اغلب اصلاً نیازی به بررسیِ نوع ندارد و به Duck Typing تکیه می‌کند.)

## بخش د: کدنویسی و کدخوانی

**۱۳.** خروجی:

```
A
B
C
```

`C().setup()` اول `super().setup()` را صدا می‌زند که به `B.setup` می‌رود؛ آن هم اول `super().setup()` را صدا می‌زند که به `A.setup` می‌رود. `A` چاپ می‌کند `A`، کنترل به `B` برمی‌گردد که `B` را چاپ می‌کند، و در آخر `C` چاپ می‌شود. یعنی زنجیره‌ی `super()` از پایین به بالا می‌رود و از بالا به پایین چاپ می‌شود.

**۱۴.** کلاسِ `Temperature` با کپسول‌سازی:

```python
class Temperature:
    def __init__(self, celsius=0):
        self.set_celsius(celsius)      # use the same validation

    def get_celsius(self):
        return self._celsius

    def set_celsius(self, value):
        if value < -273.15:
            raise ValueError("temperature cannot be below absolute zero")
        self._celsius = value

t = Temperature(25)
print(t.get_celsius())   # 25
t.set_celsius(-300)      # ValueError
```

**راهِ دیگر (پایتونیک‌تر، با `@property`):** به‌جای متدهای `get`/`set`، می‌توان از `@property` استفاده کرد تا کاربر مثلِ یک attribute معمولی با آن کار کند اما اعتبارسنجی همچنان اجرا شود (این ابزار موضوعِ فصل پنجم است):

```python
class Temperature:
    def __init__(self, celsius=0):
        self.celsius = celsius

    @property
    def celsius(self):
        return self._celsius

    @celsius.setter
    def celsius(self, value):
        if value < -273.15:
            raise ValueError("temperature cannot be below absolute zero")
        self._celsius = value

t = Temperature(25)
t.celsius = 30      # like a simple attribute, but with validation
```

این نسخه رابطِ تمیزتری دارد و در پایتون ترجیح داده می‌شود.

**۱۵.** چندریختی با اشکالِ هندسی:

```python
class Circle:
    def __init__(self, r):
        self.r = r
    def area(self):
        return 3.14159 * self.r ** 2

class Rectangle:
    def __init__(self, w, h):
        self.w, self.h = w, h
    def area(self):
        return self.w * self.h

class Triangle:
    def __init__(self, base, height):
        self.base, self.height = base, height
    def area(self):
        return 0.5 * self.base * self.height

def describe_all(shapes):
    for shape in shapes:
        print(f"{type(shape).__name__}: area = {shape.area():.2f}")

describe_all([Circle(2), Rectangle(3, 4), Triangle(6, 8)])
```

**افزودنِ شکلِ جدید:** فقط یک کلاسِ تازه با متدِ `area()` می‌نویسید و آن را به لیست اضافه می‌کنید؛ تابعِ `describe_all` **اصلاً تغییر نمی‌کند**. این همان قدرتِ چندریختی و اصلِ Open/Closed است (فصل هشتم). مقایسه کنید با نسخه‌ی بدِ پر از `if isinstance(...)` که برای هر شکلِ جدید باید تابع را دستکاری می‌کردید.

**۱۶.** کلاسِ صداپذیر با `__call__`:

```python
class Adder:
    def __init__(self, base):
        self.base = base

    def __call__(self, x):
        return self.base + x

add10 = Adder(10)
print(add10(5))          # 15 — as if add10 were a function
print(callable(add10))   # True
```

`__call__` شیء را «قابلِ فراخوانی مثلِ تابع» می‌کند، اما شیء همچنان حالت نگه می‌دارد (`base`). این ترکیبِ «رفتارِ تابع + حالتِ شیء» در Decoratorها و ماشین‌های حالت بسیار کاربردی است.

## بخش ه: مقایسه و پرسشِ تحلیلی

**۱۷.** این جمله **نادرست** است و کپسول‌سازی را با «پنهان‌کردنِ همه‌چیز» اشتباه می‌گیرد. مشکلات:
- `__` (Name Mangling) هدفش امنیت نیست، بلکه جلوگیری از تصادمِ نام در ارث‌بری است؛ استفاده‌ی افراطی از آن، ارث‌بری و تستِ کد را **سخت** می‌کند، چون زیرکلاس‌ها هم به آن attributeها دسترسیِ راحت ندارند.
- کپسول‌سازیِ افراطی کد را دست‌وپاگیر می‌کند؛ گاهی یک attribute واقعاً باید عمومی باشد.
- فلسفه‌ی پایتون «همه بزرگسالیم» است؛ پیش‌فرض باید `_` (قرارداد) باشد، و `__` فقط با دلیلِ مشخص.

پاسخِ درست: کپسول‌سازی ابزارِ **کنترلِ دسترسی** است، نه هدفی که هرچه بیشتر بهتر. هر attribute را به‌اندازه‌ی نیازش محافظت کنید.

**۱۸.** این چهار اصل ابزارند؛ یعنی فقط وقتی به کار می‌روند که مشکلی را حل کنند. مثالِ **ارث‌بریِ بی‌جا:** فرض کنید چون `Stack` (پشته) هم مثلِ `list` عناصر نگه می‌دارد، بنویسیم `class Stack(list): ...`. این وسوسه‌انگیز است اما اشتباه: حالا پشته **همه‌ی متدهای لیست** را به ارث می‌برد (`insert`، `append`، دسترسی با اندیس، `sort`...) و کاربر می‌تواند قواعدِ «فقط از بالا اضافه/حذف کن» را دور بزند — یعنی رابطه واقعاً Is-A نبود.

راهِ درست: **Composition** — پشته یک لیست را **در خود نگه دارد** و فقط متدهای `push`/`pop` را بیرون بدهد:

```python
class Stack:
    def __init__(self):
        self._items = []          # it "has" a list, it is not one
    def push(self, x):
        self._items.append(x)
    def pop(self):
        return self._items.pop()
```

درسِ کلی: پیش از استفاده از هر اصل بپرسید «اینجا چه مشکلی را حل می‌کند؟». اگر پاسخی نداشتید، احتمالاً لازمش ندارید. تفکر به Tradeoff همیشه بالاتر از پیرویِ کورکورانه از اصول است.
