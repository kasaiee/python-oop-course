# درس تکمیلی: تاریخچه‌ی شیءگرایی در پایتون (از نسخه‌ی ۱ تا ۳)

> این یک درسِ **تکمیلی و اختیاری** است. در [فصل اول](01-philosophy-history-oop.md) خلاصه‌ای از این تاریخچه دیدید؛ اینجا آن را گام‌به‌گام و با آرامش باز می‌کنیم. اگر تازه‌کارید، می‌توانید این درس را رد کنید و بعد از یادگیریِ ارث‌بری (فصل چهارم) و ارث‌بری پیشرفته (فصل ششم) به آن برگردید — آن‌وقت خیلی روشن‌تر خواهد بود.

---

## مقدمه

در فصل اول گفتیم پایتون از ابتدا شیءگرا بود، اما در مسیرش یک «نقطه‌ی عطف» مهم داشت. این درس دقیقاً همان داستان است: پایتونِ اولیه چه مشکلی داشت، چطور آن را حل کرد، و چرا این برای شما — حتی به‌عنوان تازه‌کار — گاهی به‌کار می‌آید.

نگران نباشید اگر بعضی اصطلاح‌ها (مثل ارث‌بری، متد، یا MRO) هنوز کاملاً برایتان جا نیفتاده. اینجا فقط می‌خواهیم تصویری کلی بسازیم؛ جزئیاتِ فنیِ هرکدام در فصل‌های مربوطه می‌آید.

---

## چرا باید این تاریخچه را بدانیم؟

سه دلیلِ عملی:

- گاهی به کدِ قدیمی برمی‌خورید که چیزهای عجیبی مثل `class Foo(object):` دارد. اگر ندانید چرا، گیج می‌شوید.
- بعضی رفتارهای امروزِ پایتون (مثلِ اینکه «همه‌چیز یک شیء است») ریشه‌ی تاریخی دارند؛ دانستنِ ریشه، فهم را عمیق‌تر می‌کند.
- وقتی مستنداتِ قدیمی یا پرسش‌وپاسخ‌های اینترنتی را می‌خوانید، بین «سبک قدیم» و «سبک جدید» گیر نمی‌کنید.

اما یک خیالِ راحت: برای نوشتنِ کدِ **امروز** با پایتون ۳، هیچ‌کدام از این‌ها را لازم ندارید. این درس برای «فهمیدن» است، نه برای «به‌کاربستنِ روزمره».

---

## چطور؟ داستان قدم‌به‌قدم

### گام ۱: پایتونِ اولیه (نسخه‌ی ۱.۰، سال ۱۹۹۴)

پایتون از همان اولین نسخه‌ی عمومی، کلاس و شیء داشت. می‌توانستید کلاس تعریف کنید، از آن شیء بسازید، و ارث‌بری کنید. تا اینجا خوب بود.

اما یک مشکلِ ریشه‌ای وجود داشت که فوراً به چشم نمی‌آمد.

### گام ۲: مشکل — دو دنیای جدا

در پایتونِ اولیه، **دو نوع موجودیتِ متفاوت** وجود داشت که با هم ناسازگار بودند:

1. **نوع‌های توکارِ خودِ پایتون** مثل `list`، `dict`، `int` — که در دلِ زبان ساخته شده بودند.
2. **کلاس‌هایی که شما تعریف می‌کردید** — که موجوداتِ سطحِ دیگری بودند.

این جدایی یک نتیجه‌ی آزاردهنده داشت: **نمی‌توانستید از نوع‌های توکار ارث ببرید.** مثلاً اگر می‌خواستید یک «لیستِ سفارشی» بسازید که مثل `list` رفتار کند اما یک قابلیتِ اضافه داشته باشد، راهِ ساده‌ای نبود.

بیایید تصور کنیم می‌خواهیم یک لیست بسازیم که هر بار یک عنصر به آن اضافه می‌شود، یک پیام هم چاپ کند. در پایتونِ امروز ساده است:

```python
# Python 3 today — inheriting from a built-in type works cleanly
class LoggingList(list):
    def append(self, item):
        print(f"adding: {item}")
        super().append(item)

nums = LoggingList()
nums.append(10)      # adding: 10
nums.append(20)      # adding: 20
print(nums)          # [10, 20]
```

در پایتونِ اولیه، همین کارِ ساده دردسر داشت، چون `list` و کلاس‌های کاربر در دو دنیای جدا بودند. این محدودیت، نشانه‌ی یک مشکلِ عمیق‌تر بود.

### گام ۳: نقطه‌ی عطف — New-Style Classes (نسخه‌ی ۲.۲، سال ۲۰۰۱)

خودِ خالقِ پایتون، **گیدو فن روسوم** (Guido van Rossum)، در نسخه‌ی ۲.۲ راه‌حلی به نام **New-Style Classes** («کلاس‌های سبکِ جدید») معرفی کرد که این شکاف را پر کرد. ایده‌ی کلیدی ساده بود: **همه‌چیز باید از یک ریشه‌ی مشترک بیاید** — و آن ریشه، `object` نام گرفت.

مهم‌ترین تغییرهایی که New-Style Classes آورد:

- **هر کلاس از `object` ارث می‌برد** (مستقیم یا غیرمستقیم). این یعنی بالاخره «همه‌چیز یک شیء است» به‌معنای واقعی درست شد.
- **می‌شد از نوع‌های توکار مثل `list` ارث برد** — همان چیزی که قبلاً ممکن نبود.
- ابزارهای قدرتمندی مثل `property` (فصل ۵) و `super()` (فصل ۴) درست کار کردند.
- ترتیبِ جست‌وجوی متدها (MRO) به الگوریتمِ منظم‌ترِ **C3** تغییر کرد (فصل ۶).

### گام ۴: دو سبک در کنار هم (فقط در پایتون ۲)

نکته‌ی مهم این است: در پایتون ۲، هم سبکِ قدیم و هم سبکِ جدید **همزمان** وجود داشتند، و شما باید انتخاب می‌کردید. تفاوت در ظاهر یک چیزِ کوچک بود — نوشتن یا ننوشتنِ `(object)`:

```python
# Python 2 — old-style class (the problematic one)
class Account:
    pass

# Python 2 — new-style class (the correct one)
class Account(object):
    pass
```

در پایتون ۲، این `(object)` **واقعاً فرق ایجاد می‌کرد**: نسخه‌ی دوم رفتارِ درست و مدرن داشت، نسخه‌ی اول رفتارِ قدیمی و مشکل‌دار. به همین دلیل، برنامه‌نویسانِ باتجربه در پایتون ۲ همیشه `(object)` را می‌نوشتند.

### گام ۵: پایتون ۳ — همه‌چیز یکدست شد (سال ۲۰۰۸)

در پایتون ۳، سبکِ قدیمی **کاملاً حذف شد**. حالا همه‌ی کلاس‌ها به‌طور خودکار New-Style هستند. یعنی این دو تعریف دقیقاً یکی‌اند:

```python
# Python 3 — these two are exactly identical
class Account:
    pass

class Account(object):
    pass

print(Account.__bases__)   # (<class 'object'>,)
```

خطِ آخر نشان می‌دهد که حتی وقتی `(object)` را ننوشتید، پایتون خودش `object` را به‌عنوان والد در نظر گرفته است. برای همین امروز نیازی نیست `(object)` بنویسید.

می‌توانید خودتان ببینید که حتی نوع‌های توکار هم از `object` می‌آیند:

```python
# Python 3 — everything traces back to object
print(int.__bases__)     # (<class 'object'>,)
print(str.__bases__)     # (<class 'object'>,)
print(list.__bases__)    # (<class 'object'>,)

print(isinstance(42, object))     # True
print(isinstance("hi", object))   # True
```

این همان «همه‌چیز یک شیء است» است که در فصل اول دیدیم — و حالا می‌دانید از کجا آمده.

---

## این تاریخچه در عمل به چه دردِ شما می‌خورد؟

سه نتیجه‌ی کاملاً عملی:

**۱. کدِ امروز را ساده بنویسید.** در پایتون ۳ هرگز نیازی به نوشتنِ `(object)` ندارید. `class Foo:` کافی و کامل است.

**۲. کدِ قدیمی را بفهمید.** اگر در پروژه‌ای قدیمی (یا آموزشی) به `class Foo(object):` برخوردید، حالا می‌دانید که این یک بازمانده‌ی دورانِ پایتون ۲ است و در پایتون ۳ هیچ اثری ندارد. لازم نیست نگرانش باشید.

**۳. قدرتِ ارث از نوع‌های توکار را بشناسید.** به‌لطفِ همان تحولِ ۲.۲، امروز می‌توانید از `list`، `dict` و... ارث ببرید و نوع‌های سفارشیِ خودتان را بسازید — قابلیتی که در فصل‌های بعد به‌کار می‌آید.

## خلاصه

```
مسیر شیءگرایی در خودِ پایتون
════════════════════════════════════════════════
  Python 1.0 (۱۹۹۴)  ──►  کلاس و شیء دارد، اما دو دنیای جدا
        │                  (نمی‌شد از list ارث برد)
        ▼
  Python 2.2 (۲۰۰۱) ──►  New-Style Classes
        │                  • همه از object ارث می‌برند
        │                  • ارث از نوع‌های توکار ممکن شد
        │                  • property، super()، C3 MRO
        ▼
  Python 3 (۲۰۰۸)   ──►  سبکِ قدیمی حذف شد
                           همه‌ی کلاس‌ها New-Style‌اند
```

| موضوع | نکته‌ی کلیدی |
|-------|--------------|
| مشکلِ اولیه | کلاس‌های کاربر و نوع‌های توکار، دو دنیای جدا |
| New-Style Classes | نسخه‌ی ۲.۲؛ همه‌چیز از `object` |
| `(object)` در پایتون ۲ | فرق ایجاد می‌کرد (جدید در برابر قدیم) |
| `(object)` در پایتون ۳ | هیچ فرقی ندارد؛ خودکار اعمال می‌شود |
| نتیجه‌ی امروز | `class Foo:` کافی است؛ ارث از نوع‌های توکار ممکن است |

**نکته‌ی طلایی:** این درس را نه برای حفظ‌کردن، بلکه برای «آهان، پس برای همین!» بخوانید. تنها چیزی که واقعاً باید به‌خاطر بسپارید این است: در پایتون ۳، همه‌چیز یک شیء است و از `object` می‌آید — و شما لازم نیست کارِ خاصی برایش انجام دهید.
