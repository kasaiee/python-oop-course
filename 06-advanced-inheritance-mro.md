# فصل ششم: ارث‌بری پیشرفته، MRO و وراثت چندگانه

---

## مقدمه

در فصل چهارم با ارث‌بری پایه آشنا شدیم: یک کلاس از یک والد ارث می‌برد. اما پایتون قابلیتی دارد که خیلی زبان‌ها (مثل جاوا) از آن پرهیز می‌کنند: **وراثت چندگانه** — ارث‌بری از چند والد همزمان.

این قابلیت قدرتمند است اما خطرناک. اگر درست استفاده نشود، به یکی از بدترین آشفتگی‌های ممکن در کد منجر می‌شود. در این فصل یاد می‌گیریم چطور مهارش کنیم: مسئله‌ی الماس چیست، MRO و الگوریتم C3 چطور آن را حل می‌کنند، و **Mixin**ها — که کاربردی‌ترین و امن‌ترین شکل استفاده از وراثت چندگانه‌اند.

---

## چرا باید ارث‌بری پیشرفته را یاد بگیریم؟

اگر فقط کد خودتان را بنویسید، شاید هرگز وراثت چندگانه‌ی پیچیده نسازید. اما لحظه‌ای که کد جنگو، DRF یا کتابخانه‌های بزرگ را باز کنید، با کلاس‌هایی روبه‌رو می‌شوید که از چند Mixin ارث برده‌اند. بدون درک این مفاهیم:

- نمی‌فهمید چرا یک متد به‌شکل خاصی صدا زده می‌شود
- در طراحی خودتان به Diamond Problem می‌خورید و گیج می‌شوید
- نمی‌توانید از قدرت Mixinها برای کد ماژولار استفاده کنید
- کدهای فریمورک‌ها برایتان جعبه‌ی سیاه می‌مانند

## وراثت چندگانه

### چرا پایتون این قابلیت را دارد و جاوا ندارد؟

جاوا وراثت چندگانه را **ممنوع** کرده، چون طراحانش آن را منبع پیچیدگی و ابهام می‌دانستند (به‌خصوص Diamond Problem که می‌بینیم). به‌جایش `interface` را گذاشتند که فقط امضای متد دارد، نه پیاده‌سازی.

پایتون راه دیگری رفت: وراثت چندگانه را **اجازه داد** اما یک الگوریتم دقیق (C3) گذاشت تا ابهام را حل کند. این با فلسفه‌ی «همه بزرگسالیم» جور است: ابزار قدرتمند را می‌دهد و مسئولیت درست‌استفاده‌کردن را به شما می‌سپارد.

آیا این خوب است یا بد؟ **هر دو.** وراثت چندگانه امکاناتی می‌دهد که در جاوا سخت‌اند، اما اگر بی‌محابا استفاده شود، فاجعه می‌سازد. کلید، **انضباط** است — که در بخش Mixinها می‌بینیم.

```python
class Serializable:
    def to_json(self):
        import json
        return json.dumps(self.__dict__)

class Comparable:
    def __lt__(self, other):
        return self.value < other.value

class Number(Serializable, Comparable):    # inherited from two parents
    def __init__(self, value):
        self.value = value

n = Number(5)
print(n.to_json())          # {"value": 5}   ← from Serializable
print(Number(3) < Number(7))  # True         ← from Comparable
```

### مسئله‌ی الماس (Diamond Problem)

یک خاطره پیش از تعریف رسمی: یک‌بار در بازبینی کد یک تیم، باگی دیدم که سه روز دنبالش گشته بودند — متد `save` گاهی دو بار اجرا می‌شد و رکورد تکراری می‌ساخت. کد پر از وراثت چندگانه بود و هر کلاس در `save` مستقیماً `ParentClass.save(self)` صدا می‌زد، نه `super().save()`. نتیجه: کلاس مشترک بالای سلسله‌مراتب، از دو مسیر دو بار اجرا می‌شد. این دقیقاً همان چیزی است که الان می‌خواهیم بشکافیم — و وقتی فصل تمام شود، آن باگ سه‌روزه برای شما یک نگاه آب می‌خورد.

مسئله‌ی الماس وقتی رخ می‌دهد که یک کلاس از دو والد ارث ببرد که **هر دو از یک جد مشترک** آمده‌اند. شکلش مثل الماس است:

```
        A
       / \
      B   C
       \ /
        D
```

سوال کلیدی: اگر `A` متدی داشته باشد که `B` و `C` هر دو Override کرده‌اند، وقتی `D` آن را صدا می‌زند، نسخه‌ی `B` اجرا شود یا `C`؟ و اگر هر دو `super()` صدا بزنند، `A` چند بار اجرا می‌شود؟

```python
class A:
    def __init__(self):
        print("A")

class B(A):
    def __init__(self):
        print("B")
        super().__init__()

class C(A):
    def __init__(self):
        print("C")
        super().__init__()

class D(B, C):
    def __init__(self):
        print("D")
        super().__init__()

D()
# Output:
# D
# B
# C
# A   ← only once! not twice
```

نکته‌ی مهم: `A` فقط **یک‌بار** اجرا شد، با اینکه هم `B` و هم `C` از آن ارث می‌برند و هر دو `super()` صدا زدند. این معجزه‌ی MRO و الگوریتم C3 است.

### MRO و الگوریتم C3

**MRO** (Method Resolution Order) ترتیبی است که پایتون برای جست‌وجوی متدها دنبال می‌کند. الگوریتم **C3 Linearization** این ترتیب را طوری می‌سازد که سه قاعده را تضمین کند:

1. زیرکلاس همیشه پیش از والدهایش می‌آید.
2. ترتیب نوشتن والدها در تعریف کلاس حفظ می‌شود (`class D(B, C)` یعنی `B` پیش از `C`).
3. اگر کلاسی چند مسیر داشته باشد، فقط **یک‌بار** و در جای درست می‌آید.

```python
print(D.__mro__)
# (<class 'D'>, <class 'B'>, <class 'C'>, <class 'A'>, <class 'object'>)
```

این ترتیب توضیح می‌دهد چرا خروجی بالا `D → B → C → A` بود. `super()` در هر مرحله «نفر بعدی در MRO» را صدا می‌زند، نه لزوماً والد مستقیم را. پس در `B`، عبارت `super().__init__()` به `C` می‌رود (نفر بعدی در MRO)، نه به `A`. برای همین `A` فقط یک‌بار، در انتها، اجرا می‌شود.

این دقیقاً همان چیزی است که وراثت چندگانه را در پایتون **امن** می‌کند: تا وقتی همه از `super()` استفاده کنند، هر کلاس در زنجیره دقیقاً یک‌بار اجرا می‌شود.

**قاعده‌ی عملی:** در وراثت چندگانه، همیشه از `super()` استفاده کنید، نه از نام مستقیم والد (مثل `A.__init__(self)`). استفاده‌ی مستقیم، زنجیره‌ی MRO را می‌شکند و می‌تواند باعث اجرای چندباره یا حذف یک کلاس شود.

### MRO از صفر: توضیح گام‌به‌گام

> اگر از [فصل اول](01-philosophy-history-oop.md) به اینجا ارجاع داده شده‌اید: خوش‌آمدید. حالا که ارث‌بری را کامل یاد گرفته‌اید، MRO خیلی روشن‌تر خواهد بود. از صفر شروع می‌کنیم.

بیایید ساده‌ترین حالت را ببینیم. وقتی یک کلاس **فقط یک والد** دارد، ماجرا بدیهی است: پایتون اول در خود کلاس دنبال متد می‌گردد، اگر نیافت به والد می‌رود، بعد به والد والد، و در نهایت به `object`.

```python
class Animal:
    def describe(self):
        return "an animal"

class Dog(Animal):
    pass

d = Dog()
print(d.describe())        # "an animal" — not found in Dog, so found in Animal
print([c.__name__ for c in Dog.__mro__])
# ['Dog', 'Animal', 'object']
```

این «مسیر جست‌وجو» همان **MRO** است. با `Dog.__mro__` (یا `Dog.mro()`) می‌توانید دقیقاً ببینید پایتون به چه ترتیبی جست‌وجو می‌کند: اول `Dog`، بعد `Animal`، بعد `object`.

تا وقتی ارث‌بری یک‌خطی است، MRO فقط یک زنجیره‌ی ساده است. **پیچیدگی از جایی شروع می‌شود که وراثت چندگانه وارد می‌شود** — همان مسئله‌ی الماس. اینجاست که الگوریتم C3 لازم می‌شود.

**بیایید ببینیم C3 چطور ترتیب را می‌سازد.** برای همان الماس `D(B, C)` که `B` و `C` از `A` ارث می‌برند، C3 این‌طور فکر می‌کند:

```
گام ۱: از خود D شروع کن.
        ترتیب تا اینجا: [D]

گام ۲: حالا باید والدها (B و C) و اجدادشان را بچینی.
        قاعده: هر کلاس باید پیش از والدهایش بیاید،
        و ترتیب نوشتن (اول B بعد C) حفظ شود.

گام ۳: B را اضافه کن  →  [D, B]
        اما A را هنوز اضافه نکن! چون C هم از A ارث می‌برد،
        و A باید بعد از هر دو B و C بیاید.

گام ۴: C را اضافه کن  →  [D, B, C]

گام ۵: حالا که هم B و هم C آمده‌اند، A را اضافه کن  →  [D, B, C, A]

گام ۶: در نهایت object  →  [D, B, C, A, object]
```

نکته‌ی کلیدی گام ۳: C3 اجازه نمی‌دهد `A` زودتر از `C` بیاید، چون یک قاعده‌ی محکم دارد: **هیچ کلاسی نباید پیش از زیرکلاس‌هایش در فهرست ظاهر شود.** چون `C` یک زیرکلاس `A` است، پس `A` باید بعد از `C` بیاید. همین قاعده است که تضمین می‌کند `A` فقط **یک‌بار** و در **درست‌ترین جای ممکن** (انتها) قرار بگیرد.

بیایید صحتش را با کد ببینیم:

```python
class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass

print([c.__name__ for c in D.mro()])
# ['D', 'B', 'C', 'A', 'object']  — exactly what we built by hand
```

اگر روزی سلسله‌مراتبی بسازید که C3 نتواند ترتیب سازگاری برایش پیدا کند (مثلاً ترتیب والدها متناقض باشد)، پایتون **هنگام تعریف کلاس** خطای `TypeError: Cannot create a consistent method resolution order` می‌دهد — یعنی همان لحظه به شما هشدار می‌دهد، نه بعداً در زمان اجرا.

**جمع‌بندی ذهنی:** MRO فقط «فهرست ترتیب‌دار جست‌وجوی متد» است. برای ارث‌بری ساده یک زنجیره است؛ برای وراثت چندگانه، C3 آن را طوری می‌چیند که سه قاعده‌ی بالا رعایت شوند. هر وقت شک کردید، `ClassName.mro()` را چاپ کنید و ترتیب را با چشم خودتان ببینید.

### تفاوت با سایر زبان‌ها

| زبان | وراثت چندگانه | راه‌حل ابهام |
|------|----------------|--------------|
| Python | ✅ کامل | الگوریتم C3 (MRO) |
| Java | ❌ فقط interface | اصلاً اجازه نمی‌دهد |
| C++ | ✅ دارد | ارث‌بری مجازی (virtual)، دستی‌تر |
| Ruby | تاحدی (Module/Mixin) | ترتیب include |

پایتون و روبی هر دو رویکرد «اجازه بده اما با قاعده» را دارند؛ جاوا رویکرد «ممنوع کن تا امن بماند».

## Mixinها

### Mixin چیست؟

**Mixin** یک کلاس کوچک است که یک قابلیت مشخص و مستقل را فراهم می‌کند و قرار نیست به‌تنهایی نمونه‌سازی شود. آن را به کلاس اصلی «می‌چسبانید» (mix in) تا آن قابلیت را اضافه کنید. Mixin، امن‌ترین و مفیدترین شکل وراثت چندگانه است.

تفاوت با کلاس معمولی: یک Mixin یک موجودیت کامل نیست («کاربر» یا «سفارش»)؛ فقط یک تکه‌رفتار است («قابلیت لاگ‌گرفتن»، «قابلیت سریال‌شدن»).

```python
class JSONMixin:
    """adds JSON conversion capability"""
    def to_json(self):
        import json
        return json.dumps(self.__dict__, ensure_ascii=False)

class TimestampMixin:
    """adds creation-time recording capability"""
    def __init__(self, *args, **kwargs):
        import time
        self.created_at = time.time()
        super().__init__(*args, **kwargs)

class User(TimestampMixin, JSONMixin):
    def __init__(self, name):
        self.name = name
        super().__init__()

u = User("ali")
print(u.to_json())        # {"name": "ali", "created_at": ...}  ← from JSONMixin
print(u.created_at)       # creation time                          ← from TimestampMixin
```

`User` موجودیت اصلی است؛ `JSONMixin` و `TimestampMixin` قابلیت‌های افزوده. هرکدام را می‌توان به هر کلاس دیگری هم چسباند. این ماژولاریتی، جوهر Mixin است.

### چه زمانی از Mixin استفاده کنیم و چه زمانی نه؟

**مناسب است وقتی:** یک قابلیت مستقل و قابل‌استفاده‌ی مجدد دارید که به چند کلاس نامرتبط باید اضافه شود (لاگ، سریال‌سازی، مقایسه‌پذیری).

**مناسب نیست وقتی:** رابطه واقعاً Is-A است (آن‌وقت ارث‌بری معمولی)، یا وقتی قابلیت را می‌شود با ترکیب (Composition، فصل بعد) تمیزتر افزود.

قاعده‌ی نام‌گذاری: Mixinها را با پسوند `Mixin` نام‌گذاری کنید تا نیت‌تان روشن باشد.

### Mixin در برابر کلاس انتزاعی (ABC)

این دو گاهی اشتباه گرفته می‌شوند:

| | Mixin | کلاس انتزاعی (ABC) |
|---|-------|---------------------|
| هدف | **افزودن** پیاده‌سازی آماده | **الزام** به پیاده‌سازی یک قرارداد |
| متدها | معمولاً پیاده‌سازی‌شده | معمولاً انتزاعی (خالی) |
| استفاده | «این قابلیت را بگیر» | «این متدها را باید بسازی» |

Mixin می‌گوید «من این کار را برایت انجام می‌چهاردهم»؛ ABC می‌گوید «تو باید این کار را انجام دهی». ABCها را در فصل نهم کامل می‌بینیم.

## مفاهیم پیشرفته

### __init_subclass__: پیکربندی زیرکلاس‌ها

`__init_subclass__` (از پایتون ۳.۶) یک قلاب است که **هنگام تعریف هر زیرکلاس** صدا زده می‌شود — نه هنگام ساخت شیء. با آن می‌توانید زیرکلاس‌ها را در لحظه‌ی تعریف بررسی یا پیکربندی کنید، بدون نیاز به متاکلاس (که پیچیده‌تر است).

```python
class Plugin:
    registry = []

    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        Plugin.registry.append(cls)      # each subclass is registered automatically
        print(f"Plugin registered: {cls.__name__}")

class EmailPlugin(Plugin):
    pass

class SMSPlugin(Plugin):
    pass

# output at definition time (not object creation):
# Plugin registered: EmailPlugin
# Plugin registered: SMSPlugin
print([c.__name__ for c in Plugin.registry])   # ['EmailPlugin', 'SMSPlugin']
```

این الگوی «ثبت خودکار» در سیستم‌های پلاگین‌محور بسیار پرکاربرد است.

### الزام پیاده‌سازی با __init_subclass__

می‌توان با آن، زیرکلاس‌ها را وادار به رعایت قرارداد کرد:

```python
class Handler:
    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        if not hasattr(cls, "handle"):
            raise TypeError(f"{cls.__name__} must have a handle method")

class GoodHandler(Handler):
    def handle(self): ...

# class BadHandler(Handler):    # TypeError: BadHandler must have a handle method
#     pass
```

### __init_subclass__ در برابر متاکلاس

هر دو رفتار ساخت کلاس را کنترل می‌کنند، اما `__init_subclass__` بسیار ساده‌تر است. **قاعده:** اگر فقط می‌خواهید هنگام تعریف زیرکلاس کاری انجام دهید، `__init_subclass__` کافی است و نیازی به متاکلاس ندارید. متاکلاس (فصل شانزدهم) را فقط برای کنترل عمیق‌تر فرایند ساخت کلاس نگه دارید.

### کلاس توکار Meta

الگوی رایجی که در جنگو و کتابخانه‌های دیگر می‌بینید، تعریف یک کلاس داخلی `Meta` برای نگه‌داری متادیتاست:

```python
class Model:
    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        meta = getattr(cls, "Meta", None)
        cls._table = getattr(meta, "table_name", cls.__name__.lower())

class User(Model):
    class Meta:                       # metadata separate from the main logic
        table_name = "users"

class Product(Model):
    pass                              # without Meta → default

print(User._table)      # users     ← read from Meta
print(Product._table)   # product   ← default
```

مزیت این الگو: متادیتای پیکربندی (نام جدول، ترتیب پیش‌فرض و...) از منطق اصلی کلاس جدا می‌ماند و کد خواناتر می‌شود. اگر تا حالا مدل جنگو نوشته‌اید، `class Meta` را دیده‌اید — حالا می‌دانید پشتش چه می‌گذرد.

## خلاصه

```
وراثت چندگانه در پایتون
═══════════════════════════════════════════
  مسئله‌ی الماس:      A
                     / \
                    B   C
                     \ /
                      D
        │
        ▼
  راه‌حل: MRO با الگوریتم C3
        │
        ▼
  D → B → C → A → object  (هر کلاس یک‌بار)
        │
        ▼
  کلید امنیت: همیشه super() نه نام مستقیم والد
```

| مفهوم | نکته‌ی کلیدی |
|-------|--------------|
| وراثت چندگانه | ارث از چند والد؛ قدرتمند اما نیازمند انضباط |
| Diamond Problem | جد مشترک؛ C3 آن را با اجرای یک‌باره حل می‌کند |
| MRO / C3 | ترتیب جست‌وجوی متد؛ `__mro__` نمایشش می‌دهد |
| `super()` | نفر بعدی در MRO را صدا می‌زند، نه لزوماً والد مستقیم |
| Mixin | کلاس کوچک افزاینده‌ی قابلیت؛ امن‌ترین شکل وراثت چندگانه |
| `__init_subclass__` | قلاب هنگام تعریف زیرکلاس؛ جایگزین ساده‌ی متاکلاس |
| کلاس `Meta` | جداسازی متادیتا از منطق اصلی |

**نکته‌ی طلایی:** وراثت چندگانه شمشیر دولبه است. Mixinهای کوچک و مستقل، استفاده‌ی درست آن‌اند؛ سلسله‌مراتب‌های عمیق و درهم، سوءاستفاده. قبل از هر ارث‌بری چندگانه از خود بپرسید: «آیا این واقعاً وراثت است یا فقط می‌خواهم قابلیتی اضافه کنم؟» اگر پاسخ دومی است، شاید Composition (فصل بعد) انتخاب بهتری باشد.

## پرسش‌های مصاحبه

**۱. «MRO چیست و پایتون با Diamond Problem چه می‌کند؟»** — ترتیب قطعی جست‌وجوی متدها که الگوریتم C3 از سلسله‌مراتب می‌سازد؛ در الماس، کلاس مشترک بالایی فقط یک‌بار و در جای درست می‌آید. کد اثبات: `D.__mro__`. جواب ممتاز اضافه می‌کند که `super()` یعنی «بعدی MRO»، نه «پدر من» — همین تمایز، رفتار زنجیره‌ای چندوراثتی را توضیح می‌دهد.

**۲. «Mixin چیست و چه فرقی با ارث‌بری معمولی دارد؟»** — کلاس کوچک تک‌رفتاره که به‌تنهایی معنا ندارد و برای «افزودن قابلیت» کنار کلاس اصلی می‌نشیند (JSONMixin، LogMixin)؛ رابطه‌اش Is-A واقعی نیست، «Can-Do» است. تفاوت با ABC را هم بگویید: Mixin پیاده‌سازی *می‌دهد*، ABC پیاده‌سازی *می‌خواهد*.

**۳. «__init_subclass__ به چه دردی می‌خورد و چه چیزی را از دوش متاکلاس برداشت؟»** — هوک زمان تعریف زیرکلاس: ثبت خودکار (registry/پلاگین)، اعتبارسنجی زیرکلاس، تزریق پیکربندی. تا قبل از پایتون ۳.۶ این‌ها بهانه‌ی اصلی متاکلاس‌نویسی بودند؛ حالا قاعده این است: تعریف زیرکلاس → `__init_subclass__`؛ کنترل عمیق‌تر ساخت کلاس → متاکلاس.
