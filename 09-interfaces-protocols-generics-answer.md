## بخش الف: Abstract Base Classes (ABC)

**۱.** ABC کلاسی است که قرار **نیست** مستقیماً نمونه‌سازی شود؛ فقط یک **قرارداد** تعریف می‌کند که زیرکلاس‌ها باید رعایتش کنند (متدهای انتزاعیِ بدون بدنه که زیرکلاس موظف به پیاده‌سازی‌شان است). مزیتِ اصلی، **شکستِ زودهنگام** است: بدونِ ABC، اگر زیرکلاس متدی را پیاده نکند، خطا فقط هنگامِ **فراخوانیِ** آن متد رخ می‌دهد (شاید خیلی دیر، در تولید). با ABC، خطا هنگامِ **ساختِ شیء** رخ می‌دهد — بسیار زودتر و امن‌تر.

**۲.** بله، `BrokenGateway()` خطا می‌دهد — **هنگامِ ساختِ شیء**، نه هنگامِ فراخوانیِ `refund`. چون `BrokenGateway` متدِ انتزاعیِ `refund` را پیاده نکرده، پایتون همان لحظه‌ی `BrokenGateway()` یک `TypeError` می‌دهد (چیزی شبیهِ *Can't instantiate abstract class BrokenGateway with abstract method refund*). این دقیقاً همان «شکستِ زودهنگام» است: خطا پیش از آنکه کد به جای خطرناکی برسد، آشکار می‌شود.

**۳.**
- **مناسب است وقتی:** می‌خواهید یک خانواده از کلاس‌ها را (که خودتان می‌سازید) وادار به رعایتِ یک قرارداد کنید، و رابطه‌ی ارث‌بریِ اسمی («این یک نوعِ Gateway است») معنا دارد.
- **مناسب نیست وقتی:** فقط می‌خواهید بگویید «هر چیزی که این متدها را دارد قبول است» بدونِ الزام به ارث‌بری — که کارِ Protocol است.

## بخش ب: Protocolها

**۴.** Protocol (از پایتون ۳.۸) اجازه می‌دهد اینترفیس را بر اساسِ **ساختار** تعریف کنید، نه ارث‌بری. «Structural Subtyping» یعنی: «هر شیئی که این متدها را داشته باشد، این Protocol را برآورده می‌کند» — بدونِ نیاز به ارث‌بری از چیزی. تفاوت با ABC: ABC **اسمی** (Nominal) است (باید صریحاً ارث ببری تا زیرنوع حساب شوی)، اما Protocol **ساختاری** است (فقط داشتنِ شکلِ درست کافی است). Protocol در واقع همان Duck Typing است، اما در سطحِ تایپ‌هینتر و قابل‌بررسی توسطِ ابزارهایی مثل mypy.

**۵.**

| ویژگی | ABC | Protocol |
|-------|-----|----------|
| نوعِ رابطه | اسمی (Nominal) — باید ارث ببری | ساختاری (Structural) — فقط شکل مهم است |
| الزامِ ارث‌بری | بله (`class X(MyABC)`) | خیر |
| مناسبِ | خانواده‌ی کلاس‌های تحتِ کنترلِ شما | تطبیقِ کلاس‌هایی که کنترلشان ندارید |

**۶.** پیش‌فرض، Protocolها فقط برای تایپ‌چکِ ایستا هستند و `isinstance` رویشان کار نمی‌کند؛ `@runtime_checkable` این را فعال می‌کند تا بتوانید در زمانِ اجرا با `isinstance` بررسی کنید که شیء شکلِ Protocol را دارد یا نه. **محدودیت/Tradeoff:** این `isinstance` فقط **وجودِ** متدها را چک می‌کند، نه امضا یا رفتارشان را. یعنی شیئی که متدِ `close` دارد اما با پارامترهای اشتباه، باز هم `True` می‌شود. پس صددرصد امن نیست؛ برای بررسیِ دقیق باید به تایپ‌چکِ ایستا (mypy) تکیه کنید.

**۷.** **Protocol** مناسب است. چون شما کدِ `open()` استاندارد یا کلاس‌های کتابخانه‌های دیگر را **کنترل نمی‌کنید** و نمی‌توانید مجبورشان کنید از یک ABC ارث ببرند. با Protocol فقط کافی است آن شیء متدِ `read()` را داشته باشد — بدونِ هیچ ارث‌بری. اگر ABC استفاده می‌کردید، همه باید از یک پایه ارث می‌بردند که برای کدِ خارج از کنترلِ شما غیرممکن است.

## بخش ج: ژنریک‌ها (Generics)

**۸.** بدونِ ژنریک، یا باید برای هر نوع موجودیت یک کلاسِ `Repository` جدا بنویسید (تکرارِ کد)، یا از `Any` استفاده کنید. مشکلِ `Any` این است که تایپ‌چک را **خاموش** می‌کند: وقتی `get(0)` را صدا می‌زنید، تایپ‌چکر نمی‌داند خروجی `User` است یا `Product`، پس هیچ کمکی نمی‌کند و اگر اشتباهی نوعِ غلط اضافه کنید هشدار نمی‌دهد. ژنریک این مشکل را حل می‌کند: یک کلاس، همه‌ی انواع، اما با حفظِ ایمنیِ نوع.

**۹.** Repository ژنریک:

```python
from typing import TypeVar, Generic

T = TypeVar("T")

class Repository(Generic[T]):
    def __init__(self):
        self._items: list[T] = []

    def add(self, item: T) -> None:
        self._items.append(item)

    def get(self, index: int) -> T:
        return self._items[index]

class User: ...
class Product: ...

user_repo: Repository[User] = Repository()
product_repo: Repository[Product] = Repository()

user_repo.add(User())
u = user_repo.get(0)      # the type checker knows u is a User, not Any
```

مزیت: یک کلاس نوشتیم که با هر نوعی کار می‌کند، اما `user_repo.get(0)` را تایپ‌چکر به‌عنوانِ `User` می‌شناسد و اگر محصولی اشتباه اضافه کنید هشدار می‌دهد.

**۱۰.** `TypeVar` با `bound` نوعِ پارامتری را به یک نوعِ خاص و زیرکلاس‌هایش **محدود** می‌کند. مشکلی که حل می‌کند: به تایپ‌چکر اطمینان می‌دهد که `T` قابلیت‌های خاصی دارد. در مثالِ `bound=Comparable`، تضمین می‌شود هر `T` حتماً متدِ `compare` را دارد، پس `item.compare(...)` مجاز است. **بدونِ `bound`،** تایپ‌چکر `T` را «هر نوعی» فرض می‌کند و نمی‌داند `compare` وجود دارد، پس هنگامِ `item.compare(...)` هشدار می‌دهد.

## بخش د: تایپ‌هینترهای پیشرفته و کدخوانی

**۱۱.** این دو **دقیقاً یکسان‌اند**: `Optional[str]` معادلِ `str | None` است (سینتکسِ `|` از پایتون ۳.۱۰ مدرن‌تر و رایج‌تر است). نکته‌ی مهم: `Optional[str]` به‌معنای «آرگومانِ اختیاری» **نیست**؛ یعنی «مقدار می‌تواند `str` یا `None` باشد». (اختیاری‌بودنِ آرگومان با دادنِ مقدارِ پیش‌فرض تعیین می‌شود، نه با `Optional`.) این یک اشتباهِ رایج است.

**۱۲.** `Any` یعنی «هر نوعی، بی‌بررسی»؛ استفاده‌اش تایپ‌چک را برای آن مقدار **خاموش** می‌کند. در کدِ نمونه، چون `data` از نوعِ `Any` است، تایپ‌چکر فرض می‌کند `data` هر کاری می‌تواند بکند، پس `data.anything()` را بدونِ هشدار می‌پذیرد — حتی اگر چنین متدی وجود نداشته باشد. جایگزینِ امن‌تر: اگر واقعاً نوع نامشخص است از `object` استفاده کنید (که شما را مجبور می‌کند پیش از استفاده نوع را چک کنید)، یا بهتر، یک **Protocol** که دقیقاً قابلیت‌های موردنیاز را توصیف می‌کند.

**۱۳.** `@overload` وقتی به کار می‌رود که یک تابع بسته به نوعِ ورودی، خروجیِ متفاوتی دارد؛ با آن می‌توانید به تایپ‌چکر بگویید «اگر ورودی `int` بود خروجی `int` است، اگر `str` بود خروجی `str`». در **زمانِ اجرا**، تعریف‌های `@overload` نادیده گرفته می‌شوند و فقط **پیاده‌سازیِ واقعیِ آخر** (تابعِ بدونِ دکوراتور) اجرا می‌شود. یعنی `@overload` صرفاً برای تایپ‌چکر است، نه رفتارِ زمانِ اجرا.

**۱۴.** چون `Iterable[int]` **کمترین قابلیتِ لازم** را می‌خواهد: هر چیزی که بتوان رویش `for` زد — `list`، `tuple`، `set`، `generator` و... همه قبول می‌شوند. اما `list[int]` تابع را به یک نوعِ مشخص محدود می‌کند و مثلاً یک `tuple` یا `generator` را رد می‌کند. این با اصلِ **Interface Segregation** (فصل هشتم) هم‌راستاست: «کمترین قابلیتِ لازم را بخواه، نه بیشتر». تابعِ منعطف‌تر، قابلِ استفاده‌ی مجددتر است.

## بخش ه: پرسشِ تحلیلی

**۱۵.** چون Protocol با روحِ **Duck Typing**ِ پایتون سازگارتر است: «مهم نیست چه هستی، مهم است چه می‌توانی بکنی». Protocol ارث‌بریِ اجباری تحمیل نمی‌کند، پس:
- با کلاس‌هایی که کنترلشان ندارید هم کار می‌کند.
- کد را کمتر به یک سلسله‌مراتبِ ارث‌بریِ خاص گره می‌زند (وابستگیِ کمتر).
- انعطاف‌پذیرتر است و به فلسفه‌ی پایتون نزدیک‌تر.

ABC را وقتی نگه دارید که واقعاً می‌خواهید یک خانواده‌ی تحتِ کنترلِ خودتان را با ارث‌بری و بررسیِ هنگامِ ساخت، به قرارداد ملزم کنید.

**۱۶.** این جمله **نادرست** است. تایپ‌هینترها در زمانِ اجرا **اجرا نمی‌شوند** و هیچ بررسیِ خودکاری انجام نمی‌دهند؛ پایتون آن‌ها را عملاً نادیده می‌گیرد. ارزشِ واقعی‌شان در **ابزارهای تایپ‌چکِ ایستا** (مثلِ mypy یا Pyright) است که کد را **پیش از اجرا** تحلیل می‌کنند و باگ‌های نوع را می‌گیرند. بدونِ چنین ابزاری، تایپ‌هینت فقط یک **مستندسازیِ** خوانا برای برنامه‌نویس است (که آن هم بی‌ارزش نیست، اما «امنیتِ زمانِ اجرا» نیست). اگر ایمنیِ واقعیِ زمانِ اجرا می‌خواهید، باید صریحاً اعتبارسنجی کنید (مثلاً با `isinstance` یا کتابخانه‌هایی مثل Pydantic).
