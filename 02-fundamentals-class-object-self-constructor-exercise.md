## بخش الف: پرسش‌های مفهومی

**۱.** تفاوت «کلاس» و «شیء» را با استعاره‌ی نقشه و خانه توضیح دهید. در کد زیر، خروجی `type(u1)` و `u1 is u2` چه خواهد بود و هرکدام چه چیزی را ثابت می‌کند؟

```python
u1 = User("ali")
u2 = User("sara")
```

**۲.** `self` دقیقاً چیست؟ چرا می‌گوییم `u.greet()` فقط یک میان‌بُر برای `User.greet(u)` است؟ آیا `self` یک کلمه‌ی کلیدی جادویی پایتون است؟

**۳.** مدل ذهنی «متغیرها برچسب‌اند نه جعبه» و مفهوم Heap چه چیزی درباره‌ی `self` به ما می‌گویند؟ چرا وقتی داخل یک متد `self.username` را عوض می‌کنیم، تغییر **ماندگار** است و روی «کپی» اعمال نمی‌شود؟

**۴.** تفاوت Instance Variable و Class Variable چیست؟ هرکدام معمولاً در کجای کلاس تعریف می‌شوند و چه زمانی باید از هرکدام استفاده کنیم؟

**۵.** تفاوت `__init__` و `__new__` چیست؟ کدام‌یک شیء را «می‌سازد» و کدام‌یک آن را «مقداردهی» می‌کند؟ در چه موارد نادری مجبور به نوشتن `__new__` می‌شویم؟

## بخش ب: دام‌ها و کدخوانی

**۶. (کدخوانی)** خروجی این کد چیست؟ چرا `a.count` برابر ۱ می‌شود اما `Counter.count` برابر ۰ باقی می‌ماند؟ در توضیح خود به معنای دقیق `self.count += 1` اشاره کنید.

```python
class Counter:
    count = 0
    def increment(self):
        self.count += 1

a = Counter()
a.increment()
print(a.count)
print(Counter.count)
```

**۷. (کدخوانی + رفع اشکال)** خروجی این کد چیست و چرا این یک «دام» محسوب می‌شود؟ کد را طوری اصلاح کنید که هر تیم لیست اعضای مستقل خودش را داشته باشد.

```python
class Team:
    members = []
    def add(self, name):
        self.members.append(name)

t1 = Team()
t2 = Team()
t1.add("ali")
print(t2.members)
```

**۸.** تفاوت «متد» و «تابع معمولی» در چیست؟ چرا متد به `self` نیاز دارد اما تابع مستقل نه؟

## بخش ج: کدنویسی

**۹.** یک کلاس `Rectangle` بنویسید که در سازنده `width` و `height` بگیرد و دو متد داشته باشد: `area()` (مساحت) و `perimeter()` (محیط). سپس یک مستطیل ۳×۴ بسازید و هر دو مقدار را چاپ کنید.

**۱۰. (کاربرد Class Variable)** یک کلاس بنویسید که بشمارد تا کنون **چند نمونه** از آن ساخته شده است. راهنمایی: از یک Class Variable به‌عنوان شمارنده‌ی مشترک استفاده کنید که در `__init__` افزایش می‌یابد.

**۱۱. (چالش اصلی)** یک کلاس `BankAccount` بنویسید که:
- در سازنده `owner` و `balance` (با مقدار پیش‌فرض صفر) بگیرد،
- متد `deposit(amount)` که مبلغ منفی یا صفر را با `ValueError` رد کند،
- متد `withdraw(amount)` که هم مبلغ نامعتبر و هم برداشت بیش از موجودی را با `ValueError` رد کند،
- به هر حساب یک `account_id` یکتا بدهد (با استفاده از یک شمارنده‌ی مشترک).

**۱۲. (کدخوانی)** آیا کد زیر کار می‌کند؟ خروجی دو `print` چیست؟ آیا `email` روی نمونه‌های دیگر `User` هم ظاهر می‌شود؟

```python
u = User("ali")
u.email = "ali@example.com"
print(u.email)
other = User("sara")
print(hasattr(other, "email"))
```

## بخش ه: پرسش تحلیلی (تفکر انتقادی)

**۱۵.** این جمله را با توجه به مفهوم Tradeoff نقد کنید: «چون اشیای بی‌حالت امن‌تر و ساده‌تر برای تست هستند، بهتر است همیشه همه‌ی کلاس‌ها را بی‌حالت طراحی کنیم.»

**۱۶.** نویسنده می‌گوید بیشتر باگ‌های تازه‌کارها از درک نادرست اینکه «داده **کجا** زندگی می‌کند و `self` **به چه** اشاره می‌کند» سرچشمه می‌گیرد. چرا این دو سوءتفاهم این‌قدر پرخطرند؟ یک مثال بزنید که درک نادرست آن، به یک باگ واقعی منجر شود.
