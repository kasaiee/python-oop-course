# تمرین‌های فصل هجدهم: ماژول‌ها، پکیج‌ها و شیءگرایی

> این تمرین‌ها سازمان‌دهی فیزیکی کد (ماژول، پکیج، `__init__.py`) و رابطه‌اش با شیءگرایی را پوشش می‌دهند. ابتدا خودتان پاسخ دهید، سپس با `18-modules-packages-answer.md` مقایسه کنید.

---

## بخش الف: ماژول و پکیج

**۱. (کدخوانی)** ماژول چیست و چرا می‌گوییم رفتار Singleton دارد؟ در کد زیر، جمله‌ی «config بارگذاری شد» چند بار چاپ می‌شود و چرا؟

```python
# config.py
print("config loaded")
settings = {"debug": True}

# main.py
import config
import config
```

**۲.** پکیج چیست و چه تفاوتی با ماژول دارد؟ نقش فایل `__init__.py` چیست؟

**۳.** `__all__` چه کاری انجام می‌دهد؟ چه ارتباطی با `from module import *` دارد و چرا حتی بدون استفاده از `import *` هم تعریفش می‌تواند مفید باشد؟

**۴.** تفاوت دو سبک `import shop.models` و `from shop.models import User` چیست؟ Tradeoff هرکدام کدام است؟

## بخش ب: کاربردها و طراحی

**۵.** چرا می‌گوییم «ماژول، پایتونی‌ترین شکل Singleton است»؟ اگر به یک نمونه‌ی مشترک سراسری نیاز داشته باشید، چرا اغلب لازم نیست یک کلاس Singleton بنویسید؟

**۶.** الگوی Factory در سطح ماژول (مثل `create_notifier`) چه مزیتی دارد؟ چطور به کاربر اجازه می‌دهد از جزئیات کلاس‌های داخلی بی‌نیاز بماند و با کدام اصل SOLID هم‌راستاست؟

**۷.** چیدمان یک پکیج شیءگرا (فایل‌های `models.py`، `services.py`، `repositories.py`، `exceptions.py`) چطور به معماری لایه‌ای فصل هفتم و اصل Single Responsibility مربوط است؟

## بخش ج: کدنویسی و تشخیص

**۸. (کدنویسی)** فرض کنید پکیجی به نام `shop` دارید با ماژول‌های `models.py` (شامل `User`، `Product`) و `services.py` (شامل `OrderService`). یک `__init__.py` بنویسید که یک «رابط عمومی تمیز» بسازد تا کاربر بتواند `from shop import User, OrderService` بنویسد.

**۹. (کدنویسی)** یک Factory ماژولی به نام `create_notifier(kind)` بنویسید که برای `"email"` یک `EmailNotifier` و برای `"sms"` یک `SMSNotifier` بسازد و برای نوع ناشناخته `ValueError` بدهد.

**۱۰. (کدنویسی)** یک «Singleton ماژولی» بسازید: یک کلاس `_Database` (با `_` خصوصی) و یک نمونه‌ی واحد به نام `db` در سطح ماژول، طوری که هر جای برنامه که `db` را import کند، همان یک نمونه را بگیرد. چرا این واقعاً یک Singleton است؟

**۱۱. (تشخیص)** در ساختار پروژه‌ی زیر، برای هر فایل بگویید چه مسئولیتی دارد و به کدام فصل/مفهوم مربوط است:

```
shop/
├── __init__.py
├── models.py
├── exceptions.py
├── repositories.py
├── services.py
└── database.py
```

## بخش د: پرسش تحلیلی

**۱۲.** نویسنده می‌گوید «سازمان‌دهی خوب، همان اصول شیءگرایی است که به سطح فایل‌ها بالا رفته». این جمله را با اشاره به اصل Single Responsibility و انتزاع (`__init__.py`) توضیح دهید.

**۱۳.** این جمله را نقد کنید: «هر وقت به یک حالت سراسری مشترک (مثل تنظیمات یا اتصال دیتابیس) نیاز داریم، باید یک کلاس Singleton بنویسیم.»

**۱۴.** چرا «تزریق وابستگی» (فصل هشتم) اغلب انتخاب تمیزتری از Singleton سراسری است؟ Singleton چه هزینه‌ای بر تست‌پذیری و شفافیت وابستگی‌ها تحمیل می‌کند و چه زمانی استفاده از آن واقعاً موجه است؟

## بخش ه: طراحی پکیج در عمل

**۱۵. (طراحی)** برای ابزار زیر یک چیدمان پکیج پیشنهاد دهید (درخت فایل‌ها + برای هر فایل یک جمله مسئولیت): «برنامه‌ای که سفارش‌ها و کاربران را مدل می‌کند، قوانین تخفیف دارد، داده را در فایل JSON ذخیره/بازیابی می‌کند، خطاهای اختصاصی دارد، و یک رابط خط فرمان.»

**۱۶. (کدخوانی)** با توجه به ماژول زیر، بعد از `from tools import *` کدام نام‌ها در دسترس‌اند؟ اگر خط `__all__` حذف شود چه تغییری می‌کند؟

```python
# tools.py
__all__ = ["parse", "Formatter"]

def parse(text): ...
def _tokenize(text): ...
class Formatter: ...
class _Cache: ...
```

**۱۷. (سناریو)** ماژول `models.py` برای متد `total_price` به تابعی از `services.py` نیاز دارد و `services.py` هم کلاس `Order` را از `models.py` import می‌کند — و برنامه با `ImportError: cannot import name ...` می‌شکند. علت این خطای import حلقوی را توضیح دهید و **دو** راه‌حل متفاوت پیشنهاد دهید؛ بگویید کدام را ترجیح می‌دهید و چرا.

**۱۸. (تفکر انتقادی)** این سیاست تیمی را نقد کنید: «هر پوشه باید پکیج شود و `__init__.py` هر پکیج باید *همه‌ی* کلاس‌ها و توابع داخلش را re-export کند تا importها همه‌جا کوتاه شوند.» به مرز رابط عمومی/خصوصی، هزینه‌ی import زنجیره‌ای، و اثر آن بر ریفکتور اشاره کنید.
