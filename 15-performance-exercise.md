**۳.** «بارگذاریِ تنبل» (Lazy Loading) یعنی چه؟ `cached_property` چطور هم بارگذاریِ تنبل و هم کش را با هم فراهم می‌کند؟

**۴.** `functools.lru_cache` چه کاری انجام می‌دهد؟ Tradeoffِ آن چیست و فقط برای چه نوع توابعی **امن** است؟

## بخش ب: چندنخی و چندپردازشی

**۵.** GIL (Global Interpreter Lock) چیست و چه محدودیتی ایجاد می‌کند؟

**۶.** تفاوتِ کارهای «CPU-bound» و «I/O-bound» چیست؟ برای هرکدام، `threading` مناسب است یا `multiprocessing`؟ چرا؟

**۷. (کدخوانی)** چرا کلاسِ `UnsafeCounter` زیر در محیطِ چندنخی خطرناک است؟ اسمِ این مشکل چیست و چطور با `Lock` رفعش می‌کنیم؟

```python
class UnsafeCounter:
    def __init__(self):
        self.count = 0
    def increment(self):
        self.count += 1
```

## بخش د: کدنویسی و کدخوانی

**۱۲. (کدخوانی)** در کدِ زیر، جمله‌ی «محاسبه‌ی سنگین...» چند بار چاپ می‌شود و چرا؟

```python
from functools import cached_property

class Report:
    def __init__(self, data):
        self.data = data

    @cached_property
    def summary(self):
        print("Heavy computation...")
        return sum(self.data)

r = Report(list(range(1000)))
print(r.summary)
print(r.summary)
```

**۱۳. (کدنویسی)** یک کلاسِ `SafeCounter` بنویسید که thread-safe باشد: با یک `threading.Lock`، متدِ `increment` را طوری بنویسید که هر بار فقط یک نخ بتواند `count` را تغییر دهد.

**۱۴. (کدنویسی)** با استفاده از `functools.lru_cache`، یک تابعِ `fibonacci(n)` بنویسید که به‌لطفِ کش، سریع باشد. توضیح دهید چرا بدونِ کش، این تابع بسیار کند می‌شود.

## بخش ه: پرسشِ تحلیلی
