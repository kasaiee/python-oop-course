## تمرین ۱ - وراثت چندگانه و مسئله‌ی الماس

**۱.** جاوا وراثت چندگانه را ممنوع کرد چون طراحانش آن را منبع پیچیدگی و ابهام می‌دانستند (به‌خصوص مسئله‌ی الماس)؛ به‌جایش `interface` را گذاشتند که فقط امضای متد دارد نه پیاده‌سازی. پایتون راه دیگری رفت: وراثت چندگانه را اجازه داد اما یک الگوریتم دقیق (C3) گذاشت تا ابهام را حل کند. این با فلسفه‌ی «همه بزرگسالیم» جور است — ابزار قدرتمند را می‌دهد و مسئولیت درست‌استفاده‌کردن را به شما می‌سپارد. جاوا محافظه‌کارتر و «امن به‌طور پیش‌فرض» است؛ پایتون آزادتر و «قدرتمند اما نیازمند انضباط».

**۲.** مسئله‌ی الماس وقتی رخ می‌دهد که یک کلاس از دو والد ارث ببرد که **هر دو از یک جد مشترک** آمده‌اند:

```
        A
       / \
      B   C
       \ /
        D
```

ابهام: اگر `A` متدی داشته باشد که `B` و `C` هر دو Override کرده‌اند، وقتی `D` آن را صدا می‌زند نسخه‌ی کدام اجرا شود؟ و اگر همه `super()` صدا بزنند، آیا `A` دو بار اجرا می‌شود؟ بدون یک قاعده‌ی روشن، این وضعیت گیج‌کننده و خطاخیز است.

**۳.** خروجی:

```
D
B
C
A
```

`A` فقط یک‌بار اجرا می‌شود چون پایتون از **MRO** پیروی می‌کند: ترتیب `D → B → C → A → object`. عبارت `super()` در هر مرحله «نفر بعدی در MRO» را صدا می‌زند، نه والد مستقیم را. پس `super()` داخل `B` به `C` می‌رود (نه به `A`)، و `A` تنها زمانی اجرا می‌شود که هم `B` و هم `C` از آن عبور کرده باشند — یعنی یک‌بار، در انتها. این همان چیزی است که وراثت چندگانه را در پایتون امن می‌کند.

## تمرین ۲ - MRO و الگوریتم C3

**۴.** MRO مخفف **Method Resolution Order** (ترتیب جست‌وجوی متد) است؛ ترتیبی که پایتون برای پیداکردن یک متد در سلسله‌مراتب کلاس‌ها دنبال می‌کند. سه قاعده‌ی C3:
1. زیرکلاس همیشه پیش از والدهایش می‌آید.
2. ترتیب نوشتن والدها حفظ می‌شود (`class D(B, C)` یعنی `B` پیش از `C`).
3. اگر کلاسی چند مسیر داشته باشد، فقط **یک‌بار** و در جای درست می‌آید (هیچ کلاسی پیش از زیرکلاس‌هایش نمی‌آید).

**۵.** ساخت دستی:

```
[D]                         → از خود D شروع کن
[D, B]                      → B را اضافه کن (اما A را نه، چون C هم از A ارث می‌برد)
[D, B, C]                   → C را اضافه کن
[D, B, C, A]                → حالا که B و C آمدند، A را اضافه کن
[D, B, C, A, object]        → و در نهایت object
```

پس:

```python
print([c.__name__ for c in D.mro()])
# ['D', 'B', 'C', 'A', 'object']
```

قاعده‌ی کلیدی: چون `C` زیرکلاس `A` است، `A` باید بعد از `C` بیاید — همین تضمین می‌کند `A` یک‌بار و در انتها قرار گیرد.

**۶.** یعنی در سلسله‌مراتب چندگانه، `super()` لزوماً به والد نوشته‌شده در تعریف کلاس نمی‌رود، بلکه به **کلاس بعدی در فهرست MRO** می‌رود. در الماس سوال ۳، `B` مستقیماً از `A` ارث می‌برد، پس انتظار داریم `super()` داخل `B` به `A` برود؛ اما چون MRO `D` برابر `[D, B, C, A, object]` است، نفر بعدی بعد از `B` در این فهرست **`C`** است، نه `A`. این «عجیب» به‌نظر می‌رسد چون `B` اصلاً از `C` ارث نبرده — اما دقیقاً همین رفتار است که باعث می‌شود همه‌ی شاخه‌ها یک‌بار و به‌ترتیب اجرا شوند.

**۷.** چون استفاده از نام مستقیم والد (`A.__init__(self)`) **زنجیره‌ی MRO را می‌شکند**. اگر در الماس، `B` به‌جای `super().__init__()` مستقیم `A.__init__(self)` را صدا بزند، شاخه‌ی `C` به‌کل نادیده گرفته می‌شود (یا در سناریوهای دیگر، `A` دو بار اجرا می‌شود). `super()` تضمین می‌کند هر کلاس در زنجیره دقیقاً یک‌بار و به‌ترتیب درست اجرا شود؛ نام مستقیم این تضمین را از بین می‌برد.

## تمرین ۳ - Mixinها

**۸.** Mixin یک کلاس کوچک است که **یک قابلیت مشخص و مستقل** را فراهم می‌کند و قرار نیست به‌تنهایی نمونه‌سازی شود؛ آن را به کلاس اصلی «می‌چسبانید» تا آن قابلیت را اضافه کنید. تفاوت با کلاس معمولی: کلاس معمولی یک موجودیت کامل است («کاربر»، «سفارش»)، اما Mixin فقط یک **تکه‌رفتار** است («قابلیت سریال‌شدن»، «قابلیت لاگ‌گرفتن») که موجودیت مستقلی نیست.

**۹.**
- **مناسب است وقتی:** یک قابلیت مستقل و قابل استفاده‌ی مجدد دارید که باید به چند کلاس **نامرتبط** اضافه شود. مثال: `JSONMixin` که قابلیت تبدیل به JSON را به هر کلاسی می‌دهد.
- **مناسب نیست وقتی:** رابطه واقعاً Is-A است (آن‌وقت ارث‌بری معمولی درست است)، یا وقتی قابلیت را می‌توان با **Composition** (فصل بعد) تمیزتر افزود. مثال نامناسب: ساختن `Mixin` برای چیزی که در واقع یک موجودیت مستقل است.

**۱۰.**

| ویژگی | Mixin | کلاس انتزاعی (ABC) |
|-------|-------|---------------------|
| هدف | **افزودن** پیاده‌سازی آماده | **الزام** به پیاده‌سازی یک قرارداد |
| متدها | معمولاً پیاده‌سازی‌شده (کامل) | معمولاً انتزاعی (خالی) |
| پیام به زیرکلاس | «من این کار را برایت انجام می‌چهاردهم» | «تو باید این کار را انجام دهی» |

## تمرین ۴ - مفاهیم پیشرفته و کدنویسی

**۱۱.** `__init_subclass__` (از پایتون ۳.۶) یک قلاب است که **هنگام تعریف هر زیرکلاس** صدا زده می‌شود، **نه** هنگام ساخت شیء. با آن می‌توانید زیرکلاس‌ها را در لحظه‌ی تعریف بررسی یا پیکربندی کنید. مزیتش نسبت به متاکلاس: بسیار **ساده‌تر** است؛ برای کارهای رایج (ثبت خودکار، الزام قرارداد) نیازی به پیچیدگی متاکلاس ندارید.

**۱۲.** `ReprMixin`:

```python
class ReprMixin:
    def __repr__(self):
        attrs = ", ".join(f"{k}={v!r}" for k, v in self.__dict__.items())
        return f"{type(self).__name__}({attrs})"

class Point(ReprMixin):
    def __init__(self, x, y):
        self.x = x
        self.y = y

p = Point(1, 2)
print(p)        # Point(x=1, y=2)
```

`type(self).__name__` نام کلاس واقعی را می‌گیرد، پس اگر این Mixin را به هر کلاس دیگری هم بچسبانید، درست کار می‌کند — همین قابلیت استفاده‌ی مجدد، جوهر Mixin است.

**۱۳.** ثبت خودکار پلاگین:

```python
class Plugin:
    registry = []

    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        Plugin.registry.append(cls)

class EmailPlugin(Plugin):
    pass

class SMSPlugin(Plugin):
    pass

print([c.__name__ for c in Plugin.registry])
# ['EmailPlugin', 'SMSPlugin']
```

توجه کنید که ثبت **هنگام تعریف کلاس** رخ می‌دهد؛ حتی اگر هیچ شیئی از `EmailPlugin` نسازیم، باز در `registry` هست.

**راه دیگر (با دکوراتور کلاس):** همین کار را می‌توان با یک دکوراتور کلاس هم انجام داد:

```python
registry = []

def register(cls):
    registry.append(cls)
    return cls

@register
class EmailPlugin:
    pass
```

Tradeoff: دکوراتور باید روی **هر** زیرکلاس صریحاً نوشته شود (ممکن است فراموش شود)، اما `__init_subclass__` خودکار روی همه اعمال می‌شود. در عوض دکوراتور برای کسی که ارث‌بری نمی‌خواهد صریح‌تر و ساده‌تر است.

**۱۴.** مقادیر:

```
User._table   →  "users"     (از Meta خوانده شد)
Product._table →  "product"   (پیش‌فرض: نام کلاس با حروف کوچک)
```

مزیت الگوی `Meta`: **متادیتای پیکربندی** (نام جدول، ترتیب پیش‌فرض و...) از **منطق اصلی** کلاس جدا می‌ماند. این جداسازی کد را خواناتر می‌کند و پیکربندی را در یک جای مشخص و قابل‌کشف جمع می‌کند — دقیقاً همان کاری که مدل‌های جنگو با `class Meta` می‌کنند.

## تمرین ۵ - پرسش تحلیلی

**۱۵.** این جمله **افراطی** است و از یک نقص خاص، یک نتیجه‌ی کلی نادرست می‌گیرد. درست است که وراثت چندگانه‌ی بی‌محابا خطرناک است، اما:
- پایتون مسئله‌ی الماس را با **C3/MRO حل کرده**؛ مشکل ابهام دیگر وجود ندارد، تا وقتی از `super()` استفاده کنید.
- Mixinها یک استفاده‌ی **امن و بسیار مفید** از وراثت چندگانه‌اند که کد ماژولار می‌سازند و در فریمورک‌های بزرگ (جنگو، DRF) همه‌جا هستند.
- ممنوعیت کامل، شما را از ابزاری محروم می‌کند که گاهی تمیزترین راه‌حل است.

پاسخ درست «ممنوع کن» نیست، بلکه «با **انضباط** استفاده کن»: Mixinهای کوچک و مستقل خوب‌اند، سلسله‌مراتب‌های عمیق و درهم بد. این همان تفکر Tradeoff است، نه پرهیز کورکورانه.

**۱۶.** این سوال مهم است چون وراثت (چه یگانه چه چندگانه) یک ابزار **قوی اما پرهزینه** است: زیرکلاس را به‌شدت به والد گره می‌زند و سلسله‌مراتب را شکننده می‌کند. بسیاری اوقات چیزی که واقعاً می‌خواهید «افزودن یک قابلیت» است، نه «بودن یک نوع از». اگر پاسخ «فقط می‌خواهم قابلیت اضافه کنم» باشد، گزینه‌های بهتر عبارت‌اند از:
- **Mixin:** اگر قابلیت واقعاً یک تکه‌رفتار مستقل و قابل استفاده‌ی مجدد است.
- **Composition (فصل بعد):** اگر می‌توانید قابلیت را به‌شکل یک شیء جداگانه **در خود نگه دارید** به‌جای ارث‌بردن از آن. این معمولاً وابستگی کمتر و انعطاف بیشتری می‌دهد.

خلاصه: وراثت را برای رابطه‌ی واقعی Is-A نگه دارید؛ برای «فقط قابلیت می‌خواهم»، Mixin یا Composition اغلب انتخاب بهتری‌اند.
