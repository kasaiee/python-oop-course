# فصل چهارم: اصول چهارگانه شیءگرایی — کپسول‌سازی، انتزاع، ارث‌بری و چندریختی

---

## مقدمه

در فصل‌های قبل بلوک‌های پایه را ساختیم (کلاس، شیء، `self` و سازنده) و یاد گرفتیم چطور مثل یک طراح شیءگرا فکر کنیم و کلاس‌های درست را کشف کنیم. حالا وقت آن است که این بلوک‌ها را به یک ساختمان تبدیل کنیم.

چهار اصل بنیادین شیءگرایی — **کپسول‌سازی (Encapsulation)، انتزاع (Abstraction)، ارث‌بری (Inheritance) و چندریختی (Polymorphism)** — ستون‌های هر زبان شیءگرا هستند. این چهار، اسم‌های پرطمطراقی دارند اما هرکدام یک ایده‌ی ساده و کاربردی‌اند. درک درستشان، مرز میان کسی است که «کلاس بلد است» و کسی که «شیءگرا طراحی می‌کند».

این فصل بلند است، چون هر چهار اصل را با مثال واقعی پوشش می‌دهیم. با حوصله جلو بروید؛ این‌ها تا پایان دوره با شما هستند.

---

## چرا باید چهار اصل شیءگرایی را یاد بگیریم؟

فرض کنید کدی نوشته‌اید که کار می‌کند. اما آیا در برابر تغییر مقاوم است؟ آیا وقتی یک نوع پرداخت جدید اضافه می‌شود، باید ده‌جای کد را دستکاری کنید؟ آیا کسی که کد شما را می‌خواند، می‌فهمد چه خبر است؟

این چهار اصل دقیقاً برای همین‌اند:

- **کپسول‌سازی** جلوی دستکاریِ ناخواسته‌ی داده را می‌گیرد و کنترل را در دست کلاس نگه می‌دارد.
- **انتزاع** پیچیدگی را پنهان می‌کند تا کاربرِ کلاس با یک رابط ساده کار کند.
- **ارث‌بری** به شما اجازه می‌دهد کد مشترک را یک‌بار بنویسید و بازاستفاده کنید.
- **چندریختی** کدی می‌سازد که با انواع مختلف، بدون `if/else`های بی‌پایان، کار کند.

بدون این‌ها، کد شما ممکن است اجرا شود، اما با هر تغییرِ کوچک، شکننده‌تر می‌شود.

## Encapsulation — کپسول‌سازی

### کپسول‌سازی چیست و چرا مهم است؟

**کپسول‌سازی** یعنی بسته‌بندی داده و رفتارِ مربوط به آن در یک واحد، و **کنترل دسترسی** به آن داده. ایده‌ی اصلی: داده‌ی درونیِ یک شیء نباید بی‌واسطه از بیرون دستکاری شود؛ باید از طریق متدهایی که خودِ کلاس تعریف کرده، و با قوانین آن، تغییر کند.

چرا؟ چون بدون کنترل، هر کسی می‌تواند شیء را به حالتِ نامعتبر ببرد:

```python
class BankAccount:
    def __init__(self, balance):
        self.balance = balance

acc = BankAccount(1000)
acc.balance = -5000     # disaster! nobody stopped this
```

با کپسول‌سازی، داده را «محافظت‌شده» می‌کنیم و فقط از راه متدهای کنترل‌شده اجازه‌ی تغییر می‌دهیم:

```python
class BankAccount:
    def __init__(self, balance):
        self._balance = balance          # a sign of "internal, don't touch"

    def withdraw(self, amount):
        if amount > self._balance:
            raise ValueError("insufficient balance")
        self._balance -= amount

    def get_balance(self):
        return self._balance
```

### Public، Protected و Private در پایتون

پایتون برخلاف جاوا، کلیدواژه‌ی `private` یا `public` ندارد. به‌جای آن از **قرارداد نام‌گذاری** استفاده می‌کند:

| نام | قرارداد | معنا |
|-----|---------|------|
| `name` | Public | آزادانه قابل استفاده |
| `_name` | Protected | «داخلی است؛ لطفاً از بیرون دست نزن» (فقط قرارداد) |
| `__name` | Private | Name Mangling فعال می‌شود (تقریباً پنهان) |

یک زیرخط (`_`) صرفاً یک **پیام به برنامه‌نویس** است: «این جزئیاتِ داخلی است، به آن تکیه نکن.» پایتون هیچ اجباری اعمال نمی‌کند؛ فنی هنوز قابل دسترسی است.

```python
class Config:
    def __init__(self):
        self.public_value = 1
        self._internal_value = 2     # convention: internal

c = Config()
print(c._internal_value)   # 2 — works, but you "shouldn't" do this
```

### Name Mangling و فلسفه‌ی «همه بزرگسالیم»

دو زیرخط (`__`) چیز متفاوتی است. پایتون نام آن را **دستکاری** می‌کند (Name Mangling): `__x` درون کلاس `Foo` در پشت‌صحنه به `_Foo__x` تبدیل می‌شود. این باعث می‌شود دسترسی مستقیم با نام اصلی کار نکند:

```python
class Account:
    def __init__(self):
        self.__pin = "1234"      # Python turns this into _Account__pin

a = Account()
# print(a.__pin)        # AttributeError!
print(a._Account__pin)  # 1234 — still accessible, just harder
```

پس چرا هنوز می‌شود به آن رسید؟ چون فلسفه‌ی پایتون این است: *«We are all consenting adults here»* — «اینجا همه‌ی ما بزرگسالیم». پایتون به‌جای اینکه با قفل جلوی شما را بگیرد، فرض می‌کند شما مسئولید. Name Mangling برای **پنهان‌کاریِ امنیتی نیست**؛ هدف اصلی‌اش جلوگیری از **تصادم نام** در ارث‌بری است (که یک زیرکلاس سهواً attribute والد را بازنویسی نکند).

**Tradeoff:** جاوا با `private` اجباری، امنیت بیشتری می‌دهد اما انعطاف کمتری. پایتون آزادی و انعطاف می‌دهد اما مسئولیت را به شما می‌سپارد. هیچ‌کدام مطلقاً بهتر نیستند.

### چه زمانی از attribute خصوصی استفاده کنیم؟

از یک زیرخط (`_`) استفاده کنید وقتی چیزی جزئیاتِ داخلیِ پیاده‌سازی است و نمی‌خواهید بیرونی‌ها به آن تکیه کنند. از دو زیرخط (`__`) فقط وقتی واقعاً می‌خواهید از تصادم نام در ارث‌بری جلوگیری کنید — که مورد نادری است. **به‌طور پیش‌فرض، ساده نگه دارید و از یک زیرخط استفاده کنید؛** دو زیرخط را فقط با دلیل مشخص به کار ببرید.

## Abstraction — انتزاع

### انتزاع چیست و چه فرقی با کپسول‌سازی دارد؟

**انتزاع** یعنی نمایش‌دادن فقط جزئیاتِ ضروری به کاربر و پنهان‌کردن پیچیدگیِ پشت آن. وقتی رانندگی می‌کنید، با فرمان و پدال کار می‌کنید، نه با جزئیاتِ احتراق موتور. فرمان، یک **رابط انتزاعی** است روی سیستمِ پیچیده.

تفاوت با کپسول‌سازی ظریف اما مهم است:

- **کپسول‌سازی** درباره‌ی **کنترل دسترسی** است: «چه کسی اجازه دارد این داده را ببیند یا تغییر دهد؟»
- **انتزاع** درباره‌ی **ساده‌سازیِ رابط** است: «کاربر چه چیزی را باید بداند و چه چیزی را نه؟»

این دو معمولاً با هم کار می‌کنند اما یکی نیستند.

### طراحی یک رابط ساده و سطح‌بالا

مثال واقعی: یک کلاس ارسال پیامک. کاربرِ کلاس نباید بداند پشت‌صحنه چه اتصال HTTP، چه احراز هویت و چه فرمتی رد و بدل می‌شود. او فقط می‌خواهد یک پیام بفرستد.

```python
class SMSSender:
    def __init__(self, api_key):
        self._api_key = api_key                 # encapsulated

    def send(self, to, message):                # simple interface (abstraction)
        self._authenticate()
        payload = self._build_payload(to, message)
        return self._post(payload)

    # complex details, hidden from the user:
    def _authenticate(self): ...
    def _build_payload(self, to, message): ...
    def _post(self, payload): ...

sender = SMSSender("secret-key")
sender.send("0912...", "Hello")   # the user only sees this
```

کاربر با یک متدِ `send` کار می‌کند. سه متدِ `_`دار، پیچیدگیِ پنهان‌اند. اگر فردا کل منطق ارسال عوض شود، تا وقتی امضای `send` ثابت بماند، کدِ کاربر دست‌نخورده می‌ماند. این قدرتِ انتزاع است.

(ابزارهای رسمی‌ترِ انتزاع — یعنی کلاس‌های انتزاعی و پروتکل‌ها — را در فصل نهم عمیق بررسی می‌کنیم.)

## Inheritance — ارث‌بری

### ارث‌بری چیست و چه مشکلی را حل می‌کند؟

**ارث‌بری** به یک کلاس اجازه می‌دهد داده و رفتارِ کلاس دیگری را به ارث ببرد و بر آن بیفزاید یا تغییرش دهد. رابطه‌ای که می‌سازد از نوع **«یک نوعِ...»** (Is-A) است: «مدیر یک نوعِ کارمند است»، پس `Manager` از `Employee` ارث می‌برد.

مشکلی که حل می‌کند: **جلوگیری از تکرار کد**. اگر چند کلاس رفتار مشترک دارند، آن را یک‌بار در کلاس والد می‌نویسید.

```python
class Employee:
    def __init__(self, name, salary):
        self.name = name
        self.salary = salary

    def pay(self):
        return f"{self.name}: {self.salary} Toman"

class Manager(Employee):              # Manager inherits from Employee
    def __init__(self, name, salary, team_size):
        super().__init__(name, salary)    # call the parent's constructor
        self.team_size = team_size

    def bonus(self):
        return self.salary * 0.1 * self.team_size

m = Manager("ali", 5_000_000, 4)
print(m.pay())      # ali: 5000000 Toman — method inherited from Employee
print(m.bonus())    # 2000000.0 — method specific to Manager
```

### super() چطور کار می‌کند؟

`super()` به شما اجازه می‌دهد به پیاده‌سازیِ کلاس والد دسترسی داشته باشید. رایج‌ترین کاربرد در `__init__` است: زیرکلاس اول سازنده‌ی والد را صدا می‌زند تا داده‌ی مشترک مقداردهی شود، بعد داده‌ی خودش را اضافه می‌کند — همان‌طور که در `Manager` بالا دیدید.

`super()` در ارث‌بریِ چندلایه هم به‌درستی زنجیره را دنبال می‌کند:

```python
class A:
    def setup(self):
        print("A")

class B(A):
    def setup(self):
        super().setup()      # goes to A
        print("B")

class C(B):
    def setup(self):
        super().setup()      # goes to B (which itself goes to A)
        print("C")

C().setup()
# Output:
# A
# B
# C
```

مکانیزم دقیقِ `super()` در ارث‌بریِ چندگانه به MRO (که در فصل اول دیدیم) وابسته است و در **فصل ششم** کامل بازش می‌کنیم.

### Override: بازنویسی رفتار والد

**Override** یعنی زیرکلاس، متدی از والد را با پیاده‌سازیِ خودش جایگزین کند. دو نوع دارد:

**۱. Override کامل** — والد را کاملاً کنار می‌گذاریم:

```python
class Employee:
    def role(self):
        return "regular employee"

class Contractor(Employee):
    def role(self):                  # full Override
        return "contractor"
```

**۲. Override توسعه‌ای (با super)** — رفتار والد را نگه می‌داریم و چیزی به آن می‌افزاییم:

```python
class Employee:
    def describe(self):
        return f"Name: {self.name}"

class Manager(Employee):
    def describe(self):              # extending Override
        base = super().describe()    # take the parent's behavior
        return f"{base} | role: manager"   # and extend it
```

انتخاب بین این دو، بسته به این است که آیا رفتار والد هنوز ارزش دارد یا نه.

### __str__ و __repr__

دو متد جادویی که اغلب Override می‌شوند و برای خوانایی و دیباگ حیاتی‌اند. این‌ها را چون مستقیماً به بحثِ Override مربوط‌اند همین‌جا و به‌اختصار معرفی می‌کنیم؛ خانواده‌ی کاملِ متدهای جادویی را در فصل پنجم می‌بینیم:

- `__str__`: نمایشِ **خوانا برای کاربر**. وقتی `print(obj)` یا `str(obj)` صدا زده می‌شود.
- `__repr__`: نمایشِ **دقیق برای برنامه‌نویس** (ایدئال: بتوان با آن شیء را بازساخت). وقتی در کنسول شیء را تایپ می‌کنید یا `repr(obj)` صدا زده می‌شود.

```python
class Money:
    def __init__(self, amount, currency):
        self.amount = amount
        self.currency = currency

    def __str__(self):
        return f"{self.amount:,} {self.currency}"        # for the user

    def __repr__(self):
        return f"Money({self.amount!r}, {self.currency!r})"  # for the programmer

m = Money(1000, "Toman")
print(m)          # 1,000 Toman   ← __str__
print(repr(m))    # Money(1000, 'Toman')   ← __repr__
print([m])        # [Money(1000, 'Toman')] ← inside a container, __repr__ is used
```

قاعده: همیشه دست‌کم `__repr__` را تعریف کنید (کمکِ بزرگی به دیباگ است)؛ `__str__` را وقتی نمایشِ کاربرپسندِ متفاوتی لازم دارید.

### چرا استثناهای سفارشی تعریف می‌کنیم؟

ارث‌بری فقط برای مدل‌های داده نیست؛ یکی از کاربردی‌ترین جاهایش **استثناهای سفارشی** است. با ارث‌بری از `Exception`، خطاهای معنادارِ مخصوص پروژه می‌سازید:

```python
class PaymentError(Exception):
    """base error related to payment"""

class InsufficientFundsError(PaymentError):
    """insufficient balance"""

class CardExpiredError(PaymentError):
    """card has expired"""

def charge(account, amount):
    if account.balance < amount:
        raise InsufficientFundsError("insufficient balance")

try:
    charge(account, 1000)
except PaymentError as e:      # catch all payment errors at once
    print(f"Payment failed: {e}")
```

مزیت: با یک `except PaymentError`، هم `InsufficientFundsError` و هم `CardExpiredError` را می‌گیرید، چون هر دو زیرکلاسِ آن‌اند. این قدرتِ سلسله‌مراتبِ استثناست که در فصل یازدهم کاملش می‌کنیم.

### چه زمانی از ارث‌بری استفاده نکنیم؟

ارث‌بری وسوسه‌انگیز است اما دام دارد. فقط وقتی از آن استفاده کنید که رابطه واقعاً **Is-A** باشد. اگر رابطه از نوع **Has-A** («دارد») است، ارث‌بری اشتباه است و باید از **ترکیب (Composition)** استفاده کنید — که موضوع فصل هفتم است. مثلاً «ماشین یک موتور دارد» (Has-A)، پس `Car` نباید از `Engine` ارث ببرد؛ باید یک `Engine` را در خود نگه دارد.

## Polymorphism — چندریختی

### چندریختی چیست و چه فرقی با Override دارد؟

**چندریختی** (به‌معنای «چند شکلی») یعنی توانایی کار با اشیای مختلف از طریق یک رابط مشترک، بی‌آنکه نوع دقیقشان را بدانید. شما یک پیام را می‌فرستید و هر شیء به شیوه‌ی خودش پاسخ می‌دهد.

تفاوت با Override: **Override مکانیزم است** (زیرکلاس متد والد را بازنویسی می‌کند)؛ **چندریختی نتیجه است** (می‌توانید با انواع مختلف یکسان رفتار کنید). Override یکی از راه‌های رسیدن به چندریختی است.

```python
class Circle:
    def __init__(self, r): self.r = r
    def area(self):
        return 3.14159 * self.r ** 2

class Rectangle:
    def __init__(self, w, h): self.w, self.h = w, h
    def area(self):
        return self.w * self.h

# polymorphism: one function, unaware of the exact type, works with all
def total_area(shapes):
    return sum(shape.area() for shape in shapes)

shapes = [Circle(2), Rectangle(3, 4)]
print(total_area(shapes))   # 24.56636
```

نکته‌ی کلیدی: `total_area` نمی‌داند و **نیازی ندارد بداند** که با دایره کار می‌کند یا مستطیل. فقط می‌داند هر عضو یک `area()` دارد. این **Duck Typing** پایتون است: «اگر مثل اردک راه می‌رود و مثل اردک صدا می‌کند، اردک است.» مهم نوعِ شیء نیست، مهم این است که رفتارِ لازم را دارد.

### چندریختی چطور به انعطاف کمک می‌کند؟

فرض کنید بدون چندریختی می‌خواستیم مساحت را حساب کنیم:

```python
# without polymorphism — fragile and full of ifs
def total_area_bad(shapes):
    total = 0
    for shape in shapes:
        if isinstance(shape, Circle):
            total += 3.14159 * shape.r ** 2
        elif isinstance(shape, Rectangle):
            total += shape.w * shape.h
        # every new shape means a new elif here!
    return total
```

هر بار که شکل جدیدی اضافه می‌شد، باید این تابع را دستکاری می‌کردیم. با چندریختی، شکل جدید فقط یک کلاس تازه با متد `area()` است و `total_area` دست‌نخورده می‌ماند. این همان اصل Open/Closed است که در فصل هشتم می‌بینیم.

### isinstance در برابر type()

برای بررسی نوعِ یک شیء دو راه هست، اما یکی درست‌تر است:

```python
class Animal: pass
class Dog(Animal): pass

d = Dog()
print(type(d) == Dog)          # True
print(type(d) == Animal)       # False — type checks the exact type
print(isinstance(d, Animal))   # True — isinstance also sees inheritance
```

`isinstance` برتری دارد چون **ارث‌بری را در نظر می‌گیرد**: یک `Dog` واقعاً یک `Animal` هم هست، و `isinstance` این را می‌فهمد اما `type() ==` نه. در کد شیءگرا تقریباً همیشه `isinstance` انتخاب درست است. (هرچند بهترین کد شیءگرا اغلب اصلاً نیازی به بررسی نوع ندارد و به Duck Typing تکیه می‌کند.)

### متدهای جادویی و __call__

**متدهای جادویی** (Magic/Dunder Methods) به کلاس شما اجازه می‌دهند با عملگرها و ساختارهای داخلی پایتون تعامل کند. `__str__` و `__repr__` را دیدیم. `__len__` طول را تعریف می‌کند، `__eq__` برابری را، و بسیاری دیگر — که در فصل پنجم مفصل می‌آیند.

یکی از جالب‌ترین‌ها `__call__` است: شیء را **قابل‌فراخوانی مثل تابع** می‌کند.

```python
class Multiplier:
    def __init__(self, factor):
        self.factor = factor

    def __call__(self, x):           # makes the object callable
        return x * self.factor

double = Multiplier(2)
print(double(10))       # 20 — as if double were a function!
print(callable(double)) # True
```

این الگو وقتی مفید است که می‌خواهید چیزی مثل تابع رفتار کند اما حالت هم نگه دارد (اینجا `factor`). کاربرد واقعی‌اش در Decoratorها و ماشین‌های حالت است که در فصل پنجم می‌بینیم.

## خلاصه

| اصل | ایده‌ی اصلی | کلمه‌ی کلیدی |
|-----|-------------|--------------|
| کپسول‌سازی | کنترل دسترسی به داده | `_`, `__`, Name Mangling |
| انتزاع | پنهان‌کردن پیچیدگی، رابط ساده | متدهای Public در برابر `_`دار |
| ارث‌بری | بازاستفاده از کد (رابطه Is-A) | `super()`, Override |
| چندریختی | یک رابط، رفتارهای مختلف | Duck Typing, `isinstance` |

```
چهار ستون شیءگرایی
════════════════════════════════════
  کپسول‌سازی  ──►  «داده‌ام را خودم کنترل می‌کنم»
  انتزاع      ──►  «فقط چیزی که لازم داری را می‌بینی»
  ارث‌بری     ──►  «رفتار مشترک را یک‌بار می‌نویسم»
  چندریختی    ──►  «با همه یکسان رفتار می‌کنم»
```

**نکته‌ی طلایی:** این چهار اصل ابزارند، نه فرمان. کپسول‌سازیِ افراطی کد را دست‌وپاگیر می‌کند؛ ارث‌بریِ بی‌جا آن را شکننده می‌کند. همیشه بپرسید این اصل اینجا چه مشکلی را حل می‌کند — اگر پاسخی نداشتید، شاید لازمش ندارید. تفکر به Tradeoff، همیشه بالاتر از پیروی کورکورانه از اصول است.

## پرسش‌های مصاحبه

چهار ستون، چهار سوالِ همیشگی:

**۱. «فرق انتزاع و کپسوله‌سازی؟»** — پرتکرارترین سوالِ تئوریِ مصاحبه‌های فارسی. کپسوله‌سازی درباره‌ی **محافظت** است (جمع‌کردنِ داده و رفتار و کنترلِ دسترسی)؛ انتزاع درباره‌ی **سادگیِ رابط** است (پنهان‌کردنِ جزئیات پشتِ رابطی سطح‌بالا). یکی «چه چیزی را نبینند»، دیگری «چه چیزی را ببینند».

**۲. «پایتون که private واقعی ندارد؛ پس کپسوله‌سازی‌اش شوخی است؟»** — نه؛ فلسفه‌اش فرق دارد: قرارداد به‌جای قفل («همه اینجا بزرگسالیم»). `_` یعنی «دست نزن»، `__` با Name Mangling دست‌درازیِ اتفاقی (و تصادمِ نام در ارث‌بری) را سخت می‌کند. نکته‌ی طلایی: هدفِ کپسوله‌سازی جلوگیری از «دشمن» نیست؛ جلوگیری از «اشتباهِ صادقانه» است.

**۳. «چندریختی را با یک مثالِ کدِ واقعی توضیح بده.»** — تابعی که با `Drawable`های مختلف کار می‌کند بی‌آنکه نوعشان را بپرسد؛ حذفِ زنجیره‌ی if/elif با واگذاری به متدِ مشترک. اگر گفتید «چندریختی همان چیزی است که Open/Closed را ممکن می‌کند»، پیوندِ فصل‌ها را هم نشان داده‌اید.

**۴. «isinstance بهتر است یا type()؟ و کِی هیچ‌کدام؟»** — `isinstance` زیرکلاس‌ها را می‌پذیرد و با چندریختی سازگار است؛ `type() ==` سلسله‌مراتب را نادیده می‌گیرد. و «هیچ‌کدام» وقتی که اصلاً نباید نوع بپرسید — Duck Typing: به‌جای پرسیدن، متد را صدا بزن و به قرارداد اعتماد کن.

---

**در فصل بعد:** سراغ ابزارهای پیشرفته‌ترِ پایتون می‌رویم — `@property`، دکوراتورهای کلاسی و متدهای جادویی — که به شما اجازه می‌دهند کدِ تمیزتر و حرفه‌ای‌تری بنویسید و از قابلیت‌های منحصربه‌فرد پایتون بهره ببرید.
