# درس تکمیلی: شیءگراییِ پایتون در برابر زبان‌های دیگر

> این یک درسِ **تکمیلی و اختیاری** است. اگر با زبان‌های دیگری مثل جاوا، C++ یا جاوااسکریپت آشنایی ندارید، اصلاً نگرانِ این درس نباشید — می‌توانید کاملاً بی‌خیالش شوید و روی خودِ پایتون تمرکز کنید. این درس بیشتر برای کسانی مفید است که قبلاً زبانِ دیگری دیده‌اند یا می‌خواهند تفاوت‌ها را بدانند.

---

## مقدمه

در فصل اول گفتیم «شیءگرایی در هر زبان لهجه‌ی خودش را دارد». این درس همان لهجه‌ها را کنار هم می‌گذارد. هدف این است که وقتی روزی کد یا مستنداتِ زبانِ دیگری را دیدید، گیج نشوید و ندانسته انتظارِ نادرست از پایتون نداشته باشید.

مهم: قرار نیست جاوا یا C++ یاد بگیرید. فقط می‌خواهیم ببینیم پایتون در چند تصمیمِ کلیدی، چه راهی رفته و چرا.

---

## چرا این مقایسه مفید است؟

- **مهاجرت از زبانِ دیگر:** اگر از جاوا یا ++C به پایتون آمده‌اید، بعضی عادت‌هایتان در پایتون اشتباه است (مثلاً نوشتنِ getter/setter برای همه‌چیز). دانستنِ تفاوت‌ها این عادت‌ها را اصلاح می‌کند.
- **خواندنِ منابع:** خیلی از کتاب‌ها و مقالاتِ شیءگرایی با جاوا نوشته شده‌اند. اگر تفاوت‌ها را بدانید، آن‌ها را درست به پایتون ترجمه می‌کنید.
- **درکِ عمیق‌تر:** فهمیدنِ اینکه پایتون *چه چیزی را انتخاب نکرده*، به شما می‌گوید چرا *این* را انتخاب کرده.

---

## چطور؟ تفاوت‌های کلیدی

بیایید چند محورِ مهم را یک‌به‌یک ببینیم. برای هر محور، یک جمله‌ی ساده و در صورت لزوم یک مثالِ کوتاه.

### محور ۱: ارث‌بری چندگانه

- **پایتون:** اجازه می‌دهد یک کلاس از **چند** والد ارث ببرد (فصل ۶).
- **جاوا:** ارث‌بریِ چندگانه‌ی کلاس را **ممنوع** کرده؛ فقط از طریقِ `interface` معادلِ محدودی دارد.
- **C++:** دارد، اما پیچیده‌تر و دستی‌تر است (نیازمندِ ارث‌بریِ مجازی).
- **جاوااسکریپت:** ارث‌بریِ کلاسیک ندارد؛ مدلش مبتنی بر **Prototype** است.

```python
# Python — multiple inheritance is allowed
class Serializable:
    def to_json(self): ...

class Comparable:
    def __lt__(self, other): ...

class Product(Serializable, Comparable):   # two parents at once
    pass
```

### محور ۲: کپسول‌سازی (Public/Private)

این مهم‌ترین تفاوت برای مهاجران از جاواست.

- **پایتون:** کپسول‌سازی **قراردادی** است. `private`ِ اجباری ندارد؛ فقط از قرارداد `_` و `__` استفاده می‌کند (فصل ۴).
- **جاوا و C++:** کپسول‌سازی **اجباری** است؛ `private` واقعاً جلوی دسترسی از بیرون را می‌گیرد.
- **جاوااسکریپت:** از ES2022 با پیشوندِ `#` فیلدِ خصوصیِ واقعی دارد.

```python
# Python — "private" is only a convention, not enforced
class Account:
    def __init__(self):
        self._balance = 100     # underscore says "please don't touch"

a = Account()
print(a._balance)               # 100 — it works, but you "shouldn't"
```

در جاوا، دسترسی به یک فیلدِ `private` از بیرون اصلاً کامپایل نمی‌شود. در پایتون، کار می‌کند — چون فلسفه‌ی پایتون *«We are all consenting adults here»* است (فصل ۴).

### محور ۳: «همه‌چیز یک شیء است»

- **پایتون:** بله، همه‌چیز — عدد، رشته، تابع، و حتی خودِ کلاس‌ها — یک شیء است.
- **جاوا و C++:** نه کاملاً؛ نوع‌های ساده (مثلِ `int` در جاوا) شیء نیستند و جدا مدیریت می‌شوند.
- **جاوااسکریپت:** تقریباً؛ ولی با استثناهایی.

```python
# Python — even integers and functions are objects
print(isinstance(42, object))          # True
print(isinstance(len, object))         # True
print((42).bit_length())               # 6 — an int has methods!
```

### محور ۴: تعریف Interface (قرارداد)

- **پایتون:** با **ABC** یا **Protocol** (فصل ۹). به‌ویژه Protocol با فلسفه‌ی Duck Typing سازگار است.
- **جاوا:** کلیدواژه‌ی صریحِ `interface`.
- **C++:** «کلاسِ مجازیِ خالص» (Pure Virtual Class).
- **جاوااسکریپت:** به‌شکلِ رسمی ندارد.

### محور ۵: نوع‌بندی (Typing)

- **پایتون:** **پویا** (Dynamic) — نوع در زمانِ اجرا مشخص می‌شود. به‌علاوه Type Hints اختیاری برای کمک به ابزارها (فصل ۹).
- **جاوا و C++:** **ایستا** (Static) — نوع در زمانِ کامپایل بررسی می‌شود.
- **جاوااسکریپت:** پویا.

```python
# Python — dynamic typing; the same variable can hold different types
x = 42          # now an int
x = "hello"     # now a str — perfectly legal
```

در جاوا، اگر یک متغیر را `int` تعریف کنید، هرگز نمی‌توانید در آن رشته بگذارید؛ کامپایلر جلویش را می‌گیرد.

---

## جدولِ جمع‌بندیِ مقایسه

| ویژگی | Python | Java | C++ | JavaScript |
|-------|--------|------|-----|------------|
| ارث‌بری چندگانه | ✅ دارد | ❌ فقط Interface | ✅ دارد (پیچیده) | ❌ مبتنی بر Prototype |
| کپسول‌سازی | قراردادی (`_`, `__`) | اجباری (`private`) | اجباری (`private`) | قراردادی (`#` از ES2022) |
| «همه‌چیز شیء است»؟ | ✅ بله | ❌ نوع‌های ساده جدا | ❌ نوع‌های ساده جدا | تقریباً |
| تعریف Interface | Protocol و ABC | کلیدواژه‌ی `interface` | کلاس مجازیِ خالص | ندارد |
| نوع‌بندی | پویا + Type Hints | ایستا | ایستا | پویا |
| مدل کلی | نزدیک به Ruby | اسمی (Nominal) | اسمی (Nominal) | مبتنی بر Prototype |

## شبیه‌ترین زبان به پایتون: روبی

اگر بخواهیم یک زبان را «خویشاوندِ نزدیکِ» پایتون بنامیم، آن روبی است. هر دو:

- **پویا** هستند (نوع در زمانِ اجرا).
- **ارث‌بریِ چندگانه** (یا معادلش با ماژول‌ها و Mixin) دارند.
- **قراردادمحور** هستند نه اجبارمحور؛ به بلوغِ برنامه‌نویس تکیه می‌کنند به‌جای قفلِ اجباری.

اگر روزی روبی ببینید، احساسِ آشنایی خواهید کرد — درست همان‌طور که یک فارسی‌زبان با شنیدنِ زبانی هم‌ریشه، بعضی واژه‌ها را می‌فهمد.

---

## Tradeoff: کدام رویکرد بهتر است؟

هیچ‌کدام مطلقاً بهتر نیستند — این همان درسِ همیشگیِ Tradeoff است:

- **رویکردِ جاوا (اجبار و ایستایی):** امنیتِ بیشتر، خطاهای زودهنگام در زمانِ کامپایل، اما انعطافِ کمتر و کدِ پرگوتر.
- **رویکردِ پایتون (قرارداد و پویایی):** انعطافِ بالا، کدِ مختصرتر و سریع‌تر برای نوشتن، اما مسئولیتِ بیشتر بر دوشِ برنامه‌نویس و خطاهایی که ممکن است دیرتر (در زمانِ اجرا) ظاهر شوند.

انتخابِ پایتون، آزادی و سرعتِ توسعه را بر سخت‌گیریِ اجباری ترجیح داده. این با کلِ فلسفه‌ی زبان هماهنگ است.

## خلاصه

```
لهجه‌ی پایتون در شیءگرایی
════════════════════════════════════════════════
  ارث‌بری چندگانه  ✅  (برخلافِ جاوا)
  کپسول‌سازی       قراردادی  (برخلافِ جاوا/C++)
  همه‌چیز شیء است  ✅  (برخلافِ جاوا/C++)
  نوع‌بندی          پویا  (برخلافِ جاوا/C++)
  خویشاوندِ نزدیک   Ruby
```

| اگر از این زبان می‌آیید | این عادت را در پایتون کنار بگذارید |
|-------------------------|-------------------------------------|
| Java / C++ | نوشتنِ getter/setter برای همه‌چیز (به‌جایش `@property`، فصل ۵) |
| Java / C++ | تکیه بر `private`ِ اجباری (پایتون قرارداد دارد) |
| Java | ترسِ از ارث‌بریِ چندگانه (پایتون مدیریتش می‌کند، فصل ۶) |
| JavaScript | فکرکردن با Prototype (پایتون کلاسِ واقعی دارد) |

**نکته‌ی طلایی:** پایتون مسیرِ «آزادی و انعطاف با تکیه بر مسئولیتِ برنامه‌نویس» را انتخاب کرده. اگر از زبانی سخت‌گیرتر می‌آیید، این آزادی اول کمی نگران‌کننده است — اما خیلی زود متوجه می‌شوید که چقدر توسعه را سریع‌تر و لذت‌بخش‌تر می‌کند.
