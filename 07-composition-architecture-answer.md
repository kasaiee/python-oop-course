## بخش الف: Composition

**۱.** Composition یعنی یک کلاس نمونه‌هایی از کلاس‌های دیگر را **در خود نگه دارد** و کارش را با کمکِ آن‌ها انجام دهد، به‌جای ارث‌بری. رابطه‌ای که می‌سازد **Has-A («دارد»)** است. `class Car(Engine)` اشتباه است چون رابطه‌ی واقعی **Is-A نیست**: ماشین یک «نوع موتور» نیست، بلکه یک موتور **دارد**. شکلِ درست:

```python
class Engine:
    def start(self):
        return "engine started"

class Car:
    def __init__(self):
        self.engine = Engine()          # Car has an Engine (Has-A)

    def start(self):
        return self.engine.start()      # delegates the work to the engine
```

`Car` موتور را در خود دارد و کار را به آن **واگذار (delegate)** می‌کند — طبیعی‌تر و منعطف‌تر.

**۲.** انفجارِ کلاس یعنی وقتی هر بُعدِ جدیدِ تنوع، تعدادِ کلاس‌ها را **ضربدری** می‌کند. با ارث‌بری، برای دو بُعدِ «نوعِ استخدام» (تمام‌وقت/پاره‌وقت) و «روشِ پرداخت» (بانکی/نقدی)، مجبورید همه‌ی ترکیب‌ها را بسازید: `FullTimeBankEmployee`، `FullTimeCashEmployee`، `PartTimeBankEmployee`، `PartTimeCashEmployee` و... با بُعدِ سوم، این عدد باز هم ضرب می‌شود. Composition این را حل می‌کند چون هر بُعد یک **قطعه‌ی مستقل** می‌شود که به کلاسِ اصلی تزریق می‌شود:

```python
class Employee:
    def __init__(self, name, payment_method):
        self.name = name
        self.payment_method = payment_method   # behavior is injected

    def receive_salary(self, amount):
        return self.payment_method.pay(amount)
```

حالا افزودنِ روشِ پرداختِ جدید فقط یک کلاسِ تازه است، بدونِ دست‌زدن به `Employee` و بدونِ انفجار. (این همان الگوی **Strategy** است.)

**۳.**

| ویژگی | ارث‌بری (Is-A) | Composition (Has-A) |
|-------|----------------|----------------------|
| زمانِ اتصال | ثابت، در زمانِ تعریف | منعطف، حتی در زمانِ اجرا |
| نوعِ وابستگی | محکم (به کلِ والد) | سست (فقط به رابطِ جزء) |
| انفجارِ کلاس | با هر بُعدِ جدید رخ می‌دهد | ندارد |

**۴.** نه، همیشه بهتر نیست. **Tradeoff:** Composition گاهی کدِ بیشتری می‌طلبد (باید متدها را دستی واگذار کنید) و یک لایه‌ی غیرمستقیم اضافه می‌کند؛ ارث‌بری مختصرتر است اما وابستگیِ محکم‌تری می‌سازد. ارث‌بری هنوز درست است وقتی رابطه‌ی واقعیِ **Is-A** باشد و بخواهید رفتار و رابطِ والد را کاملاً به ارث ببرید — مثلِ استثناهای سفارشی (فصل چهارم) یا کلاس‌های پایه‌ی فریمورک. قاعده «ترکیب را ترجیح بده» است، نه «همیشه ترکیب».

## بخش ب: اصول طراحی

**۵.** قانون دیمتر می‌گوید هر واحد باید فقط با «دوستانِ نزدیکش» حرف بزند، نه با غریبه‌ها؛ عملاً یعنی از زنجیره‌های طولانیِ نقطه پرهیز کنید. زنجیره‌ی `self.customer.wallet.account.balance` باعث می‌شود `Order` به ساختارِ داخلیِ `customer`، `wallet` و `account` وابسته شود؛ اگر هرکدام تغییر کند، `Order` می‌شکند (وابستگیِ پنهان و شکننده). اصلاح: از دوستِ مستقیم یک نتیجه‌ی ساده بخواهید و جزئیاتش را نپرسید:

```python
class Customer:
    def available_balance(self):
        return self.wallet.balance()     # Customer manages its own details

class Order:
    def total_with_discount(self):
        return self.customer.available_balance() * 0.9
```

**۶.** Tell, Don't Ask می‌گوید به‌جای پرسیدنِ داده‌ی یک شیء و تصمیم‌گیریِ بیرونی، به شیء **بگو** چه کند و بگذار خودش تصمیم بگیرد.
- **رویکرد ۱ (Ask):** بیرون داده را می‌پرسد و تصمیم می‌گیرد؛ منطقِ «قانونِ موجودی» در سراسرِ کد پخش می‌شود و هرجا که برداشت لازم است تکرار می‌شود.
- **رویکرد ۲ (Tell):** خودِ `account` قانون را می‌داند و اجرا می‌کند؛ منطق **متمرکز** می‌ماند.

رویکرد ۲ بهتر است چون منطقِ کسب‌وکار درونِ شیء کپسوله می‌شود، تکرار حذف می‌شود، و تغییرِ قانون فقط در یک جا انجام می‌شود.

**۷.**
- **Anemic:** فقط داده دارد، بدونِ رفتار؛ منطق در جای دیگری (سرویس‌ها) پخش می‌شود.
- **Rich:** داده و رفتارِ مربوط را با هم دارد؛ منطق نزدیکِ داده‌اش می‌ماند.

**Tradeoff:** کلاس‌های Rich معمولاً شیءگراییِ بهتری‌اند و منطق را جمع نگه می‌دارند. اما در بعضی معماری‌ها (لایه‌های داده‌ی خالص، DTOها، یا مدل‌های `dataclass` فصل یازدهم) کلاسِ Anemic **عمداً** انتخاب می‌شود — وقتی هدف فقط انتقالِ داده است و رفتاری در کار نیست. باز هم بستگی به موقعیت دارد.

**۸.** معماری لایه‌ای کد را به لایه‌های افقی با مسئولیت‌های مجزا تقسیم می‌کند. سه لایه‌ی رایج:
1. **Presentation** (رابط کاربر/API): ورودی/خروجی و نمایش.
2. **Business Logic** (منطق کسب‌وکار): قوانینِ اصلیِ برنامه.
3. **Data Access** (دسترسی به داده): ذخیره و بازیابی.

جریانِ وابستگی از **بالا به پایین** است؛ هر لایه فقط با لایه‌ی زیرینش حرف می‌زند. تفاوت با هگزاگونال: در معماری لایه‌ای جریانِ وابستگی یک‌طرفه و سلسله‌مراتبی است؛ در هگزاگونال (Ports & Adapters) هسته‌ی منطق در مرکز است و همه‌ی وابستگی‌های بیرونی از طریقِ «پورت»های انتزاعی وارد می‌شوند — انعطاف و تست‌پذیریِ بیشتر، اما پیچیدگیِ بیشتر.

## بخش ج: کدنویسی و کدخوانی

**۹.** بازنویسی با رعایتِ دیمتر:

```python
class Customer:
    def available_balance(self):
        return self.wallet.account.balance     # the details stay inside Customer

class Order:
    def total_with_discount(self):
        return self.customer.available_balance() * 0.9
```

`Order` فقط با `Customer` حرف می‌زند و یک عددِ ساده می‌گیرد. اگر ساختارِ داخلیِ `Customer` عوض شود (مثلاً `wallet` حذف شود)، `Order` دست‌نخورده می‌ماند.

**۱۰.** سیستمِ اعلان:

```python
class Channel:
    def send(self, message): ...

class EmailChannel(Channel):
    def send(self, message):
        print(f"Email: {message}")

class SMSChannel(Channel):
    def send(self, message):
        print(f"SMS: {message}")

class Notifier:
    def __init__(self, channels):
        self.channels = channels

    def broadcast(self, message):
        for channel in self.channels:
            channel.send(message)

notifier = Notifier([EmailChannel(), SMSChannel()])
notifier.broadcast("Hello")
```

**افزودنِ کانالِ جدید:** فقط یک کلاسِ تازه با متدِ `send` می‌نویسید (مثلاً `PushChannel`) و به لیست اضافه می‌کنید؛ `Notifier` **اصلاً تغییر نمی‌کند**. افزودنِ کانال یعنی «افزودنِ کدِ تازه»، نه «ویرایشِ کدِ موجود».

**۱۱.** بازنویسی با چندریختی:

```python
class Regular:
    def apply(self, amount):
        return amount

class VIP:
    def apply(self, amount):
        return amount * 0.8

class Employee:
    def apply(self, amount):
        return amount * 0.5

def discount(customer, amount):
    return customer.apply(amount)

print(discount(VIP(), 1000))       # 800.0
print(discount(Employee(), 1000))  # 500.0
```

حالا نوعِ مشتریِ جدید فقط یک کلاسِ تازه با متدِ `apply` است؛ تابعِ `discount` دست‌نخورده می‌ماند (اصلِ Open/Closed).

**راهِ دیگر (تابع به‌جای کلاس):** چون هر «استراتژیِ تخفیف» بی‌حالت است، می‌توان به‌جای کلاس از توابع یا یک دیکشنری از توابع استفاده کرد:

```python
strategies = {
    "regular": lambda a: a,
    "vip": lambda a: a * 0.8,
    "employee": lambda a: a * 0.5,
}
def discount(customer_type, amount):
    return strategies[customer_type](amount)
```

Tradeoff: نسخه‌ی تابعی مختصرتر است؛ نسخه‌ی کلاسی وقتی هر استراتژی به حالت یا داده‌ی بیشتری نیاز دارد بهتر مقیاس می‌گیرد.

**۱۲.** تست‌پذیری با جایگزینِ جعلی:

```python
class FakeChannel:
    def __init__(self):
        self.sent = []
    def send(self, message):
        self.sent.append(message)

fake = FakeChannel()
notifier = Notifier([fake])
notifier.broadcast("test")
assert fake.sent == ["test"]     # no network, fast and deterministic
```

ارتباط با **Dependency Injection:** چون `Notifier` کانال‌ها را از **بیرون** دریافت می‌کند (تزریق) و به **رابط** (متدِ `send`) وابسته است نه به کلاسِ مشخص، در تست می‌توانیم یک نسخه‌ی جعلی تزریق کنیم. همین تزریقِ وابستگی، تست را از منابعِ واقعی (شبکه، دیتابیس) جدا و سریع می‌کند.

## بخش د: مقایسه و تحلیل

**۱۳.** در کدِ رویه‌ای، منطق در سراسرِ توابع **پخش** است؛ تغییرِ یک قاعده یعنی پیداکردن و ویرایشِ همه‌ی جاهایی که آن قاعده تکرار شده — و جاماندنِ یکی، یعنی باگ. در کدِ شیءگرا، هر قاعده در **یک** کلاس متمرکز است؛ ریفکتور یعنی تغییرِ همان یک‌جا، و مرزهای روشنِ کلاس‌ها جلوی نشتِ خطا را می‌گیرند. پس شیءگرایی ریفکتور را **امن‌تر** و **محلی‌تر** می‌کند.

**۱۴.** چون در Composition، یک شیء اجزایش را از بیرون می‌گیرد و به **رابط** آن‌ها وابسته است نه به کلاسِ مشخص. در تست می‌توانید یک جزءِ جعلی (که همان رابط را دارد اما رفتارِ ساده و قابل‌کنترل) تزریق کنید. این باعث می‌شود تست بدونِ وابستگی به منابعِ واقعی (که کند، غیرقطعی، یا پرهزینه‌اند) اجرا شود. کلیدِ ماجرا «وابستگی به رابط» است: تا وقتی جعلی همان متدها را داشته باشد، شیءِ اصلی تفاوتی نمی‌بیند.

## بخش ه: پرسشِ تحلیلی

**۱۵.** این جمله قاعده را به یک **مطلقِ نادرست** تبدیل می‌کند. «ترجیح بده» به‌معنای «همیشه» نیست. ارث‌بری هنوز جای درستِ خودش را دارد:
- برای رابطه‌ی واقعیِ **Is-A** (مثلِ `class InsufficientFundsError(PaymentError)`).
- برای استفاده از کلاس‌های پایه‌ی فریمورک‌ها که انتظارِ ارث‌بری دارند.
- وقتی می‌خواهید رابط و رفتارِ والد را کاملاً و یک‌جا به ارث ببرید.

استفاده‌ی کورکورانه از Composition همه‌جا، کد را با لایه‌های واگذاریِ بی‌مورد شلوغ می‌کند. پاسخِ درست: رابطه‌ی واقعیِ دامنه را بسنجید — اگر Is-A است ارث‌بری، اگر Has-A است Composition. قاعده یک **پیش‌فرض** است، نه یک قانونِ بی‌استثنا.

**۱۶.** انتخاب بینِ معماری لایه‌ای و هگزاگونال یک Tradeoffِ واقعی است، نه یک «درست/غلطِ» ثابت:
- **معماری لایه‌ای** ساده، آشنا و سریع برای پیاده‌سازی است؛ برای پروژه‌های کوچک و متوسط اغلب کافی و مناسب‌تر است.
- **هگزاگونال** تست‌پذیرتر و منعطف‌تر است، اما پیچیدگی و کدِ بیشتری می‌طلبد؛ برای سیستم‌های بزرگ با وابستگی‌های بیرونیِ متعدد ارزشش را دارد.

اگر برای یک پروژه‌ی کوچک هگزاگونالِ کامل بزنید، پیچیدگیِ بی‌مورد اضافه کرده‌اید؛ اگر برای یک سیستمِ بزرگ فقط لایه‌ای بزنید، ممکن است بعداً به مشکلِ تست و تغییر بخورید. مهارتِ واقعیِ طراحی، **سنجیدنِ موقعیت** است — اندازه‌ی پروژه، نیازِ تست، تعدادِ وابستگی‌های بیرونی — و انتخابِ مناسب، نه پیرویِ کورکورانه از «بهترین معماری». همین است که «هیچ راه‌حلِ مطلقاً درستی وجود ندارد».
