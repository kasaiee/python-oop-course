## بخش الف: dataclass

**۱.** مشکل: در یک کلاسِ حاملِ داده، نامِ هر فیلد چند بار تکرار می‌شود (در `__init__`، در `__repr__`، در `__eq__`)؛ این هم خسته‌کننده است و هم مستعدِ خطا (اگر یک فیلد را جایی جا بیندازید). `@dataclass` این متدها را از روی تعریفِ فیلدها **خودکار** می‌سازد: `__init__`، `__repr__`، `__eq__` (و بسته به پارامترها، متدهای مقایسه و `__hash__`).

**۲.**

```python
from dataclasses import dataclass

@dataclass
class Product:
    name: str
    price: int
    stock: int = 0
    category: str = "general"

p = Product("Keyboard", 500_000)
print(p)   # Product(name='Keyboard', price=500000, stock=0, category='general')
```

با ۵ خط، هم `__init__` و هم `__repr__` و `__eq__` ساخته شدند.

**۳.**
- `frozen=True`: شیء را **تغییرناپذیر** می‌کند؛ تلاش برای تغییرِ فیلد `FrozenInstanceError` می‌دهد.
- `order=True`: متدهای مقایسه (`__lt__`، `__le__`، `__gt__`، `__ge__`) را می‌سازد تا بتوان نمونه‌ها را با `<` و `>` مقایسه/مرتب کرد.

مزیتِ اضافه‌ی `frozen=True`: چون شیء تغییرناپذیر است، پایتون `__hash__` هم می‌سازد. این یعنی شیء **hashable** می‌شود و می‌توان آن را در `set` یا به‌عنوانِ کلیدِ `dict` استفاده کرد (که برای اشیای تغییرپذیر مجاز نیست).

**۴.** `__post_init__` متدی است که **بلافاصله بعد از `__init__`ِ خودکار** اجرا می‌شود. کاربردِ رایج: اعتبارسنجیِ فیلدها یا محاسبه‌ی فیلدهای مشتق‌شده. مثال:

```python
@dataclass
class Rectangle:
    width: int
    height: int
    area: int = 0
    def __post_init__(self):
        if self.width <= 0 or self.height <= 0:
            raise ValueError("dimensions must be positive")
        self.area = self.width * self.height
```

**۵.** نمی‌توانید `items: list = []` بنویسید چون آن لیست **یک‌بار** ساخته می‌شود و بینِ همه‌ی نمونه‌ها مشترک می‌ماند (همان دامِ پیش‌فرضِ تغییرپذیرِ فصل دهم)؛ به همین دلیل dataclass عمداً خطا می‌دهد. راهِ درست، `field(default_factory=list)` است:

```python
from dataclasses import dataclass, field

@dataclass
class Cart:
    items: list = field(default_factory=list)
```

تفاوت: `default` یک **مقدارِ ثابتِ مشترک** است (برای انواعِ تغییرناپذیر مثلِ عدد و رشته مناسب)؛ اما `default_factory` یک **تابع** است که برای **هر نمونه یک مقدارِ تازه** می‌سازد (برای انواعِ تغییرپذیر مثلِ list/dict لازم است).

## بخش ب: dataclass در برابر Pydantic

**۶.** خطا **نمی‌دهد**. خروجی: `string!`. این نشان می‌دهد `@dataclass` تایپ‌هینت‌ها را در زمانِ اجرا **اعتبارسنجی نمی‌کند**؛ تایپ‌هینت‌ها فقط راهنما (برای برنامه‌نویس و تایپ‌چکِ ایستا) هستند و پایتون نوع را اجبار نمی‌کند. پس `Product("string!")` بدونِ اعتراض ساخته می‌شود، هرچند `price` قرار بود `int` باشد.

**۷.** تفاوتِ اصلی: `@dataclass` نوع را اعتبارسنجی نمی‌کند، اما `Pydantic.BaseModel` نوع را در زمانِ اجرا **اعتبارسنجی و حتی تبدیل** می‌کند (مثلاً `price="500"` را به `int` تبدیل می‌کند و `price="hello"` را با `ValidationError` رد می‌کند). **Tradeoff:**
- `@dataclass`: سبک، بخشی از کتابخانه‌ی استاندارد، سریع — مناسبِ **داده‌ی داخلیِ مطمئن**.
- `Pydantic`: سنگین‌تر (وابستگیِ بیرونی) اما اعتبارسنجیِ قوی می‌دهد — مناسبِ **داده‌ی ورودیِ نامطمئن** (API، فرم، فایلِ کاربر).

انتخاب بستگی دارد به اینکه داده از کجا می‌آید: اگر خودتان ساخته‌اید، dataclass؛ اگر از بیرون آمده و باید مطمئن شوید درست است، Pydantic.

## بخش ج: NamedTuple، TypedDict و Enum

**۸.** `NamedTuple` یک tuple است که فیلدهایش نام دارند — تغییرناپذیر، سبک، مناسبِ رکوردهای کوچکِ ثابت. دو رفتاری که از tuple به ارث می‌برد (و dataclass به‌طورِ پیش‌فرض ندارد):
- **دسترسی با ایندکس:** `p[0]` علاوه بر `p.x`.
- **قابلیتِ unpack:** `x, y = p`.

(همچنین ذاتاً تغییرناپذیر است، مثلِ `dataclass(frozen=True)`.)

**۹.** `TypedDict` ساختارِ یک **دیکشنری** را تعریف می‌کند: کلیدهای مشخص با انواعِ مشخص. خروجیِ نهایی یک **دیکشنریِ معمولی** است (نه یک کلاس/شیءِ خاص)؛ فقط تایپ‌چکر ساختارش را می‌داند. چه زمانی: وقتی با دیکشنری کار می‌کنید (مثلاً داده‌ی JSON) و می‌خواهید ساختارش تایپ‌چک شود، بدونِ تبدیل به کلاس.

**۱۰.** `Enum` مجموعه‌ای از ثابت‌های نام‌دار است. بهتر از رشته/عدد است چون: از **مقادیرِ نامعتبر جلوگیری** می‌کند (فقط اعضای تعریف‌شده مجازند)، برای غلطِ املایی **خطای زودهنگام** می‌دهد، **خودتوضیح** و خواناست، و **قابلِ پیمایش** است. مشکلِ کدِ نمونه: `"penging"` یک غلطِ املایی است، اما چون فقط یک رشته است، پایتون هیچ خطایی نمی‌دهد — شرط صرفاً `False` می‌شود و باگ **بی‌سروصدا** می‌ماند. با Enum، `OrderStatus.PENGING` فوراً `AttributeError` می‌دهد.

**۱۱.** `@unique` تضمین می‌کند که **هیچ دو عضوی مقدارِ تکراری نداشته باشند**؛ اگر دو عضو مقدارِ یکسان بگیرند، هنگامِ تعریفِ کلاس `ValueError` می‌دهد. این از خطاهای سهوی (مثلاً دو حالت که اشتباهاً مقدارِ یکسان گرفته‌اند) جلوگیری می‌کند.

## بخش د: کدنویسی و انتخاب

**۱۲.**

```python
from enum import Enum

class OrderStatus(Enum):
    PENDING = "pending"
    PAID = "paid"
    SHIPPED = "shipped"
    DELIVERED = "delivered"

def can_cancel(status):
    return status in (OrderStatus.PENDING, OrderStatus.PAID)

print(can_cancel(OrderStatus.PENDING))    # True
print(can_cancel(OrderStatus.SHIPPED))    # False
```

**راهِ دیگر:** می‌توان یک مجموعه‌ی «حالت‌های قابلِ لغو» را به‌عنوانِ یک ثابت تعریف کرد تا منطق خواناتر و قابلِ‌گسترش شود:

```python
CANCELABLE = {OrderStatus.PENDING, OrderStatus.PAID}
def can_cancel(status):
    return status in CANCELABLE
```

**۱۳.**

```python
# a) with dataclass(frozen=True)
from dataclasses import dataclass

@dataclass(frozen=True)
class PointDC:
    x: int
    y: int

# b) with NamedTuple
from typing import NamedTuple

class PointNT(NamedTuple):
    x: int
    y: int
```

**یک تفاوتِ عملی:** `PointNT` چون tuple است، از **unpack و ایندکس** پشتیبانی می‌کند (`x, y = p` و `p[0]`)، اما `PointDC` این‌ها را ندارد (فقط `p.x`/`p.y`). در عوض `dataclass` انعطافِ بیشتری برای افزودنِ متد و پارامترهای گوناگون می‌دهد و می‌تواند تغییرپذیر هم باشد (اگر `frozen` نباشد).

**۱۴.**

| سناریو | ابزارِ مناسب |
|--------|--------------|
| مجموعه‌ی محدودی از نقش‌های کاربر (admin/editor/viewer) | `Enum` |
| داده‌ی ورودیِ یک API که باید اعتبارسنجی شود | `Pydantic` |
| رکوردِ مختصاتِ `(lat, lng)` که مثلِ tuple هم unpack شود | `NamedTuple` |
| ساختارِ یک پاسخِ JSON که فقط می‌خواهید تایپ‌چک شود | `TypedDict` |

## بخش ه: پرسشِ تحلیلی

**۱۵.** یک کلاسِ معمولی بهتر از dataclass است وقتی کلاسِ شما **رفتارِ غنی** دارد: منطقِ پیچیده، متدهای زیاد، حالتِ درونیِ کنترل‌شده. dataclass برای کلاس‌هایی طراحی شده که **عمدتاً حاملِ داده‌اند** و کدِ تکراری‌شان اذیت‌کننده است؛ اگر جوهرِ کلاس «رفتار» باشد نه «داده»، dataclass ارزشِ چندانی اضافه نمی‌کند و حتی ممکن است گمراه‌کننده باشد.

دو سوالِ راهنما:
- **«آیا این عمدتاً داده است یا رفتار؟»** اگر داده → dataclass/NamedTuple/TypedDict؛ اگر رفتار → کلاسِ معمولی.
- **«آیا باید تغییرپذیر باشد؟»** اگر باید ثابت بماند → `frozen=True` یا `NamedTuple`؛ اگر تغییر می‌کند → dataclassِ معمولی.

پاسخِ این دو، ابزارِ درست را مشخص می‌کند.

**۱۶.** این جمله **نادرست** است چون «کمترکد بودن» را تنها معیار می‌گیرد. مشکلات:
- dataclass برای کلاس‌های **رفتارمحور** مزیتی ندارد و ممکن است این پیام را بدهد که کلاس فقط یک ظرفِ داده است، درحالی‌که نیست.
- اگر کلاس منطقِ پیچیده و ثابت‌های داخلیِ زیاد دارد، تبدیلش به dataclass چیزی ساده نمی‌کند.
- انتخابِ ابزار باید بر اساسِ **ماهیتِ** کلاس باشد (داده یا رفتار؟ تغییرپذیر یا نه؟)، نه صرفاً کوتاه‌تر شدنِ کد.

پاسخِ درست: dataclass یک ابزارِ عالی برای حذفِ کدِ تکراریِ کلاس‌های **حاملِ داده** است، نه یک قالبِ همگانی برای همه‌ی کلاس‌ها. مثلِ همیشه، ابزار را متناسبِ مسئله انتخاب کنید.
