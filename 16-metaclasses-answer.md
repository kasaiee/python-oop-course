## بخش الف: مفهوم متاکلاس

**۱.** یعنی خودِ کلاس‌ها هم شیء‌اند — شیئی که می‌تواند اشیای دیگر بسازد. خروجی:

```
<class 'type'>
<class '__main__.User'>
```

- `type(User)` برابرِ `type` است → نشان می‌دهد کلاسِ `User` خودش نمونه‌ای از `type` است (یعنی `type` کلاسِ `User` را ساخته).
- `type(User())` برابرِ `User` است → نشان می‌دهد یک شیءِ معمولی، نمونه‌ای از `User` است.

پس دو سطح داریم: شیء نمونه‌ی کلاس است، و کلاس نمونه‌ی متاکلاس.

**۲.** متاکلاس «کلاسی است که کلاس‌ها را می‌سازد». همان‌طور که اشیای معمولی نمونه‌ی یک کلاس‌اند، **کلاس‌ها نمونه‌ی یک متاکلاس‌اند**. متاکلاسِ پیش‌فرضِ پایتون `type` است؛ هر کلاسی که تعریف می‌کنید، در پشتِ صحنه توسطِ `type` ساخته می‌شود. رابطه به‌شکلِ زنجیره‌ای است: `object → instance of → class → instance of → metaclass`.

**۳.**

```python
def bark(self):
    return "Woof"

Dog = type("Dog", (), {"legs": 4, "bark": bark})
#          name    parents  namespace (attributes and methods)

d = Dog()
print(d.legs)     # 4
print(d.bark())   # Woof
```

سه آرگومانِ `type`: نامِ کلاس (رشته)، تاپلِ والدها، و دیکشنریِ فضای‌نام (attributeها و متدها).

**۴.** یعنی وقتی می‌نویسید `class Dog: ...`، پایتون در پشتِ صحنه دقیقاً `type("Dog", bases, namespace)` را صدا می‌زند: نامِ کلاس، والدهایش، و بدنه‌اش (که به یک دیکشنری تبدیل می‌شود) را جمع می‌کند و به `type` می‌دهد تا کلاس را بسازد. پس `class` یک **راهِ خواناتر** برای همان کاری است که `type` انجام می‌دهد — نه چیزی جادویی و جدا.

## بخش ب: متاکلاس سفارشی و مقایسه

**۵.** پیام **هنگامِ تعریفِ کلاسِ `Service`** چاپ می‌شود، نه هنگامِ ساختِ نمونه. چون `__new__`ِ متاکلاس مسئولِ **ساختنِ خودِ کلاس** است، و ساختِ کلاس همان لحظه‌ای اتفاق می‌افتد که پایتون به تعریفِ `class Service(...)` می‌رسد. ساختِ نمونه (`Service()`) بعداً و توسطِ `__init__`/`__new__`ِ خودِ کلاس انجام می‌شود، نه متاکلاس. (این تفاوت را با `__init_subclass__` فصل ششم مقایسه کنید که آن هم هنگامِ تعریف عمل می‌کند.)

**۶.**

| ویژگی | دکوراتور کلاسی | متاکلاس |
|-------|----------------|---------|
| زمانِ عمل | **بعد از** ساختِ کلاس | **حین** ساختِ کلاس |
| انتقال به زیرکلاس‌ها | منتقل نمی‌شود | منتقل می‌شود |
| کاربردِ معمول | تغییرِ یک کلاسِ مشخص | کنترلِ خانواده‌ای از کلاس‌ها |

**۷.**
- الف) **دکوراتورِ کلاسی** — ساده‌ترین راه برای تغییرِ یک کلاسِ مشخص.
- ب) **`__init_subclass__`** — دقیقاً برای «کاری هنگامِ تعریفِ زیرکلاس» طراحی شده و از متاکلاس ساده‌تر است.
- ج) **متاکلاس** — فقط اینجا که واقعاً باید در قلبِ فرایندِ ساختِ همه‌ی کلاس‌های خانواده مداخله‌ی عمیق کنید.

## بخش ج: ابزارهای مرتبط

**۸.** `__set_name__` (از پایتون ۳.۶) به یک Descriptor اجازه می‌دهد **نامِ attributeی که به آن نسبت داده شده** را بداند. مثلاً وقتی می‌نویسید `username = Field()`، متدِ `__set_name__` صدا زده می‌شود و `name="username"` را دریافت می‌کند. ارتباط با dataclass/ORM: همین مکانیزم است که به آن‌ها اجازه می‌دهد نامِ فیلدها را **خودکار** بفهمند (بدونِ اینکه شما نام را تکرار کنید). چرا نیاز به متاکلاس را از بین برد: پیش از `__set_name__`، برای فهمیدنِ نامِ فیلدها هنگامِ ساختِ کلاس، مجبور بودید یک متاکلاس بنویسید؛ حالا این قابلیت به‌صورتِ ساده و داخلی فراهم است.

**۹.** `__class_getitem__` (از پایتون ۳.۷) تعیین می‌کند وقتی از سینتکسِ `ClassName[X]` استفاده می‌کنید چه اتفاقی بیفتد. همین است که به `Repository[User]` و `list[int]` و `dict[str, int]` معنا می‌دهد — بدونِ نیاز به متاکلاس. یعنی نوشتنِ `Container[int]` در واقع `Container.__class_getitem__(int)` را صدا می‌زند.

**۱۰.** `__instancecheck__` رفتارِ `isinstance` را سفارشی می‌کند. خروجی:

```
True
True
False
```

چون `LenMeta.__instancecheck__` می‌گوید «هر چیزی که `__len__` دارد نمونه‌ی `Sized` حساب شود». لیست و رشته `__len__` دارند (پس `True`)، اما عدد `42` ندارد (پس `False`). ارتباط با Protocol و `collections.abc`: همین مکانیزم پشتِ `isinstance(x, Iterable)` کار می‌کند — بررسیِ **قابلیت** به‌جای ارث‌بریِ واقعی، که همان روحِ Structural Subtyping و Protocolهای فصل نهم است.

## بخش د: کدنویسی

**۱۱.**

```python
def add_repr(cls):
    cls.__repr__ = lambda self: f"<{cls.__name__}>"
    return cls

@add_repr
class Point:
    pass

print(repr(Point()))     # <Point>
```

دکوراتور کلاس را می‌گیرد، `__repr__` را به آن اضافه می‌کند و همان کلاس را برمی‌گرداند. این «بعد از» ساختِ کلاس عمل می‌کند.

**۱۲.**

```python
class Field:
    def __set_name__(self, owner, name):
        self.name = name

    def __get__(self, obj, objtype=None):
        if obj is None:
            return self
        return obj.__dict__.get(self.name)

    def __set__(self, obj, value):
        obj.__dict__[self.name] = value

class User:
    username = Field()
    email = Field()

u = User()
u.username = "ali"
print(u.username)      # ali
print(u.__dict__)      # {'username': 'ali'}
```

`__set_name__` خودکار `name` را از نامِ attribute (`username`/`email`) می‌گیرد، پس لازم نیست نام را دستی تکرار کنید — همان کاری که ORMها می‌کنند.

## بخش ه: پرسشِ تحلیلی

**۱۳.** این جمله یعنی متاکلاس یک ابزارِ **بسیار تخصصی** است که فقط در موارد نادر (نوشتنِ فریمورک/کتابخانه) لازم می‌شود؛ اگر در کارِ روزمره حس می‌کنید «شاید به متاکلاس نیاز دارم»، تقریباً همیشه یک ابزارِ ساده‌تر (دکوراتور، `__init_subclass__`، Descriptor) کفایت می‌کند. پس چرا می‌خوانیمش؟ نه برای اینکه بنویسیمش، بلکه برای **درک**: وقتی کدِ جنگو، SQLAlchemy یا Pydantic را می‌خوانید که از متاکلاس استفاده می‌کنند، دیگر برایتان «جادو» نباشد و بفهمید پشتِ پرده چه می‌گذرد. فهمیدن ≠ استفاده‌کردن.

**۱۴.** برای «انجامِ کاری هنگامِ تعریفِ زیرکلاس»، ترتیب از ساده به پیچیده:
1. **`__init_subclass__`** (فصل ششم) — ساده‌ترین، دقیقاً برای همین کار ساخته شده.
2. **دکوراتورِ کلاسی** — اگر روی هر کلاس صریحاً بگذارید.
3. **متاکلاس** — فقط اگر هیچ‌کدام کافی نبودند و نیاز به مداخله‌ی عمیق در ساختِ کلاس دارید.

چرا نباید مستقیم سراغِ متاکلاس رفت: متاکلاس پیچیده است، خواندنش برای دیگران سخت است، و با ارث‌بریِ چندگانه می‌تواند تصادمِ متاکلاس بسازد. انتخابِ ساده‌ترین ابزارِ کافی، کد را خواناتر و قابل‌نگهداری‌تر می‌کند — و این خودش نشانه‌ی بلوغِ مهندسی است. قدرتِ خام (متاکلاس) را فقط وقتی خرج کنید که واقعاً لازم است.

## بخش و: متاکلاس در عمل

**۱۵.**

```
creating class Base
creating class Child
--- now instantiating ---
```

دو خطِ اول هنگامِ **تعریفِ** کلاس‌ها چاپ می‌شوند — `Meta.__new__` زمانی اجرا می‌شود که خودِ کلاس ساخته می‌شود، و `Child` هم چون از `Base` ارث می‌برد، متاکلاسِ آن را به ارث برده (متاکلاس، برخلافِ دکوراتورِ کلاسی، به زیرکلاس‌ها منتقل می‌شود). بعد از خطِ جداکننده، نمونه‌سازی‌ها **هیچ چیزی چاپ نمی‌کنند**: ساختِ شیء از مسیرِ `Meta.__new__` نمی‌گذرد؛ آن متد فقط برای ساختِ *کلاس* بود. (نمونه‌سازی از `Meta.__call__` می‌گذرد که override نکرده‌ایم.)

**۱۶.**

```python
class DocMeta(type):
    def __new__(mcs, name, bases, namespace):
        for attr, value in namespace.items():
            if callable(value) and not attr.startswith("_"):
                if not getattr(value, "__doc__", None):
                    raise TypeError(f"public method '{name}.{attr}' needs a docstring")
        return super().__new__(mcs, name, bases, namespace)

class GoodService(metaclass=DocMeta):
    def start(self):
        """Start the service."""
    def _internal(self):
        pass                          # private — no docstring required

print("GoodService created fine")

try:
    class BadService(metaclass=DocMeta):
        def stop(self):
            pass                      # public and undocumented!
except TypeError as e:
    print(e)                          # public method 'BadService.stop' needs a docstring
```

خطا در لحظه‌ی **تعریفِ** کلاس می‌ترکد، نه در اولین استفاده — همان فلسفه‌ی «شکستِ زودهنگام» که ABC هم دنبال می‌کند.

**۱۷.**

```python
class SingletonMeta(type):
    def __call__(cls, *args, **kwargs):
        if not hasattr(cls, "_instance"):
            cls._instance = super().__call__(*args, **kwargs)
        return cls._instance

class AppConfig(metaclass=SingletonMeta):
    def __init__(self):
        self.debug = False

a = AppConfig()
b = AppConfig()
print(a is b)        # True — one instance, enforced by the metaclass
```

نقد: متاکلاس فقط **مکانیزم** را شیک‌تر کرده، نه **مشکل** را. این هنوز حالتِ سراسریِ پنهان است: تست‌ها به هم نشت می‌کنند (نمونه بینِ تست‌ها زنده می‌ماند)، وابستگی‌ها نامرئی‌اند و جایگزینی با بدل سخت است. تنها مزیتِ واقعی نسبت به `__new__`، تمیزتر بودنِ نقطه‌ی کنترل (`__call__`) و قابلِ‌استفاده‌بودن روی چند کلاس است — اما توصیه‌ی فصل ۱۲ سرِ جایش است: قبل از Singleton، به ماژول یا تزریقِ وابستگی فکر کنید.

**۱۸.**

```python
class PluginBase:
    registry = {}

    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        PluginBase.registry[cls.__name__.lower()] = cls

class EmailPlugin(PluginBase): ...
class SmsPlugin(PluginBase): ...

print(sorted(PluginBase.registry))    # ['emailplugin', 'smsplugin']
```

در code review نسخه‌ی `__init_subclass__` را تأیید می‌کنم. هر دو دقیقاً یک کار می‌کنند، اما: نسخه‌ی دوم کوتاه‌تر است، نیازی به شرطِ «خودِ پایه را رد کن» ندارد (`__init_subclass__` فقط برای زیرکلاس‌ها صدا می‌شود)، برای خواننده‌ی معمولی قابلِ‌فهم‌تر است، و با متاکلاس‌های دیگر (مثلاً ABCMeta) تداخل نمی‌کند — تداخلِ دو متاکلاس یکی از دردسرهای کلاسیک است. متاکلاس را برای کاری نگه می‌داریم که `__init_subclass__` از پسش برنیاید: دست‌کاریِ namespace *قبل از* ساختِ کلاس، یا کنترلِ نمونه‌سازی مثل تمرینِ ۱۷. قدرتِ بیشتر، بدونِ نیازِ بیشتر، فقط هزینه‌ی فهم اضافه می‌کند.
