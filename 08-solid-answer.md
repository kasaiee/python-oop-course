## بخش الف: Single Responsibility و Open/Closed

**۱.** SRP می‌گوید **هر کلاس باید فقط یک دلیل برای تغییر داشته باشد** (یک مسئولیتِ منسجم). `send_email` نقضِ SRP است چون به هویتِ کاربر ربطی ندارد. دو دلیلِ مستقلِ تغییرِ این کلاس:
- تغییرِ نحوه‌ی **ذخیره‌سازی** (مثلاً عوض‌شدنِ دیتابیس) → متدِ `save`.
- تغییرِ نحوه‌ی **ارسالِ ایمیل** (مثلاً عوض‌شدنِ سرویسِ ایمیل) → متدِ `send_email`.

این دو هیچ ربطی به هم ندارند و باید در کلاس‌های جدا باشند (`UserRepository` و `EmailService`)، تا `User` فقط هویتِ کاربر بماند.

**۲.** OCP می‌گوید کلاس‌ها باید **برای توسعه باز و برای تغییر بسته** باشند. کدِ نمونه نقض می‌کند چون هر روشِ پرداختِ جدید یعنی افزودنِ یک `elif` تازه و **ویرایشِ** همین کلاس. این خطرناک است چون شما یک کدِ **آزموده‌شده و پایدار** را دستکاری می‌کنید و ریسکِ شکستنِ روش‌های موجود (که قبلاً کار می‌کردند) را می‌پذیرید — هر ویرایش، یک فرصتِ تازه برای باگ است.

**۳.** بازنویسی با OCP:

```python
class PaymentMethod:
    def pay(self, amount): ...

class CardPayment(PaymentMethod):
    def pay(self, amount):
        print(f"Card: {amount}")

class WalletPayment(PaymentMethod):
    def pay(self, amount):
        print(f"Wallet: {amount}")

class PaymentProcessor:
    def process(self, method: PaymentMethod, amount):
        method.pay(amount)          # it doesn't know and doesn't need to know which method

# adding a new method without touching existing code:
class CryptoPayment(PaymentMethod):
    def pay(self, amount):
        print(f"Crypto: {amount}")

processor = PaymentProcessor()
processor.process(CardPayment(), 1000)
processor.process(CryptoPayment(), 2000)   # the old code stayed untouched
```

`PaymentProcessor` حالا «بسته برای تغییر» ولی «باز برای توسعه» است.

## بخش ب: Liskov Substitution و Interface Segregation

**۴.** LSP می‌گوید **هرجا از والد استفاده می‌شود باید بتوان زیرکلاس را جایگزین کرد، بی‌آنکه رفتارِ درست بشکند**. `ReadOnlyStorage` این را نقض می‌کند چون متدِ `save` را که در قراردادِ `FileStorage` هست، با `NotImplementedError` خنثی کرده. هر کدی که یک `FileStorage` می‌گیرد و انتظار دارد `save` کار کند، با یک `ReadOnlyStorage` به‌شکلِ غیرمنتظره **می‌شکند** — پس `ReadOnlyStorage` جانشینِ سالمِ والد نیست.

**۵.** نشانه‌های نقضِ LSP:
- زیرکلاس یک متدِ والد را با `NotImplementedError` یا `pass`ِ خالی «خنثی» می‌کند.
- زیرکلاس شرط‌های ورودیِ **سخت‌گیرانه‌تری** می‌گذارد یا خروجیِ **ضعیف‌تری** برمی‌گرداند.
- مجبورید پیش از استفاده از زیرکلاس، نوعش را چک کنید (`if isinstance(x, ReadOnly)`) — نشانه‌ی این است که زیرکلاس واقعاً جایگزین‌پذیر نیست.

**۶.** ISP می‌گوید **هیچ کلاسی نباید مجبور به وابستگی به متدهایی باشد که استفاده نمی‌کند**. `RobotWorker`:
- **نقضِ ISP:** مجبور شده به `eat` و `sleep` وابسته شود که برایش بی‌معنا هستند (رباتی که غذا نمی‌خورد).
- **نقضِ LSP:** چون این متدها `NotImplementedError` می‌اندازند، اگر کدی یک `Worker` بگیرد و `eat` را صدا بزند، با `RobotWorker` می‌شکند.

پس یک اینترفیسِ چاق، اغلب هر دو اصل را با هم نقض می‌کند.

**۷.** شکستنِ اینترفیس:

```python
class Workable:
    def work(self): ...

class Eatable:
    def eat(self): ...

class Sleepable:
    def sleep(self): ...

class HumanWorker(Workable, Eatable, Sleepable):
    def work(self): print("works")
    def eat(self): print("eats")
    def sleep(self): print("sleeps")

class RobotWorker(Workable):        # only what it actually needs
    def work(self): print("works")
```

`RobotWorker` دیگر مجبور نیست وانمود کند غذا می‌خورد.

## بخش ج: Dependency Inversion

**۸.** DIP می‌گوید **ماژول‌های سطح‌بالا و سطح‌پایین هر دو باید به انتزاع وابسته باشند، نه یکی به دیگری**. مشکلِ کدِ نمونه: `UserService` مستقیماً `MySQLDatabase()` را داخلِ خودش می‌سازد و به آن **میخکوب** شده. اگر بخواهید به PostgreSQL کوچ کنید، یا در تست از یک دیتابیسِ حافظه‌ای استفاده کنید، مجبورید خودِ `UserService` را دستکاری کنید — یعنی یک ماژولِ سطح‌بالا به یک جزئیاتِ سطح‌پایین گره خورده.

**۹.** بازنویسی با DIP:

```python
class Database:                          # abstraction (interface)
    def save(self, data): ...

class MySQLDatabase(Database):
    def save(self, data):
        print(f"Saved in MySQL: {data}")

class InMemoryDatabase(Database):        # suitable for testing
    def __init__(self):
        self.store = []
    def save(self, data):
        self.store.append(data)

class UserService:
    def __init__(self, db: Database):    # depends on the abstraction, not the implementation
        self.db = db
    def register(self, user):
        self.db.save(user)

service = UserService(MySQLDatabase())         # in production
test_service = UserService(InMemoryDatabase()) # in tests
```

**راهِ دیگر:** در پایتون به‌لطفِ Duck Typing، حتی لازم نیست `Database` را صریحاً تعریف کنید؛ هر شیئی که متدِ `save` داشته باشد کار می‌کند. اما تعریفِ صریحِ انتزاع (با ABC یا Protocol در فصل بعد) قرارداد را روشن‌تر و کد را مستندتر می‌کند.

**۱۰.** Dependency Injection یعنی به‌جای اینکه یک شیء وابستگی‌هایش را **خودش بسازد**، آن‌ها را از **بیرون دریافت** کند (معمولاً از طریقِ سازنده). این مکملِ عملیِ DIP است: DIP می‌گوید «به انتزاع وابسته باش»، و DI **راهِ رساندنِ** آن پیاده‌سازیِ مشخص به شیء است. ارتباط با تست‌پذیری: چون وابستگی از بیرون تزریق می‌شود، در تست می‌توانید یک نسخه‌ی جعلی/ساده (مثلِ `InMemoryDatabase`) تزریق کنید و بدونِ منابعِ واقعی (دیتابیسِ کند) تست کنید.

## بخش د: تشخیص و اصولِ مکمل

**۱۱.**
- الف) **SRP** — یک کلاس با سه مسئولیتِ نامرتبط (سبد خرید، لاگ، گزارشِ HTML).
- ب) **OCP** — افزودنِ ویژگی نیازمندِ ویرایشِ کدِ موجود (`elif` تازه) است.
- ج) **DIP** — وابستگیِ میخکوب به یک پیاده‌سازیِ مشخص به‌جای انتزاع.
- د) **LSP** (و اغلب **ISP**) — زیرکلاسی که نمی‌تواند قراردادِ والد را کامل رعایت کند.

**۱۲.**
- **Composition over Inheritance:** ترکیب را بر ارث‌بری ترجیح بده (رابطه‌ی Has-A به‌جای Is-Aِ تحمیلی).
- **Law of Demeter:** با غریبه‌ها حرف نزن؛ از زنجیره‌ی `a.b.c.d` پرهیز کن.
- **DRY** (Don't Repeat Yourself): خودت را تکرار نکن؛ منطقِ تکراری را یک‌جا جمع کن.
- **YAGNI** (You Aren't Gonna Need It): پیش از نیازِ واقعی، پیچیدگی و انتزاع نساز.
- **KISS** (Keep It Simple): تا حدِ ممکن ساده نگه دار.

## بخش ه: پرسشِ تحلیلی

**۱۳.** این جمله **نادرست** است و SOLID را از راهنما به یک اجبارِ کورکورانه تبدیل می‌کند. رعایتِ افراطی خطر دارد:
- تقسیمِ همه‌چیز به کوچک‌ترین کلاس‌ها → **انفجارِ کلاس‌های ریزِ کم‌محتوا** که دنبال‌کردنشان سخت است.
- انتزاعی‌کردنِ همه‌چیز → لایه‌های غیرمستقیمِ زیاد که کد را برای مسئله‌ای که شاید هرگز تغییر نکند، بی‌جهت پیچیده می‌کنند (نقضِ YAGNI و KISS).

پاسخِ درست: SOLID را وقتی به کار ببرید که مسئله‌ی واقعی‌ای (تغییرپذیری، تکرار، شکنندگی) حل می‌کند، نه چون «باید». یک کلاسِ ساده که هرگز تغییر نمی‌کند، نیازی به پنج لایه‌ی انتزاع ندارد.

**۱۴.** **SRP افراطی** می‌تواند به ده‌ها کلاسِ ریزِ کم‌محتوا منجر شود که هرکدام فقط یک متدِ کوچک دارند؛ آن‌وقت دنبال‌کردنِ جریانِ منطق بینِ این‌همه کلاس، خودش سخت و گیج‌کننده می‌شود — یعنی «خوانایی» که قرار بود بهتر شود، بدتر می‌شود.

**تنشِ OCP و YAGNI:** OCP تشویق می‌کند لایه‌های انتزاعی بسازید تا بعداً بدونِ تغییر بتوانید توسعه دهید. اما اگر برای بخشی از کد که **هرگز تغییر نخواهد کرد** این انتزاع‌ها را بسازید، پیچیدگیِ بی‌مصرف اضافه کرده‌اید — دقیقاً همان چیزی که YAGNI («لازمش نخواهی داشت») هشدار می‌دهد. تعادل: انتزاع را جایی بگذارید که **احتمالِ واقعیِ تغییر** وجود دارد، نه همه‌جا.

**۱۵.** «قطب‌نما، نه قفس» یعنی SOLID به شما **جهت** می‌دهد اما شما را **محبوس** نمی‌کند. قطب‌نما مسیرِ کلی را نشان می‌دهد، اما تصمیمِ نهایی (کِی کاملاً پیروی کنی و کِی به‌نفعِ سادگی از آن بگذری) با شماست و به **موقعیت** بستگی دارد. یک قفس شما را مجبور می‌کند همیشه و بی‌چون‌وچرا اطاعت کنید؛ اما مهندسِ باتجربه می‌داند این اصول ابزارند، و توانایی سنجیدنِ Tradeoff — یعنی تشخیصِ زمانِ درستِ به‌کارگیریِ هر اصل — خودش نشانه‌ی بلوغِ مهندسی است.
