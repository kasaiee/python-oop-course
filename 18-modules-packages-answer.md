## بخش الف: ماژول و پکیج

**۱.** هر فایلِ `.py` یک **ماژول** است. جمله‌ی «config بارگذاری شد» فقط **یک بار** چاپ می‌شود. چون وقتی ماژولی را `import` می‌کنید، پایتون آن را **فقط یک‌بار** اجرا می‌کند و نتیجه را کش می‌کند؛ importهای بعدی همان شیءِ ماژولِ کش‌شده را برمی‌گردانند و بدنه‌اش دوباره اجرا نمی‌شود. این دقیقاً رفتارِ Singleton است: یک نمونه‌ی واحد که در سراسرِ برنامه به اشتراک گذاشته می‌شود.

**۲.** ماژول یک فایلِ `.py` است؛ **پکیج** پوشه‌ای است که چند ماژول را گروه‌بندی می‌کند. نقشِ `__init__.py`: به پایتون می‌گوید این پوشه یک پکیج است. می‌تواند خالی باشد، یا کارِ مفیدی بکند — مثلاً با import‌کردنِ نام‌های مهم از ماژول‌های داخلی، یک **رابطِ عمومیِ تمیز** برای پکیج بسازد.

**۳.** `__all__` فهرستِ نام‌هایی را مشخص می‌کند که با `from module import *` وارد می‌شوند. یعنی رابطِ عمومیِ ماژول را تعیین می‌کند و از واردشدنِ نام‌های داخلی جلوگیری می‌کند. حتی بدونِ `import *` هم مفید است چون به‌عنوانِ **مستندسازی** به خواننده می‌گوید «این‌ها API عمومیِ من‌اند؛ بقیه جزئیاتِ داخلی‌اند».

**۴.**
- `import shop.models`: باید با مسیرِ کامل استفاده کنید (`shop.models.User`)؛ **روشن‌تر** است (همیشه می‌دانید `User` از کجا آمده) اما **پرگوتر**.
- `from shop.models import User`: کوتاه‌تر است (`User()`)، اما در فایلی با importهای زیاد **منشأِ نام** کمتر پیداست.

Tradeoff: برای نام‌های پرکاربرد `from import`، و وقتی چند ماژول نامِ مشابه دارند (`shop.models.User` و `auth.models.User`)، `import`ِ ماژول برای رفعِ ابهام بهتر است.

## بخش ب: کاربردها و طراحی

**۵.** چون ماژول ذاتاً «یک‌بار اجرا و کش‌شده» است، هر متغیرِ سطحِ‌ماژول به‌طورِ خودکار یک نمونه‌ی واحدِ مشترک در سراسرِ برنامه می‌شود. پس اگر یک حالتِ سراسریِ مشترک می‌خواهید (تنظیمات، یک اتصالِ مشترک)، لازم نیست کلاسِ Singleton با `__new__`ِ پیچیده بنویسید؛ فقط یک متغیر یا شیء در سطحِ ماژول بگذارید و هرجا import کنید. این ساده‌ترین و پایتونی‌ترین شکلِ Singleton است.

**۶.** Factoryِ ماژولی (مثلِ `create_notifier`) به کاربر اجازه می‌دهد بدونِ دانستنِ کلاس‌های داخلی (`EmailNotifier`، `SMSNotifier`) فقط با یک رشته (`"email"`) شیءِ مناسب را بگیرد. مزیت: جزئیاتِ کلاس‌ها پنهان می‌ماند (انتزاع)، و افزودنِ نوعِ جدید فقط تغییرِ داخلیِ Factory است، بی‌آنکه رابطِ عمومی عوض شود. این با اصلِ **Open/Closed** (باز برای توسعه، بسته برای تغییر) هم‌راستاست.

**۷.** هر ماژول یک مسئولیتِ منسجم دارد: `models.py` فقط کلاس‌های داده، `services.py` فقط منطقِ کسب‌وکار، `repositories.py` فقط دسترسی به داده، `exceptions.py` فقط استثناها. این دقیقاً همان **Single Responsibility** است، اما در سطحِ **فایل** به‌جای کلاس. و این چیدمان، لایه‌های معماریِ لایه‌ایِ فصل هفتم (Presentation / Business / Data) را به ساختارِ فیزیکیِ پوشه‌ها و فایل‌ها ترجمه می‌کند، با وابستگی‌هایی که از لایه‌ی بالا به پایین جاری‌اند.

## بخش ج: کدنویسی و تشخیص

**۸.**

```python
# shop/__init__.py
from .models import User, Product
from .services import OrderService

# now the user can write:
#   from shop import User, OrderService
# instead of the long path:
#   from shop.models import User
#   from shop.services import OrderService
```

با این کار، جزئیاتِ داخلی (اینکه `User` در `models.py` است) پنهان می‌ماند و کاربر یک رابطِ ساده می‌بیند.

**۹.**

```python
class EmailNotifier:
    def send(self, msg):
        return f"Email: {msg}"

class SMSNotifier:
    def send(self, msg):
        return f"SMS: {msg}"

def create_notifier(kind: str):
    notifiers = {
        "email": EmailNotifier,
        "sms": SMSNotifier,
    }
    if kind not in notifiers:
        raise ValueError(f"unknown type: {kind}")
    return notifiers[kind]()

notifier = create_notifier("email")
print(notifier.send("Hello"))     # Email: Hello
```

**راهِ دیگر:** به‌جای دیکشنری، می‌توان از یک زنجیره‌ی `if/elif` استفاده کرد؛ اما نگاشتِ دیکشنری تمیزتر است و افزودنِ نوعِ جدید فقط یک ردیف است.

**۱۰.**

```python
# shop/database.py
class _Database:
    def __init__(self):
        self.connection = "single connection"
    def query(self, sql):
        return f"Result: {sql}"

db = _Database()      # a single instance at module level

# anywhere in the program:
#   from shop.database import db
#   db.query("SELECT ...")
```

چرا واقعاً Singleton است: چون ماژول فقط یک‌بار اجرا می‌شود، خطِ `db = _Database()` فقط یک‌بار اجرا می‌شود و همه‌ی importها همان نمونه را می‌گیرند. کلاسِ `_Database` با `_` خصوصی شده تا کسی وسوسه نشود نمونه‌ی دومی بسازد.

**۱۱.**
- `__init__.py` — رابطِ عمومیِ پکیج؛ انتزاع (فصل ۴/۱۸).
- `models.py` — کلاس‌های داده (مثلِ `@dataclass`، فصل ۱۰).
- `exceptions.py` — سلسله‌مراتبِ استثناها (فصل ۱۱).
- `repositories.py` — لایه‌ی دسترسی به داده (فصل ۷).
- `services.py` — لایه‌ی منطقِ کسب‌وکار (فصل ۷).
- `database.py` — Singletonِ ماژولی (اتصالِ مشترک، فصل ۱۸).

هر فایل یک مسئولیت؛ کسی که پروژه را باز کند از روی نامِ فایل‌ها می‌فهمد هر چیزی کجاست.

## بخش د: پرسشِ تحلیلی

**۱۲.** اصولِ شیءگرایی محدود به داخلِ یک کلاس نیستند؛ در سطحِ فایل‌ها هم اعمال می‌شوند:
- **Single Responsibility در سطحِ ماژول:** هر فایل یک مسئولیتِ منسجم دارد (مدل‌ها، سرویس‌ها، استثناها جدا)، درست مثلِ اینکه هر کلاس یک مسئولیت دارد.
- **انتزاع با `__init__.py`:** رابطِ عمومیِ تمیز می‌سازد و ساختارِ داخلی (اینکه کدام کلاس در کدام فایل است) را پنهان می‌کند — دقیقاً مثلِ اینکه یک کلاس جزئیاتش را پشتِ متدهای عمومی پنهان می‌کند.

نتیجه: یک پروژه‌ی خوش‌ساختار، بدونِ خواندنِ حتی یک خط کد، از روی نامِ پوشه‌ها و فایل‌هایش خودش را توضیح می‌دهد.

**۱۳.** این جمله **نادرست** است، چون در پایتون یک راهِ ساده‌ترِ داخلی وجود دارد: **ماژول**. چون ماژول فقط یک‌بار اجرا و کش می‌شود، یک متغیر یا شیءِ سطحِ‌ماژول خودش یک Singletonِ طبیعی است — بدونِ نیاز به کلاسِ Singletonِ پیچیده با `__new__`. پس برای بیشترِ نیازهای «حالتِ سراسریِ مشترک»، الگوی ماژولی کافی و تمیزتر است. نوشتنِ کلاسِ Singleton اغلب پیچیدگیِ اضافه است.

**۱۴.** تزریقِ وابستگی (پاس‌دادنِ صریحِ شیء به جایی که لازمش دارد) اغلب تمیزتر است چون:
- **تست‌پذیری:** می‌توانید در تست یک نسخه‌ی جعلی/ساده تزریق کنید؛ اما Singletonِ سراسری یک حالتِ ثابت است که جایگزین‌کردنش در تست سخت است.
- **شفافیتِ وابستگی:** وقتی وابستگی به‌صراحت پاس داده می‌شود، از امضای تابع/سازنده پیداست که چه چیزی لازم است؛ اما Singleton یک وابستگیِ **پنهان** می‌سازد (کد از یک متغیرِ سراسری استفاده می‌کند بی‌آنکه در امضا دیده شود).

Singleton (چه ماژولی چه کلاسی) حالتِ سراسریِ مشترک می‌سازد که تست را سخت‌تر و وابستگی‌ها را پنهان‌تر می‌کند. استفاده از آن فقط برای منابعی که **واقعاً باید واحد باشند** (مثلِ استخرِ اتصالِ دیتابیس) موجه است؛ در بقیه‌ی موارد، تزریقِ وابستگی انتخابِ بهتری است.

## بخش ه: طراحی پکیج در عمل

**۱۵.**

```
orderapp/
├── __init__.py          # رابطِ عمومی: re-export چیزهایی که کاربرِ پکیج لازم دارد
├── models.py            # Order و User — کلاس‌های داده (dataclass)، بدونِ منطقِ سنگین
├── discounts.py         # قوانینِ تخفیف — استراتژی‌های قابلِ‌تعویض (فصل ۱۲)
├── storage.py           # ذخیره/بازیابیِ JSON — تنها جایی که فایل لمس می‌شود
├── exceptions.py        # سلسله‌مراتبِ OrderAppError و فرزندان
└── cli.py               # رابطِ خط فرمان — فقط ترجمه‌ی ورودیِ کاربر به سرویس‌ها
```

منطقِ چیدمان: هر فایل یک مسئولیت (SRP در سطحِ فایل) و وابستگی رو به پایین — `cli` به بقیه وابسته است، `models` به هیچ‌کس. تستِ سلامت: «افزودنِ قانونِ تخفیفِ جدید» فقط `discounts.py` را لمس می‌کند؛ «تغییرِ فرمتِ ذخیره» فقط `storage.py` را.

**۱۶.** فقط `parse` و `Formatter` — دقیقاً همان‌که `__all__` اعلام کرده. اگر `__all__` حذف شود، `import *` همه‌ی نام‌های **بدونِ زیرخطِ ابتدایی** را می‌آورد: `parse` و `Formatter` (مثل قبل)، اما `_tokenize` و `_Cache` باز هم نمی‌آیند — قراردادِ زیرخط اینجا هم محترم است. تفاوتِ واقعیِ `__all__` این است که رابطِ عمومی را **صریح** می‌کند و اگر فردا تابعِ بدونِ زیرخطِ جدیدی اضافه شود، ناخواسته واردِ `import *` نمی‌شود.

**۱۷.** علت: پایتون ماژول‌ها را هنگامِ اولین import **اجرا** می‌کند؛ وقتی `models` وسطِ بارگذاری‌اش `services` را import می‌کند و `services` هم برمی‌گردد سراغِ `models`، به ماژولی می‌رسد که **نیمه‌ساخته** است — نامی که می‌خواهد هنوز تعریف نشده → `ImportError`. دو راهِ‌حل:

- **راهِ اول (ترجیحی): شکستنِ چرخه با بازطراحی.** وابستگیِ دوطرفه معمولاً بوی جابه‌جاییِ مسئولیت می‌دهد — منطقِ `total_price` که به سرویس نیاز دارد احتمالاً اصلاً جای درستش `services` است (کلاسِ Anemic در models، منطق در سرویس)، یا بخشِ مشترک به ماژولِ سومی مثل `pricing.py` منتقل شود که هر دو از آن import کنند. چرخه حذف می‌شود چون معماری درست می‌شود.
- **راهِ دوم (مسکّن): importِ تنبل.** import را از بالای فایل به داخلِ تابعی که لازمش دارد ببرید؛ در لحظه‌ی فراخوانی، هر دو ماژول کامل بارگذاری شده‌اند. کار می‌کند، اما چرخه را پنهان می‌کند نه حل — به‌عنوانِ راهِ‌حلِ موقت قابلِ‌قبول است و باید با یک TODO علامت بخورد.

ترجیح با راهِ اول است: خطای importِ حلقوی تقریباً همیشه علامتِ یک مشکلِ طراحی است، نه یک محدودیتِ زبان.

**۱۸.** سیاستِ «همه‌چیز را re-export کن» سه هزینه‌ی پنهان دارد. اول، **مرزِ عمومی/خصوصی را پاک می‌کند**: وقتی همه‌چیز از `__init__` بیرون می‌آید، هیچ سیگنالی نمی‌ماند که «به کدام‌ها می‌شود تکیه کرد»؛ کاربرانِ پکیج به جزئیاتِ داخلی وابسته می‌شوند و بعد هر ریفکتورِ داخلی، یک تغییرِ شکننده (breaking change) می‌شود — نقضِ همان انتزاعی که `__init__.py` قرار بود بسازد. دوم، **هزینه‌ی import زنجیره‌ای**: importِ پکیج یعنی اجرای همه‌ی زیرماژول‌ها، حتی برای کسی که فقط یک تابعِ کوچک می‌خواست — کندیِ راه‌اندازی و ریسکِ importِ حلقوی بالا می‌رود. سوم، اثرِ ریفکتور: جابه‌جاییِ یک کلاس بینِ فایل‌های داخلی، که باید تغییری نامرئی باشد، حالا فهرستِ re-export و امضای عمومیِ پکیج را دست می‌زند. سیاستِ سالم‌تر همان است که فصل گفت: `__init__.py` فقط **رابطِ منتخب و آگاهانه** را صادر کند — چیزهایی که واقعاً قراردادِ پکیج با دنیا هستند — و بقیه با مسیرِ کامل در دسترس بمانند برای استفاده‌ی آگاهانه.
