# تمرین‌های فصل سیزدهم: متاکلاس‌ها و برنامه‌نویسی فراکلاس

> این تمرین‌ها یکی از عمیق‌ترین مباحثِ پایتون را می‌سنجند. هدف «درک» است، نه اینکه فردا متاکلاس بنویسید. ابتدا خودتان پاسخ دهید، سپس با `13-metaclasses-answer.md` مقایسه کنید.

---

## بخش الف: مفهوم متاکلاس

**۱.** جمله‌ی «در پایتون همه‌چیز شیء است، حتی کلاس‌ها» یعنی چه؟ در کدِ زیر، خروجیِ هر دو `print` چیست و هرکدام چه چیزی را نشان می‌دهد؟

```python
class User:
    pass

print(type(User))
print(type(User()))
```

**۲.** متاکلاس چیست؟ متاکلاسِ پیش‌فرضِ پایتون کدام است؟ رابطه‌ی «شیء → کلاس → متاکلاس» را با رابطه‌ی «نمونه → کلاس» مقایسه کنید.

**۳. (کدنویسی)** `type` با سه آرگومان چه می‌کند؟ کلاسِ زیر را **بدونِ** استفاده از `class` و فقط با فراخوانیِ `type(...)` بازنویسی کنید:

```python
class Dog:
    legs = 4
    def bark(self):
        return "Woof"
```

**۴.** این جمله را توضیح دهید: «عبارتِ `class` فقط یک میان‌بُرِ نحوی برای فراخوانیِ `type(name, bases, namespace)` است.»

## بخش ب: متاکلاس سفارشی و مقایسه

**۵. (کدخوانی)** در کدِ زیر، پیامِ «کلاسِ Service ...» **کِی** چاپ می‌شود — هنگامِ تعریفِ کلاسِ `Service` یا هنگامِ ساختِ یک نمونه از آن (`Service()`)؟ چرا؟

```python
class LoggingMeta(type):
    def __new__(mcs, name, bases, namespace):
        methods = [k for k, v in namespace.items() if callable(v)]
        print(f"class {name} is being created with methods {methods}")
        return super().__new__(mcs, name, bases, namespace)

class Service(metaclass=LoggingMeta):
    def start(self): ...
    def stop(self): ...
```

**۶.** جدولِ زیر را درباره‌ی تفاوتِ «متاکلاس» و «دکوراتورِ کلاسی» کامل کنید:

| ویژگی | دکوراتور کلاسی | متاکلاس |
|-------|----------------|---------|
| زمانِ عمل | ؟ | ؟ |
| انتقال به زیرکلاس‌ها | ؟ | ؟ |
| کاربردِ معمول | ؟ | ؟ |

**۷.** برای هر سناریوی زیر، ساده‌ترین ابزارِ مناسب را انتخاب کنید (دکوراتورِ کلاسی، `__init_subclass__`، یا متاکلاس):
- الف) می‌خواهید فقط یک کلاسِ مشخص را با افزودنِ یک متد تغییر دهید.
- ب) می‌خواهید هر زیرکلاسِ یک کلاسِ پایه، هنگامِ تعریف، خودکار در یک فهرست ثبت شود.
- ج) می‌خواهید در قلبِ فرایندِ ساختِ **همه‌ی** کلاس‌های یک خانواده مداخله‌ی عمیق کنید.

## بخش ج: ابزارهای مرتبط

**۸.** `__set_name__` در یک Descriptor چه کاری انجام می‌دهد؟ چه ارتباطی با کارکردِ `dataclass` و ORMهایی مثلِ SQLAlchemy دارد؟ چرا معرفیِ آن، نیاز به متاکلاس را در بسیاری موارد از بین برد؟

**۹.** `__class_getitem__` چه کاری انجام می‌دهد؟ چه ارتباطی با سینتکسِ `Repository[User]` (فصل نهم) و `list[int]` دارد؟

**۱۰. (کدخوانی)** متدِ `__instancecheck__` چه رفتاری را سفارشی می‌کند؟ در کدِ زیر، خروجیِ هر سه `print` چیست و چرا؟ این چه ارتباطی با Protocolها و `collections.abc` دارد؟

```python
class LenMeta(type):
    def __instancecheck__(cls, instance):
        return hasattr(instance, "__len__")

class Sized(metaclass=LenMeta):
    pass

print(isinstance([1, 2, 3], Sized))
print(isinstance("Hello", Sized))
print(isinstance(42, Sized))
```

## بخش د: کدنویسی

**۱۱. (کدنویسی)** یک دکوراتورِ کلاسیِ `add_repr` بنویسید که به هر کلاسی که روی آن گذاشته می‌شود، یک `__repr__` اضافه کند که نامِ کلاس را در قالبِ `<ClassName>` نشان دهد.

**۱۲. (کدنویسی)** یک Descriptor به نامِ `Field` بنویسید که با `__set_name__` نامِ attributeی که به آن نسبت داده شده را خودکار بفهمد و مقدار را در `__dict__`ِ شیء ذخیره/بازیابی کند. سپس با یک کلاسِ `User` که دو `Field` دارد نشان دهید کار می‌کند.

## بخش ه: پرسشِ تحلیلی

**۱۳.** تیم پیترز می‌گوید: «متاکلاس‌ها جادوی عمیق‌تری هستند که ۹۹٪ کاربران هرگز لازمشان ندارند. اگر شک دارید که به آن‌ها نیاز دارید، نیاز ندارید.» این جمله یعنی چه؟ اگر ۹۹٪ برنامه‌نویسان لازمشان ندارند، پس چرا این فصل را می‌خوانیم؟

**۱۴.** نویسنده می‌گوید «انتخابِ ساده‌ترین ابزارِ کافی، خودش نشانه‌ی مهندسِ باتجربه است». برای هدفِ «انجامِ کاری هنگامِ تعریفِ زیرکلاس»، ترتیبِ ابزارها از ساده به پیچیده را بنویسید و بگویید چرا نباید مستقیم سراغِ متاکلاس رفت.
